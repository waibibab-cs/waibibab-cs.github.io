【论文阅读；ICDE'24】CAGRA：高度并行的图构造与GPU的近似最近邻搜索

Cagra: Highly parallel graph construction and approximate nearest neighbor search for gpus

摘要

近似最近邻搜索（ANNS）在数据挖掘与人工智能的诸多领域中发挥着关键作用，涵盖信息检索、计算机视觉、自然语言处理以及推荐系统等方向。近年来数据量急剧增长，穷举式精确最近邻搜索的计算代价往往高得难以承受，因此必须采用近似搜索技术。基于图的方法凭借均衡的性能与召回率表现，在近期的近似最近邻搜索算法研究中受到广泛关注。然而，尽管大规模并行通用计算已得到广泛应用，仅有少数研究探索了利用图形处理器（GPU）与多核处理器的算力优势。为弥补这一研究缺口，本文提出一种基于并行计算硬件的新型近邻图构建与搜索算法。借助现代硬件的高性能特性，本方法实现了显著的效率提升。
具体而言，本文方法在近邻图构建环节超越了现有基于中央处理器（CPU）与图形处理器（GPU）的同类方法，在大批量与小批量搜索场景下均实现更高吞吐率，同时保持相当的精度。在图构建耗时方面，本文所提方法CAGRA相较 CPU 领域顶尖实现之一的HNSW快2.2–27 倍；在90%–95% 召回率区间的大批量查询吞吐率上，本方法比 HNSW 快33–77 倍，比当前 GPU 领域顶尖实现快3.8–8.8 倍；针对单条查询，在 95% 召回率下本方法比 HNSW 快3.4–53 倍。

一、引言

近似最近邻搜索（ANNS）的重要性随数据量持续增长而显著提升。ANNS 专门用于解决 k 近邻搜索（k‑NNS）问题，即在向量数据集中，依据 L2 范数等距离度量，找出与给定查询向量最接近的 k 个向量。k‑NNS 最简单的精确求解方式是，逐一计算查询向量与数据集中所有向量的距离，再输出距离最小的 k 个向量作为结果。但在大规模数据集上，这种方法需要执行海量相似度计算，难以落地，可扩展性极差。在许多实际场景中，人们并不需要完全精确的结果，而 ANNS 能够在吞吐率与精度之间取得平衡，降低计算开销，使其可扩展至超大规模数据集。

ANNS 的应用场景广泛且影响深远，覆盖数据挖掘与人工智能的众多领域，包括自然语言处理中的大模型、计算机视觉、信息检索、推荐系统与广告投放等。ANNS 也是聚类、分类、流形学习与降维等诸多核心数据科学与机器学习算法的基础。目前学界已提出并深入研究多种 ANNS 算法框架，包括基于哈希、基于树、基于量化以及基于图的方法。

基于图的 ANNS 方法依赖近邻图 —— 即用图结构表示数据点间的相似关系。这类方法主要包含两个步骤：从数据集构建近邻图，以及在图中遍历查找与输入查询最接近的 k 个节点。最优近邻图结构难以从理论上严格证明，例如，k 近邻图的质量并不必然带来更高的精度。因此，研究者多通过启发式与实验方式优化图的效率与结构。

其中具有代表性的工作包括：Malkov 等人提出的NSW（可导航小世界）图，借助小世界现象提升近邻图搜索效率；在此基础上，HNSW（层次化可导航小世界）图解决了 NSW 中少数节点度值过大、难以高效搜索的问题，通过引入层次化结构并限制节点最大度实现优化；Fu 等人提出的 NSG（可导航扩展图） 则通过近似单调相对邻域图（MRNG）结构，保证较低的搜索复杂度。这些只是 ANNS 中典型的图结构，该领域已涌现出大量图结构相关研究。

在基于图的 ANNS 方法实现方面，很少有工作面向数据中心服务器进行高性能优化，以充分利用 GPU 提供的大规模并行能力。加载向量的内存带宽往往是搜索吞吐的瓶颈，这一问题不仅存在于基于图的方法，也普遍存在于其他 ANNS 实现中。GPU 通常具备高内存带宽，天然适合执行 ANNS 任务。

代表性工作包括：SONG是首个基于 GPU 的图搜索实现，采用开放寻址哈希表、线程束内多查询并行等优化技术，吞吐率超过 FAISS 中的 GPU 版 IVFPQ 与 CPU 版 HNSW；GANNS是面向 GPU 优化的图搜索与构建方法，支持 NSW、HNSW 与 k 近邻图，通过适配 GPU 的数据结构与操作优化，进一步提升效率；Groh 等人提出的GGNN则是专为 GPU 设计的快速图构建与搜索方法，通过优化 GPU 数据结构与使用高速共享内存提升整体性能。

部分研究利用 “近邻图一次构建可重复使用” 的特性，缓解图构建耗时较长的问题。但如何设计在构建与搜索阶段均适配 GPU 架构的高效近邻图，依然是关键挑战。尽管基于 GPU 的图式 ANNS 已取得进展，但现有工作大多只是将已有图结构适配或优化到 GPU 上，而非从底层专门设计一套能在图构建与搜索阶段充分发挥 GPU 能力的近邻图。

为此，本文提出：
1）一种适用于图构建与查询并行计算的近邻图结构；
2）面向 NVIDIA GPU 优化的快速 ANNS 图构建与搜索框架 CAGRA（CUDA ANNS Graph-based）。
本文所提图结构以搜索实现为核心，专为 GPU 这类大规模并行设备提升效率而设计，而非追求理论上的图质量。
本文主要贡献如下：

  • 提出一种适用于大规模并行计算的 ANNS 近邻图及其构建方法，通过避免精确相似度计算，降低图优化过程中可能成为性能瓶颈的内存占用与访问开销。
  • 提供面向 GPU 优化的搜索实现，采用线程束拆分与可遗忘哈希表管理机制高效利用 GPU 资源，在单查询与大批量查询场景均实现高吞吐。
  • 通过实验证明，CAGRA 在图构建与查询性能上超过当前最优的 CPU 与 GPU 图式 ANNS 实现。具体而言，CAGRA 的图构建速度比 HNSW 快 2.2–27 倍；在 90%–95% 召回率区间的大批量查询吞吐上，比 HNSW 快 33–77 倍，比 GPU 端最优实现快 3.8–8.8 倍；在 95% 召回率下单查询速度比 HNSW 快 3.4–53 倍。

2.2 CUDA

图形处理器（GPU）近年来被广泛用于通用计算，尽管其最初仅为图形处理而设计。英伟达（NVIDIA）提出并持续开发了CUDA架构，使开发者能够将 GPU 的高性能计算能力用于通用计算场景。GPU 相比 CPU 具备更高的并行度与内存带宽，但要发挥其高性能，必须从算法中抽取出并行性，并将其映射到 GPU 硬件架构上。因此，并非所有应用仅通过使用 GPU 就能获得更高性能。下文将简要介绍 NVIDIA GPU 架构，更多细节可参阅文献 [23]。

1）线程层级：在 NVIDIA GPU 线程层级中，32 个线程组成一个线程束（warp），并同时执行同一条指令。反之，同一个线程束内无法并行执行不同指令，进而引发名为线程束分化（warp divergence） 的性能下降问题。最多 32 个线程束可组成一个协作线程数组（CTA），也称线程块（thread block），CTA 会在单个 GPU 流式多处理器（SM）上执行。SM 类似于多核 CPU 中的核心，而现代 GPU 包含大量 SM，因此可同时启动并运行多个 CTA。

2）内存层级：在 GPU 内存层级中，设备内存（device memory） 容量最大，所有线程均可访问同一内存空间。与之相对，共享内存（shared memory） 是每个 CTA 内部使用的局部内存，CTA 内所有线程共享该空间。共享内存的容量远小于设备内存，但具有更低延迟与更高带宽。寄存器（registers） 是每个线程的私有数据存储，延迟比共享内存更低。

2.3 相关工作

第一节已经介绍了多种基于图的近似最近邻搜索（ANNS）算法，因此本节将重点关注高度并行化、GPU 加速的图式 ANNS 实现，并阐明本文工作与这些方法的对比。

  1. SONG：SONG 是 Zhao 等人 [33] 提出的首个面向 GPU 的图式 ANN 实现。与 CAGRA 不同，该方法并未提出更快的图构建技术，而是依赖 NSW [19]、NSG [8]、DPG [16] 等已有方法。SONG 针对 GPU 设计了数据集与多项优化，采用开放寻址哈希表、有界优先队列和动态分配削减等策略，相比单线程 CPU 版 HNSW 实现了 10–180 倍 的加速。
  2. GGNN：GGNN 是 Groh 等人 [9] 提出的面向 GPU 的图式 ANNS 实现。与 CAGRA 一样，它同时提供高吞吐搜索与可利用高并行性的快速图构建能力。实验表明，GGNN 在大批量查询场景下的性能优于 SONG。
  3. GANNS：GANNS 是 Yu 等人 [32] 提出的GPU 加速图构建与搜索方法。该工作提出了基于 GPU 的 NSW 图构建方案，其近邻图构建速度与搜索性能均优于 SONG。
    遗憾的是，目前尚无研究直接对比 GGNN 与 GANNS，因此无法确定二者谁是当前最优的 GPU 图式 ANNS 实现。此外，上述所有 GPU 方案均聚焦于大批量查询场景。据我们所知，尚无任何 GPU 实现在小批量查询场景下超越 CPU 方案。本文将证明：CAGRA 在图构建与搜索环节，同时超越现有 CPU 与 GPU 方案。

三、CAGRA

本节将详细说明CAGRA 图的设计思路与核心特性。部分图结构的设计会遵循或逼近具备单调性等理论性质的图，而 CAGRA 图是以搜索实现为核心、通过启发式方式优化的图结构。CAGRA 图具备以下特性：

  • 固定出度（d）。通过固定出度，能够高效利用 GPU 的大规模并行能力。绝大多数基于图的 ANNS 算法都会构建并使用非固定出度的图。在 CPU 单线程执行时，每个节点采用非固定出度存在一定优势：仅保留必要的边连接，减少不重要的距离计算。但在 GPU 场景下，出度过小会导致分配给每个协作线程数组（CTA）的计算资源无法完全饱和，降低硬件利用率。反之，利用全部可用计算资源扩大搜索空间，并不会增加整体计算耗时。固定出度的另一项优势是能让计算更均匀，减少并行图遍历阶段的负载不均衡，避免硬件利用率偏低。我们会根据数据集、目标召回率与吞吐率设置出度大小。
  • 有向性。由于出度固定，该图天然为有向图。
  • 无层次结构。以 HNSW 为例，这类方法会采用层次化图结构来确定底层的初始节点。而在 GPU 上，我们可通过随机选取若干节点并计算其与查询的距离，得到效果相当的初始节点，充分发挥 GPU 的高并行性与高内存带宽优势。搜索算法的细节将在第四节展开。
    CAGRA 图的构建包含两个核心步骤：
    1）构建 k-NN 图；
    2）对其进行优化，提升搜索召回率与吞吐率。
    本文选择 k-NN 图作为基础图，原因是固定出度的图结构适配 GPU 高效运算，并且即便在 GPU 上，也能借助近邻下降算法（NN-descent）快速完成构建。下一节将介绍该启发式图优化方案及其并行计算算法。

3.1 图优化

CAGRA 图优化的核心目标是提升大量节点间的可达性。可达性由两个属性刻画：
1）能否从任意一个节点遍历到另一个任意节点；
2）在指定的路径遍历步数内，从一个节点出发能访问到的节点数量。
为评估属性 1），我们计算图中的强连通分量（CC）数量。强连通分量数量按如下规则确定：基础 kNN 图无法保证是连通的 [20]，弱连通分量表示图中子图的个数。此外，对于有向边图，即使图整体未断开，也可能出现从一个节点无法到达另一节点的情况。如果图中任意节点均可到达其他任意节点，则称该图是强连通的。强连通分量数量指图中满足强连通性质的节点组个数。强连通分量数量越小越好，因为分量数量过多会导致从搜索起始节点出发存在大量不可达节点。

在 CAGRA 图优化中，对初始 kNN 图采用两项关键技术：重排序（reordering） 与反向边添加（reverse edge addition）。

  • 重排序：不按边的长度排序，而是按照能提升图多样性的顺序，对初始 kNN 图的每条边重新排序，主要作用是增加 2‑hop 节点数。
  • 反向边添加：在其他基于图的 ANN 实现中常用的技术，用于提升节点可达性并减少强连通分量数量。

3.2 图构建与优化算法


CAGRA 图为所有节点出度均相等的有向图。其构建流程分为两个阶段：1）初始图构建；2）图优化，整体流程如图 1 所示。
1）初始图构建
本文构建 k-NN 图作为初始图，每个节点的初始出度设为 k=dinit​。采用近邻下降算法（NN-Descent） 完成构图，通常将初始出度 dinit​ 设置为最终 CAGRA 图固定出度 d 的 2 倍或 3 倍。
初始图构建的最后一步，将每个节点的邻接节点列表，按与源节点的距离升序排列。由于各节点邻接列表的计算相互独立、无依赖关系，该排序操作可在 GPU 上高效并行执行。
本文假设初始 k-NN 图已具备充足的节点连通性；CAGRA 图优化的核心目标是：在保持节点可达性不变的前提下，降低图的出度、缩减图存储规模。

NN-Descent算法是什么？
NN-Descent（近邻下降）是一种快速构建 k-NN 近邻图的启发式构图算法，无需暴力计算所有向量间距离，核心原理是：先为每个节点随机初始化少量邻居，再不断迭代交换邻居的邻居，筛选出距离更近的候选节点，逐步优化邻居集合，持续收敛直至找不到更优近邻，最终得到高质量近邻图；该算法仅做局部距离计算，开销远低于暴力枚举，且天然支持 GPU 并行，常用来生成高冗余初始图，为后续图剪枝优化做铺垫。
简单举例：现有向量节点 A、B、C，真实远近关系为 A-B 最近、B-C 次之、A-C 最远；初始随机给 A 分配邻居 C、B 分配邻居 A、C 分配邻居 B，迭代时每个节点挖掘邻居的邻居，A 从邻居 C 处发现 B 并比对距离，替换掉远距离邻居 C，多轮迭代后所有节点都收敛到真实最近邻，完成高效构图。

2）图优化
优化流程包含两步：① 边重排序；② 反向边添加。
该流程以初始图为输入，输出最终的 CAGRA 图，具备两大显著优势：
（i）无需依赖原始数据集，也无需额外做距离计算；
（ii）大量计算无复杂依赖，可高度并行执行。
在边重排序阶段，算法会为每条边定义重要性权重（rank），用于在整个优化流程末尾做边裁剪。

为什么要做重排序以及重排序阶段为什么要为每条边定义重要性权重？
对图邻接列表做重排序，是为后续图剪枝做前置准备，若仅单纯按距离直接截断保留前 d 个邻居，容易误删维持图连通性的关键桥边、破坏节点可达性，造成检索召回率下降；因此重排序阶段需要为每条边定义重要性权重，该权重综合了可绕行替代路径数量与近邻优先级，用来衡量每条边的不可替代程度，权重越高代表边对维持图拓扑连通性越关键，后续便可依据权重从低到高剔除冗余边，在把图出度压缩至 d、减少存储开销的同时，稳妥保留核心连通边，保证剪枝后整张图的可达性不受破坏。

其中e(.->.)和w(.->.)分别表示两节点间的有向边与距离；Z为与节点X存在直连入边、与节点Y存在直连出边的中间节点。基于该剪枝策略，本文研究了基于距离和基于秩两种重排序方法，并在 CAGRA 图优化中选用了基于秩的重排序方案。两种重排序操作的时间复杂度均为O($Nd^3$)，但基于距离的重排序策略还需要额外计算单个节点与其各邻居之间的向量距离。

在基于距离的重排序中，我们首先统计每条边的可绕行路径数量，再按该数量升序排列节点邻接列表。一条边的可绕行路径数量越少，说明其对维持两跳节点计数的作用越关键。随后，我们将这条边在邻接列表中的位置作为其秩（rank），以此作为衡量边重要性的指标。基于距离的重排序计算具有高度并行性，因为我们可以并行统计每条边的可绕行路径数量。但该方法需要在操作过程中实时计算距离，或预先构建距离表，这使其在大规模数据集上不具备实用性。前者需要进行 N×dinit​×(dinit​−1) 次距离计算，后者则需要在内存中存储一个规模为 N×dinit​ 的距离表，其中 N 为数据集的总规模。


在基于秩的重排序中，我们将初始图构建完成时、已按距离升序排好序的邻居节点列表中，边所处的位置设为初始秩，这一点与基于距离的重排序类似。随后，我们采用与基于距离的重排序相同的方式对边进行重排序，但使用初始秩而非距离作为判断依据，如图 2 所示。换言之，我们用初始秩来近似替代距离。这种近似方法使我们无需执行不切实际的大规模距离计算，也无需在内存中存储体积庞大的距离表。我们将节点的排序索引作为秩，这一点与基于距离的重排序保持一致。本文在 CAGRA 图优化中采用基于秩的重排序，并为每个节点仅保留前 d 个邻居（剪枝）。

基于秩的重排序相较于基于距离的重排序最大优势就在于无需在初始化图构建阶段保留每个边的长度的具体数值（规模：N×dinit​）而是只需要记录索引（长度排名）

在完成重排序后，我们构建一个反向图，其中所有边的方向均与重排序剪枝后的图相反。由于重排序剪枝图中每个节点的入度（即传入边数）并不固定，反向图中每个节点的出度也不固定。但考虑到后续操作的特性，我们将反向图的度数上限设为d。同时，反向图中的边需按剪枝图中的秩升序排列，其含义为：“认为你更重要的节点，对你而言也更重要”。添加反向边的过程时间复杂度为O(Nd)。

最后，我们将剪枝后的原图与反向图进行合并。合并过程中，我们会为每个父节点从两张图中各选取约 d/2 个邻居，并将它们交错排列。若反向图中某父节点的邻居数量不足 d/2，则从剪枝原图中补充邻居。为找到最优的 d 值，我们会构建不同出度（如 32、64、96）的图并测试其检索性能。由于该参数依赖于数据集特性与用户需求，无法一次性确定其最优值，这与其他方法的超参数（例如 HNSW（libhnsw）图的最大出度）类似：增大出度会提升召回率，但会降低检索吞吐量。

四、CAGRA搜索

本节将介绍 CAGRA 检索算法原理，以及该算法如何映射到 GPU 上执行。

4.1 算法原理


如图 6 上半部分所示，CAGRA 检索算法采用串行内存缓冲区，由内部 Top-M 列表（在其他算法中通常称为优先队列）和候选列表组成。
内部 Top-M 列表长度为 M（满足 M≥k）；候选列表规模为 p×d，其中 p 为每轮图遍历的源节点数量，d 为 CAGRA 图的节点度数。
缓冲区的每个元素均为键值对，存储节点索引，以及该节点与查询向量之间的距离。

如图 6 下半部分所示，检索计算流程如下：
0 随机采样（初始化阶段）
通过伪随机数生成器均匀采样 p×d 个节点索引，计算每个节点与查询向量的距离，并将结果存入候选列表。
同时用无效占位项初始化内部 Top-M 列表，占位距离值设为足够大，保证在后续排序中排在末尾；例如浮点类型可设为 FLT_MAX（浮点最大值）。
1 内部 Top-M 列表更新
从整个缓冲区中选出距离最小的前 M 个节点，更新至内部 Top-M 列表。
2 候选列表索引更新（图遍历阶段）
选取内部 Top-M 列表中前 p 个节点的所有未作为父节点访问过的邻居节点索引，将这些节点索引存入候选列表。
本阶段不计算候选列表中节点与查询向量的距离。
3 距离计算
仅当节点首次出现在当前查询的候选列表中时，才计算该节点与查询向量的距离。
这种条件式分支判断可裁剪冗余计算：若节点距离已在之前迭代中计算过，则无需重复计算。具体逻辑为：
若节点曾出现在列表中：

  • 距离足够小、本就留在 Top-M 列表中，则无需重复加入；
  • 距离过大、本就无法进入 Top-M 列表，则也无需再次加入。

 步骤二判断“未作为父节点访问过” 和步骤三判断“是否首次出现在候选列表”中有什么区别?
 

循环迭代执行步骤 1～3，直至 Top-M 列表中的节点索引收敛（与上一轮迭代结果保持一致）。最终取内部 Top-M 列表中的前 k 个节点，作为近似最近邻检索（ANNS）的输出结果。

4.2 核心底层技术与设计

本节介绍在 GPU 上实现 CAGRA 检索算法所采用的底层关键技术与架构设计。
1）线程束拆分（Warp splitting）
如2.2节所述，GPU 中一个线程束（warp） 包含 32 个线程，所有线程同时执行同一条指令，是硬件层面最小的并行线程调度单元。
在 CAGRA 检索实现中，软件层面将一个线程束进一步拆分为更小的线程组，称为线程小队（teams）；每个小队包含固定数量线程，该数量定义为小队规模（team size）。

这种拆分设计能够提升 GPU 利用率，原因如下：

从设备内存访问延迟角度来看，128 位访存指令的内存加载效率最优。传统方案通常将一次距离计算映射给一整个线程束，并借助线程束洗牌指令（warp shuffle）实现线程间协同计算。
但若向量维度为 96、采用 float 浮点类型（4 字节），单个向量总位数为 3072 bit；而线程束 32 个线程全部发起 128 位加载时，总加载位数为 4096 bit。
这就导致线程束中有部分线程无需执行加载指令，造成 GPU 计算资源浪费、利用率低下。

若改为将一次距离计算映射给规模为 8 的线程小队，单个小队单次可加载 1024 bit，只需小队所有线程重复 3 次加载即可完整读取整个向量。
同时，同一个线程束内的其他小队可并行计算候选列表中其他节点与查询向量的距离，最大化 GPU 利用率与执行效率。
虽然在软件层对线程束做了小队拆分，但同一个线程束内所有小队仍执行相同指令，不会产生线程束分支分化（warp divergence）。

2）Top-M 计算优化
算法步骤 1 需要从缓冲区中筛选出距离最小的前 M 个条目。
由于各局部内部 Top-M 列表本身已预先排序，无需对整个缓冲区做完整全量 Top-M 排序，可大幅减少计算量。
具体实现逻辑：先对候选缓冲区排序，再通过 双调排序（bitonic sort） 的归并流程，将其与内部 Top-M 缓冲区做归并合并。

  • 当候选缓冲区大小 ≤ 512 时，采用单线程束级双调排序；
  • 当缓冲区大小 > 512 时，采用单个线程块（CTA）内基数排序。
    该设计的依据：候选列表较小时，可直接在单个线程束的寄存器内完成快速排序，无需占用共享内存；列表长度较大时则必须使用共享内存，此时性能会受限，改用基数排序可获得更优性能。

一、双调排序是什么？
双调排序（Bitonic Sort）是一种经典的并行排序算法，首先要理解什么是双调序列：
简单来说，一个序列如果是双调的，它必须满足以下条件之一：1. 先单调递增，后单调递减（例如：[1, 4, 9, 5, 2]）。2.先单调递减，后单调递增（例如：[8, 3, 2, 6, 7]）。3.可以通过循环移位达到上述状态的序列。
双调排序的核心逻辑分为两步，通常要求待处理的数据量为 $2^n$：1.构建双调序列：将无序序列变成一个大的双调序列。2.双调合并（Bitonic Merge）：这是该算法最精妙的地方。当你有一个双调序列时，将其前半部分和后半部分对应位置进行比较并交换（小的放前，大的放后），你会得到两个更小的双调序列，且左边序列的所有元素都小于右边。递归执行此操作，序列最终会变得完全有序。

举一个例子：
假设我们有初始数组：[10, 2, 5, 8]
step1.构建双调序列
先将原始序列 [10, 2, 5, 8] 每两个元素分为一组，把前一组 [10, 2] 调整为升序 [2, 10]，后一组 [5, 8] 调整为降序 [8, 5]，再将两组拼接得到 [2, 10, 8, 5]；该序列先从 2 递增到 10、再从 10 递减到 5，满足先增后减的特征，构成标准双调序列。
step2.双调合并
跨度为 2 的比较交换：比较第 1 位 (2) 和第 3 位 (8)：2 < 8，保持不变。比较第 2 位 (10) 和第 4 位 (5)：10 > 5，交换。数组变为：[2, 5, 8, 10]。跨度为 1 的比较交换：比较第 1 位 (2) 和第 2 位 (5)：2 < 5，保持不变。比较第 3 位 (8) 和第 4 位 (10)：8 < 10，保持不变。最终结果： [2, 5, 8, 10]，排序完成。

二、基数排序是什么？
基数排序（Radix Sort）是一种非比较型的整数排序算法。它的核心逻辑是将整数按位数切割成不同的数字，然后按每个位数分别比较。
详细例子：对数组 [170, 45, 75, 90, 802, 24, 2, 66] 进行排序
基数排序通常采用 LSD（最低位优先） 法：
第一轮：按“个位”排序
根据个位数（0-9）将数字放入对应的桶中：
0 桶: 170, 90
2 桶: 802, 2
4 桶: 24
5 桶: 45, 75
6 桶: 66
收集结果：[170, 90, 802, 2, 24, 45, 75, 66]（此时个位已变有序）
第二轮：按“十位”排序
在上一轮的基础上，看十位数字（没有十位的看作 0）：
0 桶: 802, 2（十位都是 0）
2 桶: 24
4 桶: 45
6 桶: 66
7 桶: 170, 75
9 桶: 90
收集结果：[802, 2, 24, 45, 66, 170, 75, 90]
第三轮：按“百位”排序
看百位数字：
0 桶: 2, 24, 45, 66, 75, 90
1 桶: 170
8 桶: 802
收集结果：[2, 24, 45, 66, 75, 90, 170, 802]

3）访问节点哈希表管理机制
算法步骤 3 规定：仅当节点首次出现在候选列表时，才计算其与查询向量的距离。
这就需要一种机制记录节点是否已被访问过。本文借鉴 SONG 算法思路，在 CAGRA 检索中采用开放寻址哈希表管理已访问节点列表。
哈希表最大条目数计算公式：Imax​×p×d，其中 Imax​ 为检索最大迭代次数。工程上将哈希表大小至少设为该值的 2 倍，降低哈希冲突概率。

如果我们把哈希表放置在有限的内存中（例如共享内存），且其容量不足以容纳所有数据，我们可以采用一种较小的哈希表并配合周期性重置。这意味着哈希表会每隔一定的间隔剔除（evict）之前访问过的节点。在重置哈希表之后，我们仅将当时存在于内部前 M 名列表（top-M list）中的节点重新注册到哈希表里。虽然这一过程可能会增加距离计算的次数，但正如文献 [33] 所述，并不会发生灾难性的召回率下降。我们将这种针对有限内存、采用重置机制的哈希表管理方式称为“易忘式哈希表管理”。我们将哈希表的条目数设定在 $2^8$ 到 $2^{13}$ 之间，重置间隔通常根据图的类型及搜索参数 $M$、$d$、$p$ 设为 1 到 4。这种哈希表管理方式减少了单次查询的共享内存占用量（通常 $\le$ 4 kB），从而在几乎所有世代的 NVIDIA GPU 上进行大批量查询时，都能获得更高的并行效率。

4）1 比特父节点标记管理
算法步骤 2 需要筛选出从未作为父节点的邻居节点，加入候选列表。
为标记节点是否已充当过父节点，该设计复用缓冲区中节点索引变量的 最高有效位（MSB） 作为标记位。
替代方案可额外再建一张哈希表做记录，但哈希表查询会引入访问延迟，因此本文不予采用。
复用最高有效位的方式只需读取节点索引的最高位，即可快速判断该节点是否已作为父节点，延迟更低。
该方案的局限：会限制数据集最大规模，索引可用数值范围减半。
例如采用 uint32_t 类型存储索引时，常规可用最大值为$2^{32}-1$；复用最高位做标记后，仅支持最大数据集规模$2^{31}-1$。

4.3 实现

本节介绍 CAGRA 检索算法在 GPU 上的实现特性与优化方案。CAGRA 检索分为单线程块和多线程块两种实现方式，二者基础检索策略与运算流程一致，仅在硬件与查询任务的映射方式上存在差异，两种实现的汇总对比如表 2 所示。

1）单线程块实现
顾名思义，单线程块实现将每条查询映射到一个线程块，从而高效处理查询任务。该方案适用于中大规模查询批次，例如 100 及以上的批次规模。借助 GPU 的并行计算能力，多个单线程块可同时执行，既能批量高效处理多条查询，也能充分利用 GPU 硬件资源；但在查询批次较小时，容易出现 GPU 资源闲置、性能无法达到最优的问题。

在实现层面，本文采用单内核函数封装完整的 0～3 步检索流程，并将哈希表存放于共享内存而非设备内存中。本文也尝试过另一种拆分实现，即为检索流程的每个步骤分别设计独立内核函数。但大量测试表明，频繁启动多个内核带来的调度开销，超过了拆分设计所能带来的潜在性能收益，因此多内核拆分方式不具备优势，最终采用单内核整体实现方案。

对实现吞吐特性分析后发现，当查询批次规模和数据集向量维度较大时，设备内存带宽会成为内核函数的性能瓶颈。为此本文采用低精度数据类型存储数据集向量，以此缓解内存带宽限制、提升整体吞吐能力。该优化通过降低向量内存占用，加快存储与计算单元之间的数据传输，从而提升吞吐；同时需要合理权衡吞吐提升与召回率损失之间的折中关系。

2）多线程块实现
多线程块实现将单条查询映射到多个线程块，适配小批次查询场景，典型批次范围为 1～100。单线程块实现所启动的线程块数量与查询批次相等，而多线程块方式利用多个线程块共同处理同一条查询，最大化 GPU 资源利用率，即使在小批次场景下也能提升查询处理效率。

该实现中哈希表存放于设备内存，以供多个线程块共享。每个线程块各自遍历图节点、维护自身的内部 Top-M 列表与候选列表，将父节点数量设置为 1，同时共用同一份哈希表。单线程块每轮最多遍历 p × d 个节点，而多线程块每轮可遍历线程块数量乘以 d 个节点。由于单线程块通常设置 p=1 以最大化吞吐，同等迭代次数下，多线程块每轮访问的节点数量更多，召回率也更高。

本文还探索了图分片方案，该方法广泛应用于多节点近似最近邻检索，初衷是提升小批次下的 GPU 资源利用率，但受实际工程限制最终放弃。图分片构建子图依赖特定图结构，还需要对索引进行重排与数据集拆分，执行与优化难度较大。后续参考 GGNN 的建图方式，为每个子数据集独立构建子图；检索阶段将每个子图分配给一个线程块，当子数据集数量足够多时确实可以提升 GPU 利用率，但仍存在诸多实际问题：最优分片数量依赖查询批次、GPU 流式多处理器数量等因素，很难为不同批次和硬件预先定制子图结构，实用性不足。但图分片技术仍适合将图结构近似最近邻检索扩展到多 GPU 环境，可由每个 GPU 独立负责一个子图的检索任务。

3）实现方式选择
如上所述，两种实现分别面向小批次和大批次场景，系统会同时依据查询批次规模与内部 Top-M 大小选择对应实现，如图 7 所示。当查询批次小于阈值 bT，或内部 Top-M 规模大于阈值 MT 时，采用多线程块实现；这类场景下若使用单线程块，步骤 1 的计算开销会显著增加整体耗时。其余场景均默认使用单线程块实现。阈值取值与硬件相关，工程经验建议设置 MT 为 512，bT 取 GPU 的 SM 数量。

实验环境

    • 小规模：SIFT-1M、GIST-1M、NYTimes、GloVe-200、DEEP-1M
    • 大规模：DEEP-10M、DEEP-100M
  • 向量精度：默认 FP32；部分实验使用 FP16 低精度加速。

评估指标

  • 召回率 recall@k：检索结果中真实近邻占比，常用 recall@10、recall@100。
  • 吞吐量 QPS（Queries Per Second）：每秒处理查询数，衡量检索速度。
  • 建图时间：包含初始 k-NN 图构建、图优化总耗时。
  • 图质量：强连通分量（CC）数量、平均 2 跳节点数（N₂hop）。

对比工作

与四类主流图基近似最近邻（ANNS）方法对比：

  1. HNSW（CPU SOTA）：经典分层小世界图，CPU 基准。
  2. GGNN（GPU SOTA）：GPU 友好图构建与检索。
  3. GANNS（GPU SOTA）：GPU 加速 NSW 图构建与检索。
  4. NSSG：随机采样起始、检索流程与 CAGRA 相似的图基方法。
    对比时统一平均出度，保证公平性。

核心结果（简要）

  • 建图速度：CAGRA 比 HNSW 快 2.2–27×；比 GGNN 快 1.1–31×；比 GANNS 快 1.0–6.1×。
  • 大批次检索（10K）：90%–95% 召回下，CAGRA 比 HNSW 快 33–77×，比 GPU SOTA 快 3.8–8.8×。
  • 单查询检索：95% 召回下，CAGRA 比 HNSW 快 3.4–53×。
  • 大规模数据：DEEP-100M 上仍保持稳定召回与吞吐量。
