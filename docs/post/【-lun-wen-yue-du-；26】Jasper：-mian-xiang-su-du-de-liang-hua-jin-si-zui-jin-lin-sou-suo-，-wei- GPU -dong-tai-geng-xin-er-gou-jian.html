Jasper: ANNS Quantized for Speed, Built for Change on GPU

摘要

近似最近邻搜索（ANNS）是多项机器学习与信息检索应用中的核心问题。GPU 为高性能近似最近邻搜索提供了一条极具前景的实现路径：GPU 能为距离计算提供大规模并行算力、在现代数据中心中易于部署，还可与下游应用共置，从而减少数据迁移开销。

尽管具备上述优势，当前基于 GPU 加速的近似最近邻搜索系统仍面临三大关键局限：
第一，实际应用场景中的数据集持续动态变化，需要快速批量更新数据，但大多数 GPU 索引在新增数据时都必须从零开始重新构建；
第二，高维向量会严重占用内存带宽，而现有 GPU 系统缺乏高效的量化技术 —— 既能减少数据传输量，又不会产生高成本的随机内存访问；
第三，贪心搜索本身存在数据依赖型内存访问，难以实现计算与内存操作的并行重叠，进而导致性能下降。

为此，本文提出Jasper—— 一款GPU 原生的近似最近邻搜索系统，兼顾高查询吞吐量与完整可更新性。Jasper 基于 Vamana 图索引构建，通过三大核心创新突破现有瓶颈：

  1. 适配 CUDA 的批量并行构建算法，支持无锁流式插入；
  2. 高效适配 GPU 的RaBitQ 量化实现，最高可将内存占用压缩至1/8，且无随机访问损耗；
  3. 优化的贪心搜索内核，提升计算资源利用率，实现更优的延迟隐藏效果，进一步提高吞吐量。

在 5 个数据集上的评测结果显示：Jasper 的查询吞吐量较当前 GPU 领域最优索引 CAGRA最高提升 1.93 倍；基于屋顶线模型测算，其峰值资源利用率可达80%。Jasper 的索引构建能力可高效扩展至大规模数据集，构建速度较 CAGRA平均快 2.4 倍，同时具备 CAGRA 所缺失的可更新能力。对比此前速度最快的 GPU 版 Vamana 实现 BANG，Jasper 的查询速度提升 19 至 131 倍。

一、引言

最近邻搜索是机器学习与信息检索领域的基础问题。该问题旨在在高维空间中，找出与查询向量最接近的 k 个点。在此框架下，图像、文档、用户画像等复杂数据会被嵌入为度量空间（如欧氏距离、曼哈顿距离、汉明距离）中的向量，通过几何距离衡量相似度。相关应用涵盖推荐系统、图像检索、异常检测，以及越来越重要的大语言模型检索增强生成（RAG）—— 这类场景需要从海量语料中实时检索相关上下文。

传统索引结构（如 k-d 树、四叉树）在低维空间中可高效实现精确最近邻搜索，但随维度升高，性能急剧下降。维度灾难导致数据点间距离趋于相近，最近邻与最远邻的距离比趋近于 1，破坏了传统索引依赖的空间剪枝策略，查询复杂度退化为暴力线性扫描。现代嵌入模型输出的向量维度可达数百至数千，使得精确搜索在实际中难以应用。

近似最近邻搜索（ANNS）通过放宽精确性约束解决该问题：不返回严格精确的最近邻，而是返回误差可控范围内的近似结果。这种速度 — 精度权衡使算法可扩展到高维大数据集。ANNS 的三大主流范式包括：倒排文件索引（IVF）、局部敏感哈希（LSH）、图索引。其中，图索引在召回率 — 吞吐量上的综合表现最优，多项实证研究均证实其优势。

图索引构建可导航搜索图：顶点对应向量，边表示邻域关系。最近邻查询通过贪心遍历（束搜索）实现：从指定起点出发，不断扩展最近的未访问顶点，直至收敛，过程中维护一组当前最优候选结果。

现代应用要求 ANNS 索引支持动态更新。现实场景中的数据持续变化，需要不断更新索引。网页级检索系统需索引数十亿向量：Facebook 相似度搜索处理超 2000 亿嵌入，微软 Bing 利用 DiskANN 检索超 10 亿文档向量。这类系统需同时支持每秒百万级查询和实时增量更新，音乐流、商品推荐、RAG 系统均依赖可扩展、可更新的相似度搜索。

GPU 加速的必要性
GPU 提供大规模并行能力，可显著提升 ANNS 的吞吐量并降低延迟。图索引 ANNS 的核心操作（成对距离计算）天然独立、可并行；索引构建与查询均以距离计算为主，GPU 可并行处理大量候选向量。随着数据规模与维度增长，GPU 的并行潜力愈发显著。

此外，GPU 已广泛部署于 ANNS 应用场景：现代机器学习基础设施基于 GPU 集群运行。使用 GPU 处理 ANNS 可复用现有硬件，无需额外搭建 CPU 检索集群；共置部署减少数据搬运：GPU 生成的嵌入可直接索引与查询，无需昂贵的 CPU-GPU 数据传输。

GPU 加速 ANNS 的三大挑战
现代 GPU 拥有数千计算单元、内存带宽超 2TB/s，但 ANNS 索引与查询仍难以发挥峰值性能，核心瓶颈有三：

  1. 可更新性不足：多数 GPU 索引在新增数据时需全量重建，源于 GPU 内存分配与释放机制限制。异常检测、RAG 等场景要求索引支持增量更新。
  2. 贪心图遍历导致不规则访存：稠密矩阵运算访存模式固定，而图搜索路径依赖数据、随查询变化。同一个线程束（warp）内线程访问不连续内存时，GPU 需多次内存事务，浪费带宽；图遍历的指针追逐特性进一步加剧该问题，每一步依赖前一步结果。
  3. ANNS 是内存密集型而非计算密集型：向量每个维度仅读取一次、用于一次乘加、随即丢弃，无数据复用。性能受限于内存读取速度而非计算速度。例如 128 维 float32 向量：GPU 执行 256 次浮点运算，但需加载 512 字节数据，运算 / 数据比远低于 GPU 峰值效率。分析显示，当前最优 ANNS 搜索内核已接近内存带宽上限，性能提升需减少数据传输。

量化技术就是减缓显存墙的一种手段

现有 GPU-ANNS 系统的局限
现有 GPU-ANNS 系统要么不可更新，要么性能低下：

  • CAGRA：当前最优 GPU-ANNS，基于 NN-Descent 优化图结构，查询吞吐量高，但批量构建、不支持增量更新；生产环境需全量重建，严重受限。
  • BANG：基于可更新图结构，但依赖乘积量化（PQ）压缩向量，性能损失严重；虽基于 Vamana 图、理论可更新，但当前未实现增量更新。
  • 其他系统：GANNS（HNSW）受限于串行构建依赖；SONG、GTS 分别面向固定秩图、树索引，各有取舍。无系统同时实现最优查询性能、高效构建、流式更新。

本文提出 Jasper——GPU 原生 ANNS 系统，兼顾高查询吞吐量与动态更新，三大核心贡献：

  1. 无锁批量并行构建算法：基于 ParlayANN 优化，适配 GPU 块级并行，支持无锁流式插入，构建性能业界最佳，首个支持增量更新的高性能 GPU-ANNS。

二、预备知识

2.1 GPU执行模型

可参考：https://waibibab-cs.github.io/post/CUDA-guan-fang-wen-dang-xue-xi-1%EF%BC%9ACUDA-jie-shao.html


GPU 架构示意图如图 1a 所示。GPU 由多个流式多处理器（SM）组成，每个 SM 最多可同时承载 1024 个线程。GPU 上的任务以称为内核（kernel）的工作单元形式执行。每个内核属于单指令多线程（SIMT）函数，所有线程并行执行该函数。GPU 内核以线程块集合的形式在 SM 上运行，每个线程块最多包含 1024 个线程。在一个线程块内部，线程被划分为线程束（warp），每组包含 32 个线程，共享执行资源并协同发起内存操作。因此，当线程束内的线程执行相同控制流、访问连续或相邻内存地址时，性能达到最优。线程束还可进一步划分为线程片（tile），将线程束内资源分配给多个任务。

在最近邻搜索过程中，ANNS 任务呈现细粒度并行特性，因为向量间距离可独立并行计算。每个向量由一个线程片负责处理，使多个候选向量能并行读取和评估。线程片密集排布在线程束中：较大的线程片可加快单个向量处理速度，提升单次探查并行度；较小的线程片则能在线程束内发起更多并发内存请求，增强延迟隐藏能力。最优的线程块与线程片配置取决于 ANNS 索引的特性，包括内存布局、探查分支数以及候选过滤开销。

2.2 GPU内存模型

可参考：https://waibibab-cs.github.io/post/CUDA-guan-fang-wen-dang-xue-xi-3%EF%BC%9ACUDA%20SIMT-he-han-shu.html

并发 GPU 数据结构的设计从根本上受到底层内存一致性模型的约束，该模型规定了来自不同线程的内存操作如何对彼此可见。与具备顺序一致性的 CPU 不同，现代 GPU 基于宽松内存排序（RMO）实现了一种弱内存模型。在宽松内存排序机制下，读写操作可以被自由重排序，除非显式添加约束；如果没有额外的同步机制，一个线程中逻辑上程序顺序晚于另一个线程写入操作的读取操作，无法保证能观测到该次写入结果。

解释：
CPU速度慢，但一致性强：线程A写完一个数据、线程B一定能读到
GPU线程多，但一致性弱，不会限制严格的先后顺序，因为同步开销太大，所以需要采用显示同步指令等待一部分操作执行完成再执行另一部分

GPU 优先保证吞吐量而非延迟。由于 GPU 上存在数千个并发线程，强制实施强排序会产生高昂的缓存一致性通信开销。同一个SM内的所有线程共享一级缓存，该缓存与其他流式多处理器上的一级缓存之间不保持一致性。这些流式多处理器独享的一级缓存连接到具备全局一致性的二级缓存，二级缓存再为 GPU 的主存提供支撑。这种共享的一级缓存对于 GPU 应用实现高性能至关重要。该缓存的一部分区域可被指定为共享内存。与普通缓存不同，这种共享内存可直接寻址，既可以作为暂存器使用，也能在线程块内的线程之间高效传递数据。

使用的共享内存大小会影响潜在占用率，即一个流式多处理器上可同时调度的最大线程块数量。内核中的每个线程块都会申请相同大小的共享内存。如果总共享内存用量超出了流式多处理器的承载能力，占用率就会降低，导致并行运行的线程块数量减少，部分线程处于空闲状态。

为了在 GPU 上实现高性能，我们必须为近似最近邻搜索索引构建与查询操作设计 GPU 内核，使其高效利用共享内存以最大化占用率。具体而言，我们的目标是将所有频繁使用的数据存放在共享内存中，并精心选择适配可用资源的数据结构，同时不降低最大占用率。

2.3 屋顶线模型

屋顶线模型是一种简洁的量化性能模型，它利用两项硬件极限——峰值计算吞吐量与峰值内存带宽——来限定应用在特定硬件上可达到的性能上限。性能被表示为运算强度的函数，运算强度定义为有效运算次数与从主存传输的字节数之比。

对于低运算强度的内核，性能受限于内存带宽，随运算强度线性增长；对于高运算强度的内核，性能会在硬件峰值计算速率处趋于饱和。从图形上看，这些极限构成一条 “屋顶线”：一条倾斜的带宽上限线与一条水平的计算上限线相交，由此可以直观判断工作负载是内存受限还是计算受限，以及其距离硬件极限的远近。

屋顶线模型被广泛用于指导性能优化与架构分析，它将复杂的微架构细节抽象为一组可测量的参数。提高数据复用的优化方法（如分块、缓存、算子融合）会通过提升运算强度，使内核在图中向右移动；而向量化、并行化与指令级优化则会让内核向上移动，靠近计算上限。

该模型的扩展形式可纳入多级内存、特定瓶颈上限与异构设备，但其核心思想始终不变：可实现的性能从根本上受计算量与数据传输量之间的平衡约束。这一分析框架能有效解释 CPU、GPU 与加速器之间的性能可移植性，并明确系统与数据密集型任务中的优化优先级。本文使用屋顶线模型评估 GPU 上近似最近邻搜索索引的实际性能与潜在性能。

三、Vamana索引

在本章节中，我们介绍 Vamana 算法的图结构、构建流程、查询算法以及批量并行构建算法。本文所提 Jasper 系统采用 Vamana 图结构 [18]，这是一种有向邻近图，被设计用于在出度受限的条件下实现高效贪心搜索，以此构建近似最近邻检索（ANNS）索引。本文选用 Vamana 图，原因在于其既支持快速且高精度的贪心检索，也兼容 ParlayANN [23] 中提出的高效并行构建算法。下一章节将阐述如何实现 GPU 加速的 Vamana 索引，并介绍多项 GPU 专属优化策略。

parlayANN：基于CPU、不涉及量化、不支持动态更新、核心贡献在于设计了无锁的、确定性的算法通过消除并行计算瓶颈和随机性来加速计算

3.1 图定义

Vamana 索引是在数据集向量集合V={x1​,...,xN​}上构建的有向图G=(V,E)，其中N为向量总数量。图中每个节点维护一张邻接表，最大出边数量不超过R。图中设有一个特殊入口节点s∈V，作为图构建过程与查询遍历过程的起始节点。边集E的选取原则是：在保持图结构稀疏性的同时，保证图具备可导航性。对于图中任意节点与查询向量，该节点的某一条出边有极高概率指向距离查询向量更近的节点。该特性使得贪心检索能够收敛到距离目标查询相近的向量集合，也是实现低延迟近似检索的核心关键。

3.2 Vamana构建算法

原始 Vamana 构建算法初始时将所有顶点纳入图中，并为每个顶点随机生成一组最大出度为R的边。Vamana 图采用逐顶点增量式方式逐步优化图结构。顶点优化过程分为两个阶段：第一阶段，Vamana 通过束搜索（beamSearch）在已有图中为新顶点筛选候选邻居；第二阶段，调用鲁棒剪枝（robustPrune）流程，在严格约束出度上限的同时，将该顶点重新接入图结构。

每一轮迭代中，算法从边界集中选出距离最近且未被访问过的顶点，加载其所有出边，计算向量x与每个邻居的距离，并将新发现的顶点加入边界集。若边界集大小超出束宽 K，则舍弃距离最远的候选顶点。该扩展过程持续进行，直至边界集中所有顶点均被访问完毕。流程最终返回两个结果：作为x近似近邻的最终边界集，以及记录插入过程局部搜索范围的完整已访问列表。

边接入
束搜索完成后，利用已访问列表中的顶点将新顶点x接入图中。首先为x与已访问列表中每个顶点添加双向边，此时暂时允许相关顶点的出度超出最大度数上限R。

3.3 ParlayANN

ParlayANN是对 Vamana 构建算法的扩展，它采用批量并行插入机制，解决了并行图构建中一个核心的可扩展性瓶颈。原始的 DiskANN [18] 算法虽支持并行插入点，但依赖细粒度锁来更新邻接表。在实际应用中，这种设计会在高中心度顶点（如数据集中心点）上产生严重的资源竞争。

在贪心搜索过程中，高中心度顶点会被频繁遍历，而并发插入操作会同时尝试向这些顶点添加边，最终导致执行串行化，限制了并行可扩展性。ParlayANN 摒弃了基于锁的更新方式，改用无锁、批量并行的构建策略，从而消除了这一瓶颈。

其核心思想是将图遍历与图修改解耦，让高开销的更新操作能够无竞争地并行执行。如算法 3 所示，ParlayANN 将图构建重构为两个相互独立且可并行的阶段：

第一阶段，对每个待插入顶点独立并行执行束搜索（算法 1）。所有搜索操作均基于当前图的只读快照运行，并生成遍历过程中遇到的顶点所对应的候选边集合。关键在于：本阶段不修改任何邻接表。所有批量生成的候选边不会立即执行更新，而是被写入一个临时边集合中。

随后对该集合执行半排序操作 —— 按目标顶点对边进行分组，其开销远低于全排序。半排序能够保证指向同一顶点的所有候选边集中存放，使得后续处理可以按顶点独立执行。

第二阶段，以批量并行的方式执行鲁棒剪枝（算法 2）。为每个接收到新候选边的顶点分配一个独立线程，该线程将新边与现有邻接表合并，并执行鲁棒剪枝以满足最大出度约束 R。

由于指向同一顶点的所有边仅由单个线程处理，本阶段无需加锁或跨线程同步。这种无锁设计消除了中心顶点的竞争问题，让 ParlayANN 能够高效利用多核并行能力。

其核心设计理念是：决定最终邻域结构且占据主要构建开销的剪枝步骤，可以通过延迟更新 + 批量并行处理，与并发图遍历操作完全解耦。最终，ParlayANN 在保留 Vamana 图结构特性、维持相近查询性能与精度的前提下，实现了大幅提速的索引构建。

无锁化设计优势：无热点竞争、并行性强、适用于大批量处理
劣势：有批次延迟、不适合实时单点插入

3.4 为什么Vamana适合用于GPU

相较于众多其他近似最近邻（ANN）索引，Vamana 图在索引构建与查询执行两个阶段都具备高度并行性，能够充分发挥现代 GPU 的硬件算力。
其建图过程天然支持并行：各顶点可独立处理，且每一步流程逻辑规整，能够最大程度减少线程分支发散。距离计算与图遍历环节均具备细粒度并行潜力。
Vamana 同时支持高效的增量更新：每次插入仅在少量顶点范围内执行有界束搜索，且最多新增 2R 条边，更新开销可控且可预测。
相比之下，NN-Descent 等算法的更新代价要高得多，这类算法需要多轮迭代优化，且遍历过的任意节点都有可能成为新邻居。
已有研究表明，在各类数据集上，Vamana 的检索效果均可持平甚至优于其他图索引算法，是构建高性能、支持动态更新近似最近邻检索系统的理想基础架构 [18]。

四、Jasper：一个GPU原生的ANNS系统



本节介绍 Jasper 系统的整体设计。基于 GPU 的近似最近邻检索（ANNS）核心难点在于如何在不规则、数据依赖、访存密集型的负载下实现高硬件利用率。现有各类系统往往会牺牲计算效率或内存带宽，而 Jasper 可同时兼顾二者。本文重构了贪心束搜索算法，围绕 GPU 执行约束重新设计算法逻辑与数据结构，充分挖掘并行度以打满计算单元，同时采用合并访存模式，使实际内存带宽逼近硬件峰值。本文还针对 GPU 执行重新设计了批量并行建图算法，实现了无锁流式插入。我们评测了建图吞吐随批次大小和索引规模的扩展性能，实验表明：即便索引规模增长一个数量级，Jasper 仍能维持高插入速率。

4.1 单查询线程块执行模型

Jasper 的核心设计思路是：为每一条查询向量或待插入向量分配一个完整的 CUDA 线程块。这种单查询独占线程块执行模型，能够精细控制束搜索各阶段的并行调度，并适配 GPU 架构约束。
距离计算等计算密集阶段可在线程块内所有线程间做全并行；而候选扩展、图遍历等分支不规则阶段则做适度串行化，以抑制控制流分支发散。这种混合执行结构，既能提升硬件利用率，又能适配图结构 ANN 检索天然的不规则访存特征。
由于单个查询全程在同一个线程块内执行，可将束搜索边界集、候选缓冲区、中间距离值等运行时数据全部存放于共享内存。这大幅减少全局内存访问流量，同时便于线程间高频复用数据。

但线程块大小（单个 CUDA 块的线程数）的选择，会在单查询计算吞吐与内存级并行度（MLP）之间形成固有权衡：
线程块越大，为单个查询分配更多线程，提升算术吞吐、指令级并行度与寄存器复用率；
线程块越小，可同时并发执行更多查询，增加未完成内存请求数量，更好掩盖全局内存访问延迟。

该权衡的取舍高度依赖数据集特征：
低维向量场景下，距离计算开销远小于内存延迟，此时提升内存级并行度收益占优。这类内存受限负载中，减小线程块大小可让每个流式多处理器（SM）容纳更多并发查询，即便单查询计算资源变少，整体有效吞吐仍会提升。例如在 BigANN 数据集上，向量仅 128 字节，实验表明：在单 SM 最大可容纳 32 个线程块的占用上限内，采用更小线程块始终更优。
而在 Gist 等高维数据集上，距离计算成为性能瓶颈。实测显示，Gist 向量的点积计算耗时是内存操作的 17.6 倍。这类计算受限负载下，减小线程块会直接降低性能：限制单查询并行度、减少寄存器复用、拉低算术吞吐。当计算开销占主导时，提升内存级并行度几乎无收益，过小的线程块反而会拖累整体吞吐。
因此，最优线程块大小与数据集强相关，且取决于距离计算内核的计算–内存开销配比：内存受限负载适合高并发与高内存级并行，计算受限负载则需要为单个查询分配尽可能多的计算资源。

距离计算优化
我们发现，平方根运算耗时占欧氏距离总计算耗时的 40% 以上。由于平方根在正实数域单调递增，可省略开方操作，直接比较距离平方值，在保证检索结果正确的同时大幅降低计算开销。

内核融合优化束搜索
进一步通过内核融合优化束搜索：将距离计算、边界集排序、邻居扩展多个阶段合并为单个 GPU 内核，由同一个线程块完整执行。
融合后省去多次内核启动开销，且所有中间数据可常驻共享内存，加速访存效率。内核融合是 GPU 通用优化手段，CAGRA [28] 也采用了该优化方式。

4.2 面向 GPU 的束搜索优化

由于 CPU 与 GPU 并行机制存在差异，本文 GPU 版本的束搜索与 ParlayANN 的束搜索实现有所不同。在 GPU 上运行束搜索时，会受执行模型与硬件资源总量的限制。本节介绍为适配 GPU 所做的各项取舍与优化设计。

ParlayANN 中的束搜索对每条查询均采用串行执行方式，因为 CPU 并行规模有限，只能为每条查询分配单个线程。而 GPU 的线程数量高出一个数量级，要充分占满 GPU 硬件资源并实现高性能，必须对束搜索内部各执行环节做细粒度并行化。

哈希表方案差异
两类实现的一大区别源于 ParlayANN 在束搜索中使用有损哈希表记录已访问节点、剔除重复顶点。原始算法采用固定大小（512 槽位）的哈希表，采用直接替换策略来记录已遍历顶点并做重复过滤。在 CPU 上，该结构由单线程串行访问，执行行为具备确定性。

但在 GPU 上，如果沿用同样的访问方式，就需要在协作线程间串行执行哈希表操作，会严重吞吐性能。将哈希表操作并行化虽然能提升性能，却会引入执行不确定性。此外实验发现，该哈希表在 GPU 场景下对检索精度和查询吞吐并无明显正向作用。因此去掉哈希表可以进一步降低共享内存占用，让 Jasper 每个线程块能够调度更多查询任务。

候选归并次序差异
Jasper 在束搜索中采用了不同于 ParlayANN 的候选归并次序。ParlayANN 实现了延迟归并优化：当候选列表规模小于束宽固定比例时，会延后排序与归并操作。在 GPU 非确定性哈希表的影响下，无法保证该机制每次都生成相同顺序的候选列表。

实验表明，若同时保留有损哈希表与延迟归并机制，会明显损害索引质量与查询精度。因此 Jasper 直接舍弃这两项设计，既简化了控制流、保证执行确定性，也消除了一处重要的线程分支发散来源。

GPU 共享内存容量限制
共享内存的资源约束进一步放大了 GPU 与 CPU 实现的差异。多线程协作执行需要为每条查询维护独立的共享内存数据结构，这会降低硬件占用率，削弱 GPU 掩盖内存延迟的能力。

本文主动接受这一取舍：收紧单查询共享内存配额、容忍更高的线程束内分支发散，以此换取更好的数据局部性与复用率，这对不规则的近似最近邻检索负载至关重要。如 6.5 节屋顶线模型分析所示，该设计仍具备很强竞争力；若再结合 Jasper RaBitQ 中的查询重排序策略，还能显著提升运算强度。

向量加载优化

为进一步榨取性能，本文对向量数据的加载方式进行优化。默认情况下，当线程束访问大型向量数据时，编译器会让单个线程连续发起多次 16 字节加载请求，难以掩盖内存延迟。反之，按元素分散加载虽能把请求分摊到多个线程，却会降低有效带宽。

本文的解决思路是：将每个向量切分为若干 16 字节的数据块，为每个线程分配一个数据块。各线程并行加载查询向量与候选向量的数据块，在本地计算部分距离分量，循环处理完整个向量后，再在线程块内做归约求和，得到最终距离值。该方式可同时发起多路符合合并访问规则的 16 字节加载，带宽利用率接近硬件峰值。

如图 4 所示，该优化在延迟敏感场景下收益明显。束宽较小时检索属于延迟受限负载，优化后可实现 14% 的加速；束宽较大时吞吐成为瓶颈，优化加载可在不损失精度的前提下持平基线吞吐。该优化对浮点向量与 RaBitQ 量化向量均有效。结合后续第 5 节介绍的 RaBitQ 向量量化方案可减少最高 50% 的数据传输量，加载优化能够为束搜索端到端性能带来可观提升。

4.3 GPU 上的批量构建

为在 GPU 上实现高性能的近似最近邻检索索引构建，我们采用了 ParlayANN [23] 提出的批量构建方法。ParlayANN 通过 ParlayLib [6] 实现分支合并并行模式，并在插入前使用半排序对边进行分组，从而支持无锁并行更新。具体来说，ParlayANN 使用分支合并并行将较大的任务（如半排序）拆分为更小的任务，并分配多个线程执行。然而，GPU 并不支持这种程度的动态并行，因为每个内核只能启动固定数量的线程。此外，我们希望尽量减少内核启动次数以避免启动开销。因此，我们将不同的构建步骤融合成独立的 GPU 内核（见图 2）。虽然这降低了一定的调度灵活性，但能够在遵循 GPU 执行模型的同时，保留批量并行搜索与剪枝带来的可扩展性优势。

GPU 高效剪枝。我们为鲁棒剪枝实现了一种混合策略。将束搜索生成的出边与需要剪枝的现有边汇总，并存储在统一的缓冲区中。该缓冲区使用 CUDA Thrust 库按照顶点编号和距离进行完全排序。与之不同，ParlayANN 使用半排序，之后在鲁棒剪枝内部用单个线程按照距离对邻居排序。我们发现，由于 GPU 单个线程性能较弱，不均匀的任务分配代价较高，因此完全排序能在 GPU 上获得更好的负载均衡。随后，为每个边列表分配一个完整的 1024 线程流式多处理器执行剪枝操作，因为该阶段以距离计算为主，高度并行化可以带来性能收益。

4.4 更新批次大小的可扩展性


为在索引构建过程中充分利用现代 GPU 架构提供的内存容量，我们根据数据集大小和图的维度选择不同的最大构建批次大小。Jasper 构建图索引时，向量数据和图索引均存储在 GPU 的设备内存中。该设计使索引能够完全在 GPU 上构建，避免主机与设备内存之间缓慢的 PCIe 数据传输。因此，剩余的设备内存可作为构建算法的可用工作空间。

以 NVIDIA A100 GPU 为例，其提供 80GB 的设备内存。对于 1 亿规模的 BigANN 数据集，数据集与图索引的总大小为 38.5GB，剩余 41.5GB 可用内存用于图构建。对于 1 亿规模的 Deep 数据集，总大小为 64.1GB，剩余 15.9GB 可用内存。在实验中我们发现，索引构建的最优批次大小为：BigANN 每批次 100 万个向量，Deep 每批次 40 万个向量，对应表 1 中所示的可用内存容量。

五、向量量化

现代嵌入模型生成的向量维度可达数百至数千维，单个向量的大小从 96 字节到 6000 字节以上不等，如 BigANN 基准测试所示 [33]。由于近似最近邻检索（ANNS）是内存受限型任务 —— 每个向量元素仅被读取一次、用于一次算术运算后即被丢弃 —— 因此减小向量大小可以直接提升查询吞吐。为在保留检索精度的同时降低单向量存储与 I/O 开销，我们引入向量量化技术，在近似保留距离信息的前提下压缩向量。

近似最近邻检索领域应用最广泛的量化方法是乘积量化（PQ）[20]。乘积量化将每个向量划分为 K 个互不重叠的子向量，并使用包含 256 个质心的小型码本对每个子向量独立量化。每个子向量最终编码为一个单字节值，代表其最近的质心，由此将高维向量压缩至仅 K 字节。距离计算通过预计算的查找表完成，该表存储了每个子空间内质心间的距离。这种方法能实现显著的压缩效果，且精度损失有限，误差仅来源于每个子向量被其最近质心近似表示的过程。

尽管乘积量化能有效压缩向量，但其查找表的访问模式在 GPU 上会导致吞吐率下降。GPU 内存以 32 字节扇区为单位进行原子加载 —— 读取单个 4 字节距离条目会浪费剩余 28 字节，造成 8 倍的读放大。直接的解决方案是将查找表存入共享内存，但即使是中等规模的配置（K=32）也需要 32×256×256×4 字节（8MB），远超共享内存容量。我们在 Jasper 中实现了乘积量化，但实验发现，在所有测试配置下，其吞吐率均低于未量化的检索方案。已有研究也得出了类似结论，指出乘积量化在 GPU 上的性能存在局限 [28]。

5.1 RaBitQ 量化


想要在 GPU 上高效使用量化技术，必须避免距离计算过程中的随机内存访问。以往 GPU 上的量化实现都陷入一种固有权衡：依靠更小的向量体积、更低的内存 I/O 为代价，换来检索吞吐性能的下降 [28,39]。本文证明这种权衡并非必然存在。Jasper 采用了RaBitQ量化算法 [11]，该方法天然适配 GPU 硬件特性。

RaBitQ 仅需顺序内存访问和简易算术运算，且完全不需要查找表。它通过量化后的数据向量与查询向量做内积运算来估计向量距离。表 2 给出了相关变量与量化向量的计算方式。

Jasper 完整集成了 GPU 加速版 RaBitQ：对于 8 比特向量可实现 2 倍压缩，对 32 比特向量最高可达 8 倍压缩。在高维数据集上，实验结果表明，搭载 RaBitQ 的 Jasper 性能显著优于采用原始精确距离计算的版本。本文将在 6.6 节对 RaBitQ 与 PQ 的性能做详细对比分析。

六、评估

本节将把 Jasper 与四种主流前沿近似最近邻（ANNS）索引算法进行性能对比，其中包含三种 GPU 方案和一种 CPU 方案。实验在五组规模、维度各不相同的真实数据集上，分别评测索引构建、查询处理与增量更新性能。同时采用屋顶线模型评估 Jasper 对 GPU 硬件资源的利用率。最后通过一系列微基准测试，分析 Jasper 各项 GPU 专属优化带来的效果，并讨论这些优化思路如何推广应用到其他 GPU 加速系统中。Jasper 的源码及基准测试部署指南已开源，项目地址：https://github.com/saltsystemslab/JasperGPUANNS。

6.1 实验环境设置

本文从三个维度对所有系统进行评测：

  • 构建吞吐：完成后续实验所用索引整体构建的总耗时。
  • 查询吞吐：每秒可完成的近似最近邻查询数量。
  • 召回率：衡量 Top-K 检索结果的精度，定义为 召回率 @k = 真实精确 Top-K 结果命中数 / K。实验评测了 1@1、10@10、50@50、100@100 这几种召回指标。

硬件环境：
所有 CPU 实验基于英特尔至强 E5-2680 v4 处理器，14 物理核（28 超线程），主频 2.40 GHz。GPU 实验使用 NVIDIA A100（SM80），搭载 80GB 高带宽显存，CUDA 版本为 12.9。

数据集：
评测所用数据集均取自 Big_ANN_Benchmarks 基准库，各项统计信息汇总如表 3 所示：

  • BigANN：1 亿条 128 维 uint8 向量，采用欧氏距离，源自 SIFT 数据集，测试集包含 1 万条查询向量。
  • Deep/Yandex：1 亿条经 PCA 降维、归一化的 GoogLeNet 嵌入向量，96 维 float32，采用欧氏距离，含 1 万条查询向量。
  • Gist：100 万条 960 维 float32 向量，采用欧氏距离，测试集含 1000 条查询向量。
  • OpenAI-Arxiv：230 万条 1536 维 float32 向量，为 arXiv 论文嵌入向量，查询集规模 1 万条。
  • Text2Image：1000 万条 200 维 float32 图像嵌入向量，源自 Se-ResNext-101 模型，采用最大内积检索（MIPS），测试集含 1 万条查询向量。

对比的 ANNS 索引方案：

  • CAGRA [28]：当前 GPU 索引构建领域的前沿方案。CAGRA 基于 NN-Descent 算法构建图索引。NN-Descent 通过两跳邻居不断扩充各节点邻域，迭代优化图的连通性；生成的图连通性好但过于稠密，超出 GPU 显存承载能力。CAGRA 采用三阶段剪枝算法：按边的长度排序，剔除存在优质两跳替代路径的冗余边，最后将原图与反向图合并后再次剪枝，得到最终索引。
  • BANG [39]：在 DiskANN 基础上向 GPU 扩展查询执行模块。BANG 不负责索引构建，需要依赖 DiskANN 预先建好的 Vamana 图结构。该方案采用乘积量化压缩向量，量化向量存放于 GPU 显存，原图结构与全精度向量存储在内存。每条查询独占一个完整流式多处理器（SM），距离计算仅依赖 PQ 码表查表，为每个距离计算分配一个独立线程。为提升查询效率，采用布隆过滤器实现已访问节点标记列表，降低检索过程的内存开销。
  • GANNS [45]：面向 GPU 优化的 HNSW 图算法实现。为适配 GPU 执行特性，GANNS 采用惰性数据结构：不维护传统优先队列，仅保留边界集与已访问列表，按需更新列表来标记已遍历节点。查询阶段不使用哈希表做去重标记，且每条查询可分配多个线程块执行。由于 HNSW 建图难以并行化，GANNS 采用并行构建大量子图、再串行合并的方式；该方案无法支持 1 亿规模的大数据集。
  • ParlayANN：Vamana 建图算法的前沿代表，实现了批量并行构建策略，相关原理在 3.3 节有详细介绍。

所有对比方案统一固定图度数上限 R=64，即每个节点最多保留 64 条出边。

6.3 查询性能


本节对比 Jasper 与其他近似最近邻索引的检索性能。
低维向量数据集：BigANN、Deep
在低维向量数据集上，Jasper 在所有可达到的召回率下均为性能最快的索引，召回率 0.31 时峰值查询吞吐可达每秒 1320 万次查询。
在 BigANN 上，Jasper RaBitQ 吞吐位列第二。Jasper 在 BigANN 上最高只能达到 0.9943 的召回率，在此召回率以内均保持检索速度最优。
在 Deep 数据集上，所有测试召回率档位下 CAGRA 均为第二快。CAGRA 在这两个数据集上能达到更高的精度上限：BigANN 1@1 最高召回 0.9999，Deep 1@1 最高召回 0.983。
依托优化的向量加载机制，Jasper 在低维向量场景下检索速度优势明显。当束宽较小时，单次向量计算工作量偏少，延迟掩盖效果有限；此时优化向量加载延迟能够显著提升内核性能，减少 GPU 计算单元的等待耗时。

小结
通过尽可能剥离共享内存占用，Jasper 可在每个流式多处理器上同时运行更多查询。结合面向延迟优化的向量加载策略，使其在小向量数据集、全束宽配置下都具备业界顶尖的查询性能。

高维向量数据集：Gist、OpenAI
在 Gist、OpenAI 等高维向量数据集上，距离计算成为运行耗时的主体。为量化各操作开销，我们在检索内核中加入时钟周期计数。
分别运行 Jasper 精确距离版本在 BigANN 与 Gist 上测试发现：Gist 的距离计算时钟周期开销高出 17.8 倍，距离计算成为整个检索流程中最耗时的环节。
这类数据集向量尺寸大，量化技术可有效降低距离计算的计算量与内存开销。在高维数据集上，Jasper RaBitQ 在所有支持配置下均为最快算法，峰值吞吐超每秒 300 万次查询，50@50 召回率最高可达 0.94。
在召回率–吞吐权衡曲线上，Jasper RaBitQ 最优，其次是 CAGRA，之后依次为 Jasper 精确版本、ParlayANN 与 GANNS。
受海量数据传输与密集距离计算影响，所有索引在高维数据集上性能均明显下降。量化可将向量体积压缩，使内存传输量降低 8 倍。
图 12 结果验证了本文观点：在 GPU 上想要发挥量化性能优势，必须支持量化向量直接做距离比对。所有参与对比的 CAGRA+PQ 版本均未获得性能提升；而 Jasper RaBitQ 相比精确距离版本最高可提速 60%。

小结
高维向量场景下，量化带来的 8 倍内存传输缩减可提升运算强度与查询吞吐。但该加速效果仅在量化向量可直接参与距离比对时才能落地；PQ 虽同样能压缩内存，却无法实现有效性能提速。

Text2Image 数据集

Text2Image 采用最大内积检索（MIPS） 而非欧氏距离，给索引构建与查询带来更大难度。MIPS 不满足三角不等式，无法构成度量空间。
三角不等式是度量空间的必要条件，也是 Vamana 算法构建优质检索图、执行鲁棒剪枝的关键前提。缺少该性质时，向量邻居之间无距离相关性：某一向量可能与查询目标距离很近，却和它的所有邻居距离都很远，导致剪枝策略失效。
为解决该问题，主流近似最近邻索引会通过增加一维将 MIPS 转换为欧氏空间，从而支持建图，但代价是向量维度增加一维。
Jasper 及 Jasper RaBitQ 在该数据集上性能均有所下降：MIPS 转欧氏距离的处理方式会损失精度；而 CAGRA 可直接基于原始 MIPS 距离建图，因此精度表现更优。
召回率 0.9 以内时，Jasper 拥有最优的召回–吞吐权衡；超过 0.9 后 CAGRA 表现反超。
由于该数据集维度仅 200 维偏低，Jasper 精确距离版本性能优于 Jasper RaBitQ；同时受 MIPS 距离特性影响，Jasper RaBitQ 的召回率也有所下降。

小结
MIPS 不满足度量空间性质，而 Vamana 的鲁棒剪枝依赖三角不等式，因此难以构建高质量检索图。Jasper 在大部分束宽下仍保持吞吐领先，但高召回率场景不及 CAGRA；原因在于 NN-Descent 建图不依赖度量空间，更适配 MIPS 场景。

6.4 RaBitQ 量化效果

Jasper RaBitQ 能够减少查询过程中加载的数据量，从而在高维向量数据集上提升性能。如图 8 所示，在 Gist 和 OpenAI 数据集上，采用精确距离计算的原版 Jasper 速度要慢于 CAGRA；但引入 RaBitQ 后，Jasper RaBitQ 在相同召回率下抹平了性能差距并反超 CAGRA，即便它采用的是有损向量表示。例如在 OpenAI 数据集 10@10、召回率 90% 的条件下，Jasper RaBitQ 的速度是原版 Jasper 的 3 倍。

对于向量维度更低的 Deep 数据集，Jasper RaBitQ 与精确距离版本性能基本持平。

在 BigANN、Text2Image 这类低维向量数据集上，由于有损近似带来的微小精度偏差，以及额外的量化运算开销，Jasper RaBitQ 速度会略慢于精确距离计算版本。但在显存受限的场景下，RaBitQ 仍具备很高实用价值：它可以降低 GPU 显存占用，让更多向量能够常驻 GPU 显存。本文将在 6.6 节把 Jasper RaBitQ 与其他量化方法进行对比评测。

本文采用屋顶线模型分析 Jasper 在低维向量数据集（Deep）与高维向量数据集（Gist）上的性能特征。图 9 对比了检索个数 K=10 时，Jasper、Jasper RaBitQ 与 CAGRA 的运算强度及实际吞吐性能。

整体结果表明，RaBitQ 采用的查询重排序与查询侧计算策略，能够通过减少内存数据搬运来提升整体性能。

小结
Jasper 与 CAGRA 均为内存受限型负载，屋顶线曲线贴近理论上限，说明内核已接近理想执行性能。量化技术可显著提升检索内核的运算强度，也意味着量化版近似最近邻检索仍有较大的性能优化空间。

线程块越小，可同时执行的查询数量越多，提升内存级并行度。

对于 Gist、OpenAI 等高维向量数据集，减小线程块大小性能会持续提升，直至每块 128 线程；若继续减小块规模，性能反而下降。原因是高维场景下距离计算成为瓶颈，线程数量过少无法高效处理高维向量运算。
对于 BigANN、Deep 等低维向量数据集，距离计算不是瓶颈。更小的线程块可以提升内存并行度，性能持续变好。最优配置为每块 32 线程（每条查询占用一个线程束），每个流式多处理器可同时并发 32 条查询，最大化内存吞吐。

实验结果证实：在 GPU 负载场景下，RaBitQ 的实际效果优于乘积量化 PQ。

七、相关工作

本节介绍本次实验未直接对比的主流近似最近邻检索（ANNS）相关工作。这类算法主要分为三类：
1）性能已被当前最新方案超越；
2）无公开可复现代码实现；
3）与近似最近邻检索技术互补、不具备直接可比性。

FAISS
FAISS 是面向向量相似度检索与聚类的开源工具库，集成了大量精确及近似最近邻算法，同时提供高效的 CPU 与 GPU 实现。FAISS 支持多种索引策略与技术，包括倒排索引、乘积量化、三级量化、HNSW、各向异性向量量化、NSW 以及二进制多索引哈希等。在 GPU 端，FAISS 利用大规模并行能力加速索引构建与查询处理，是大规模向量相似度检索场景的常用方案。

SONG
SONG 是面向 GPU 的近似最近邻检索方案，采用固定秩检索图处理查询任务。该算法依托批量计算、优先队列与布谷鸟过滤器实现高效查询处理。为提升性能，仅保留距离小于当前 Top-K 结果的候选节点，减少需要维护的节点数量，但代价是需要额外重新计算距离带来计算开销。同时对哈希表与过滤器做优化：剔除已不在当前 Top-K 范围内的索引键值，这类节点距离查询向量过远，不会再次加入搜索路径。

GTS
GTS 是基于树形结构的 GPU 精确 K 近邻检索索引。该树结构采用批量同步算法构建，除主体树形结构外，还将每个向量的节点值存入数据表。通过 FFT 支点选择策略将数据对象映射到树节点，并对列表进行划分，使设备内存中的对象物理地址连续排布。该划分过程迭代执行，直至所有叶子节点规模足够小。
系统采用批量缓存列表暂存流式更新数据，累积到一定规模后整体重新构建整棵树。查询阶段采用迭代范围查询的批量算法：维护距离最近的 Top-K 节点列表，逐层遍历子节点；若某层节点距离超出当前 Top-K 最大距离，则直接剪枝，迭代细化搜索直至叶子节点，最终筛选出 Top-K 结果。
更新时，新增数据暂存于缓冲区，每次查询都会检查缓冲区；若缓冲区数据量过大，则重新生成整棵索引树。

NEOS
NEOS 是一套向量检索工具库，可在 CPU 用户态内存中锁定 GPU 显存，并提供多项 GPU 缓冲区优化手段，包括 NVME 内存锁定、自定义 CPU–GPU 内存拷贝、更快的 CPU–GPU 轮询机制等。NEOS 本身并非向量索引算法，而是缓冲区工具，可被 HNSW 等索引复用。
NEOS 使用大型无序缓冲区聚合请求，依托 A100 的并行算力与 cuBLAS 矩阵运算，以低延迟批量校验缓冲区中的全部请求。该设计让仅支持批量插入接口的向量管理系统，也能适配流式更新场景，在不损失吞吐的前提下提升应用灵活性。

FusionANNs
FusionANNs 是一种协处理器架构的向量索引，将 GPU 作为协处理器，承担开销较大的距离比对计算。该索引基于 SPANN 的倒排链表结构设计，研究者发现原版 SPANN 的可扩展性瓶颈在于从倒排链表筛选 Top-K 时的距离计算。
因此将向量的乘积量化版本常驻 GPU 显存，倒排链表通过分层均衡聚类生成。查询时由 CPU 遍历图结构，筛选出与查询向量最接近的若干倒排链表；将链表内向量 ID 下发至 GPU，由 GPU 并行计算链表中所有向量的距离。计算结果回传 CPU，通过启发式重排序筛选优质候选向量，再从固态硬盘加载全精度向量，执行精确距离比对得到最终检索结果。该项目暂无开源代码。

LoRANN
LoRANN 提出一种名为降秩回归的新型聚类方法，可直接替代乘积量化，既能用于聚类中心生成，也可在倒排索引中完成聚类链表的查询检索。
降秩回归以全批量方式处理数据，优化目标为最小化 βRRR​=argminβ​∥Y−Xβ∥F2​，其中 Y 代表真实近邻结果。算法采用快速随机算法求解训练集近邻，并通过矩阵乘法 y^​=xTβRRR​ 完成预测。
为提升运算效率，先通过 PCA 特征提取将训练集映射到低维空间，再采用 8 位整型量化压缩。整套流程可在 CPU 与 GPU 上高效执行。实验表明，该方法可替换 FAISS 中的乘积量化，在高维向量数据集上获得性能提升。

八、结论

本文研究表明，将近似最近邻检索（ANNS）部署到 GPU 平台能够带来显著的性能提升，不仅可以实现高吞吐的查询处理，还能与后续机器学习负载高效共置部署。
然而，现代 GPU 极强的并行能力也带来了新的挑战：GPU 上的近似最近邻检索算法往往难以发挥硬件极限性能，主要受制于同步开销与有限的共享内存资源。实验结果表明，对检索内核进行精细化精简设计、降低共享内存占用，能够显著提升查询吞吐并实现更高的并行度。
研究还发现，CAGRA 等当前业界领先的 GPU 近似最近邻检索内核，运行性能已接近硬件屋顶线理论上限，意味着单纯依靠计算层面的优化，性能收益会越来越有限。在束宽为 256 的配置下，Jasper 的每秒浮点运算量比 CAGRA 高出 50%，但仅达到该运算强度下理论峰值吞吐的 69%，说明该检索内核后续最多还有 1.44 倍的性能提升空间。
