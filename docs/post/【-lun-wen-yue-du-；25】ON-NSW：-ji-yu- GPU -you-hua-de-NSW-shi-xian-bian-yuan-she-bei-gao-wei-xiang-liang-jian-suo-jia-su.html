【论文阅读；25】ON-NSW：基于 GPU 优化的NSW实现边缘设备高维向量检索加速

ON-NSW: Accelerating High-Dimensional Vector Search on Edge Devices with GPU-Optimized NSW

摘要

工业物联网（IIoT）越来越依赖向量嵌入技术，用于异常检测、预测性维护、传感器融合等数据分析与人工智能落地应用，而高效的近似最近邻检索（ANNS）是上述业务的关键支撑。基于图结构的算法是近似最近邻检索的主流方案，但层级导航小世界算法（HNSW）等现有图检索算法大多面向高端服务器 CPU 开发，并未适配边缘设备的硬件特性。
本文提出ON-NSW：一款面向边缘设备、基于 GPU 优化的 HNSW 改进算法。该算法摒弃 HNSW 分层结构，采用单层图拓扑以充分发挥 GPU 并行算力；同时结合英伟达 Jetson Orin Nano 的统一内存架构精细化部署索引数据，引入线程束粒度并行邻域遍历与轻量化同步机制，缩短检索时延。
基于真实高维数据集的实测表明：在 Jetson 边缘硬件上，ON-NSW 检索吞吐量相较原生 HNSW 最高提升1.44 倍，且召回率基本持平。实验证明，ON-NSW 可在嵌入式边缘设备上实现高效、高吞吐的向量检索。

一、引言

物联网时代，传感器数据量呈爆发式增长。各行各业布设海量传感器以优化生产流程，采集的数据可支撑预测性维护、异常检测、智能自动化等工业场景应用。例如在预测性维护中，通过分析涡扇发动机传感数据预估设备剩余使用寿命，合理安排检修；异常检测依托心跳、发动机时序信号挖掘设备运行异常特征；智能制造利用温度、振动等传感数据，实时自动调控设备运行参数与维保计划。

想要充分挖掘传感数据价值，需要低时延、高吞吐的实时检索与分析。传统方案大多将数据上传云端运算，但海量数据传输带来通信开销，不仅拖慢处理性能，还存在数据隐私泄露风险。

为解决云端方案弊端，工业物联网采用端侧本地计算成为可行路线：数据就地分析检索，无需上云，既降低延迟又保护隐私。以英伟达 Jetson 为代表的嵌入式硬件搭载高性能 GPU，可在边缘侧高效运行机器学习与向量检索算法。

在工业物联网数据分析中，近似最近邻搜索（ANNS）是解析传感器高维特征向量的基础算子，支撑各类业务的相似度查询。现有 ANNS 算法分为三类：树索引、哈希索引、图索引。KD-Tree 等树算法在低维向量表现优异，高维场景效率骤降；哈希算法性能受哈希函数约束；NSW、HNSW 等图索引因支持向量动态增删、检索高效，落地最广泛。

但现有 ANNS 算法原生面向高端服务器 CPU 开发，难以直接部署在边缘设备。边缘设备 CPU 核心少、算力有限，且 CPU 与 GPU 共用统一内存架构，需要定制化算法适配硬件特性。

本文提出ON-NSW，一款专为 Jetson 系列边缘设备定制的图结构 ANNS 算法，适配边缘 GPU 存储层级，最大化利用硬件算力提升高维向量检索速度。实测在高维数据集下，其吞吐量相较 CPU 版算法最高提升 1.44 倍。

本文主要贡献：

  1. 以 GPU 为核心重构 NSW，剔除 HNSW 上层索引，全部检索逻辑下沉至 GPU 单层图执行，适配端侧近似检索；
  2. 依托 Jetson 统一内存架构实现锁页内存零拷贝，结合 GPU 共享内存优化检索效率；
  3. 实现并行邻域遍历，利用多 GPU 核心同步读取邻接节点、并行距离计算，加速检索。

论文后续章节安排：第 2 章介绍相关技术背景，第 3 章详述算法设计，第 4 章展示实验结果，第 5 章梳理相关研究，第 6 章总结全文。

二、背景

本节简要介绍本文相关研究背景与基础概念：首先阐述近似最近邻搜索（ANNS）及其主流算法分类，随后说明作为本文算法设计基础的层级导航小世界图（HNSW），最后介绍英伟达 Jetson 设备的 GPU 架构，为本文面向 GPU 优化的 ON-NSW 算法铺垫硬件背景。

2.1 近似最近邻搜索（ANNS）

图像、视频等非结构化数据通常经由 AI 模型转化为高维向量完成结构化管理。在海量高维向量数据集场景中，查询相似向量是推荐系统、图文检索、异常检测等业务的核心环节，直接决定应用性能。k 近邻检索（k-NNS）遍历库内全部向量，逐一计算与查询向量的距离并返回距离最近的 Top-k 结果。该算法检索结果精准，但随数据量与向量维度升高，距离计算开销急剧膨胀，无法满足实时检索需求。

为此，近似最近邻搜索（ANNS） 通过小幅牺牲检索准确率换取运算效率提升：不精准定位严格近邻，而是筛选近似邻近向量。检索优劣以召回率衡量，即真实近邻命中在检索结果中的占比。

ANNS 算法分为树型、哈希型、图型三类：

  1. 树基算法（KD-Tree、Ball-Tree）通过空间划分缩减检索范围，低维向量表现优异，维度升高后效率断崖式下滑；
  2. 哈希算法（LSH、多探测 LSH）将相似向量映射至同一哈希桶实现快速查找，但检索精度高度受制于哈希函数设计；
  3. 图基算法（NSW、HNSW）以向量为节点、向量相似度为边构建拓扑图，兼顾高检索速度与高准确率，高维场景稳定性突出，是当下学术界研究与工业落地最热门的技术路线。

2.2 层级导航小世界图（HNSW）


基础导航小世界图（NSW）为单层拓扑结构：向量作为节点，节点相似度用边连接。检索从指定入口点出发，贪心遍历不断靠近查询向量的邻点。但数据集扩容后检索路径变长，检索效率严重依赖入口节点选取。

HNSW 通过分层图结构优化上述缺陷：插入向量时随机分配至若干层级；上层节点稀疏互连，用于快速逼近查询位置；下层节点稠密相连，实现精细化近邻查找。节点数量自顶层至第 0 层呈对数增长，第 0 层存储全量原始数据。检索从最高层逐层向下跳转：在稀疏高层跨远距离节点快速锁定本层最优节点，作为下层检索入口，层层收敛逼近目标向量。

相较单层 NSW，分层架构缩短检索路径、提升检索精度。但 HNSW 原生面向高端服务器 CPU 设计，边缘设备算力受限、内存架构特殊，直接部署落地困难。下文以典型边缘硬件 Jetson 为例，介绍其 GPU 与内存架构。

2.3 英伟达 Jetson 设备的 GPU 架构

英伟达 Jetson 是搭载集成 GPU 的典型边缘硬件平台。该设备包含两组处理器插槽，每组插槽集成 3 颗 ARMv8.2 架构的 Cortex‑A78AE 处理器内核；每个内核配备 64KB 一级缓存与 256KB 二级缓存，单插槽配置 2MB 三级缓存。整机搭载 8GB 位宽 128 比特的 LPDDR5 内存。其 GPU 采用安培（Ampere）架构，共计 1024 个 CUDA 核心，划分为 8 个流式多处理器（SM）；单个流式多处理器最多可同时调度 1536 个线程，且配备 164KB 片上 SRAM，该存储空间由一级缓存与共享内存共用。整块 GPU 共享一片大小为 2MB 的二级缓存。

如图 1 所示，与搭载独立显卡的常规服务器不同，Jetson 采用统一内存架构：GPU 全局内存与 CPU 主机内存共用同一块物理 LPDDR5 存储空间。CPU、GPU 可经由片上系统总线与内存控制器直接访问同一片内存，省去了服务器场景下必不可少的主机向设备的数据拷贝开销。除此之外，GPU 能够以零拷贝方式对主机的页锁定（Pinned）内存区域执行直接内存访问（DMA）。在服务器平台中，该类 DMA 操作依托 PCIe 总线实现，延迟偏高，而 Jetson 规避了此项开销。

但受操作系统限制，被标记为页锁定的内存页面无法被换出，导致 CPU 无法对其使用常规缓存策略。因此从 CPU 视角访问 Pinned 内存，延迟高于普通可分页内存；而 GPU 访问 Pinned 内存时，还要额外处理缓存一致性开销，访问延迟同样高于 GPU 私有全局内存。即便如此，Pinned 方案依旧免去显式内存拷贝，整体效率优于拷贝式数据交互方案。

原版 HNSW 面向高端 CPU 与独立 GPU 设计，无法适配 Jetson 这类边缘设备的硬件特征，尤其是 CPU 与 GPU 共享物理内存的统一存储架构。想要充分挖掘硬件性能，就必须针对性设计数据排布方案与内存访问策略，上述短板正是本文面向边缘设备重构、优化基于 GPU 的 HNSW（即 ON‑NSW）的设计动因。

三、设计

本节介绍面向边缘设备优化的 ON‑NSW 算法。首先概述整体架构，随后详述各项核心设计。

3.1 设计总览


ON‑NSW 是专为英伟达 Jetson 这类边缘设备打造的 GPU 优化型检索方案。核心思路是重构原生 HNSW 结构，最大化 GPU 并行算力、降低访存延迟。为此，ON‑NSW 舍弃多层架构，仅保留原始第 0 层全图，全部检索流程在 GPU 上执行（对应算法 2）。该架构可并行探索多条候选节点、批量同步计算候选与查询向量的距离。同时 ON‑NSW 依托 Jetson 特有的 CPU‑GPU 统一内存架构，精细化规划各类数据的内存存放位置。

ON‑NSW 的数据组成包含向量本体、访问标记数组、Top‑k 结果集、查询向量、候选数组五类。其中需要 CPU 与 GPU 双向读写的数据（向量数据、访问标记数组、Top‑k 结果）统一存放于页锁定（Pinned）内存，规避主机与设备间的数据拷贝开销；查询向量载入 GPU 全局内存，候选数组分配在片上共享内存，以此压低访问时延。算法采用线程束（Warp）粒度的并行邻域遍历策略，单次可并行计算多候选距离、批量拓展邻居节点，单轮跳转能够遍历更多候选点，加快检索收敛至近邻。

3.2 面向 GPU 并行的扁平化图结构设计

原生 HNSW 依靠多层拓扑提升检索效率：上层逐层跳转逼近查询区域，落地至第 0 层后开展精细近邻查找。若直接从第 0 层固定入口起步检索，需要遍历海量节点；借助高层导航可跳过大量无关数据，缩减 0 层检索开销。这套串行跳转逻辑在纯 CPU 环境收益显著。

但迁移至 GPU 平台时，分层串行检索会成为性能瓶颈：各层级搜索存在强先后依赖，难以拆分为并行任务。虽然可将上层检索交由 CPU、下层交由 GPU 的异构混合方案，但无法实质性削减 0 层的检索开销；加之 CPU、GPU 交互必须依赖 Pinned 内存，而该内存的 CPU 侧访问延迟高于普通可分页内存，进一步拖累整体效率。

为解决上述问题，ON‑NSW 采用单层扁平化图，整张索引仅保留原 0 层、承载全部向量数据，充分释放 GPU 并行能力。在该架构下，GPU 依托海量 CUDA 核心同步完成多节点距离运算。不同于 CPU 依靠分支预测与缓存优化串行遍历，GPU 在全平铺图上能最大化并行利用率，更快收敛至目标近邻；同时扁平化设计简化内存管理，便于按硬件特性分区存放数据（3.3 节展开）。

尽管扁平化结构只能从固定入口启动检索、单轮需要更多跳转步数，但 GPU 的并行计算能力可在单次迭代中批量拓展候选、同步算距，最终整体检索耗时更短。从理论复杂度来看，单层全图最坏为线性 O (N)，高于分层 HNSW 的对数复杂度，但依托 GPU 并行执行，在高维向量数据集上实测性能更优。

3.3 统一内存架构下的数据分区优化

Jetson 等边缘设备采用片上 SoC 架构，CPU 与 GPU 共享同一块物理 DRAM，经由片上互联总线与内存控制器完成寻址，这套特殊的存储层级提升了内存调度难度。ON‑NSW 结合硬件特性，对五类数据做精细化分区部署，如图 2 所示。

  1. 向量数据：存入 Pinned 内存。GPU 可通过片上总线以 DMA 零拷贝方式直接访问，省去传统 PCIe 的数据搬运；但 Pinned 内存页面被操作系统锁定、无法换页，CPU 无法启用常规缓存策略，因此 CPU 读取该区域数据延迟高于普通分页内存。
  2. 访问标记数组：检索时用来记录节点是否已完成距离计算，避免重复运算。数组容量和数据集总量一致，尺寸过大无法放入 GPU 片上共享内存，因此同样部署在 Pinned 内存。
  3. Top‑k 结果数组：检索结果需要 CPU、GPU 双向读取，存放至 Pinned 内存，实现结果共享、无需额外数据复制。
  4. 查询向量：仅在 GPU 内核运算阶段使用，预先拷贝至 GPU 全局内存；算法采用批量查询策略，减少频繁调用 cudaMemcpy 带来的传输开销。
  5. 候选数组：保存候选节点编号与对应距离，每轮迭代后按距离排序，保留前 ef 个候选用于下一轮邻居拓展；若排序后最优候选集合不再变动，则判定检索收敛、终止搜索。候选数组是管控邻域拓展、判定迭代终止的核心结构。

3.4 线程束粒度的并行邻域遍历

为充分挖掘 GPU 并行能力，ON‑NSW 在线程束（Warp）粒度上实现检索流程并行化，图 3 展示了其检索执行流程。

在 ON‑NSW 中，每个 GPU 线程块（Block）负责处理一条查询向量。单个线程块内部，每个 Warp 包含 32 个线程，这些线程均分一条候选向量的各个维度，协同完成距离计算；例如面对 1024 维向量时，单线程负责 32 个维度的运算。同一个线程块内的多个 Warp 可并行处理不同候选向量，最大化 GPU 硬件利用率。向量数据存放于统一内存的页锁定区域。Warp 是线程块内最小执行单元，单个 Warp 并行完成单个节点的距离运算，由此算法能够一次性批量算出候选数组中多个节点与查询的相似度。在新增候选节点阶段，ON‑NSW 并行拓展当前最优节点的全部邻居，实现检索空间的同步扩张。原生 HNSW 依靠 CPU 单线程串行处理单条查询、逐个访问节点；与之不同，ON‑NSW 在每一轮迭代中并行拓展 ef 个最优节点的所有邻居，单步跳转即可遍历更多节点，更快收敛到查询近邻，提升检索效率。

如图 3 所示，ON‑NSW 先通过cudaMemcpy()批量将查询数据拷贝至 GPU 全局内存，一个线程块对应处理一条查询。为实现各线程块之间的执行同步，算法采用基于共享标记变量的轻量化同步方案，保障所有线程块全部完成 Top‑k 结果求解。

GPU 首先初始化候选数组，该数组分为两部分：存储最优节点的 ef 列表、存放待遍历节点的候选列表。初始化时所有节点 ID 赋值为无符号整型最大值 UINT32_MAX，距离字段初始化为浮点无穷值 FLT_MAX；随后将检索入口点及其邻居的编号写入候选列表，对应距离保持无穷（尚未开展距离计算）。

每个由 32 线程组成的 Warp 负责单个候选节点的距离求解，多个 Warp 并发处理大量节点以提升效率。各线程首先依托访问标记数组判断节点是否已遍历，若已访问则跳过本次计算；未访问节点借助atomicCAS原子指令修改访问标记。原子操作修改成功代表当前线程首次处理该节点，保留数据；修改失败则说明节点已被其他线程遍历，该条目 ID 与距离重置为 UINT32_MAX、FLT_MAX，以此避免候选数组出现重复节点。

完成距离计算与访问状态更新后，ON‑NSW 按照距离升序对候选数组排序，并核查 ef 列表内容。若 ef 列表节点发生更新，则并行取出列表中全部节点、拓展其邻居并入候选列表；该拓展步骤同样基于 Warp 并行：单个 Warp 绑定一个源节点，内部多线程协同把源节点的邻居写入候选数组。

若排序后 ef 列表内容不再变化，检索终止，从 ef 列表提取 Top‑k 节点编号与距离存入结果数组。结果落地后通过atomicAdd原子指令修改全局同步标记；最终校验标记数值是否等于「线程块总数 × Top‑k 长度」，条件达成后将内部节点 ID 映射为用户自定义外部标签，返回最终检索结果。

3.5 优化策略

ON-NSW 采用了多项优化技术以提升检索效率。
基于标记的同步机制。如前文所述，ON-NSW 使用同步标记来协调 GPU 线程块的执行完成状态。启动 CUDA 内核通常需要 CPU 调用 cudaDeviceSynchronize() 等待执行结束，但我们发现这种同步方式会引入显著的开销。为降低该代价，ON-NSW 采用基于标记的轻量级同步机制。每个 GPU 线程块在完成检索任务后，通过 atomicAdd 原子操作更新标记变量。随后 CPU 通过自旋等待循环检查标记值，直到所有线程块执行完毕。该方法降低了同步开销，提升了整体性能。
向量化优化。在检索过程中，最耗时的操作是距离计算，尤其对于高维向量。为优化该过程，ON-NSW 将浮点数值向量化为 float4 类型，支持 128 位内存加载。该方法保证了合并内存访问，并提升了 GPU 内存带宽利用率。
数据预取。为掩盖页锁定内存的高访问延迟，ON-NSW 在基于线程束的距离计算期间一次性加载多个节点数据。这种预取操作使得内存访问延迟能够与计算过程重叠，从而提升检索效率。
多线程映射。检索完成后，ON-NSW 在 CPU 上使用多线程并行地将内部节点 ID 映射为外部标签，从而减少执行时间。

四、评估

本节介绍实验配置，并在边缘设备上从检索质量、吞吐量、能源效率和 GPU 利用率四个维度，将 ON-NSW 与原始 HNSW 算法进行全面对比评估。

4.1 实验环境

系统配置
我们在NVIDIA Jetson Orin Nano平台上开展实验。该设备搭载两组 Arm Cortex-A78AE v8.2 64 位处理器插槽，每组插槽包含 3 个 CPU 核心。设备总内存为 8GB，片上系统（SoC）集成了基于安培（Ampere）架构的 GPU。该 GPU 包含 8 个流式多处理器（SM）和 1024 个 CUDA 核心，最高时钟频率为 625MHz。每个线程束（warp）由 32 个线程构成，单个 GPU 线程块最多支持 1024 个线程，并配备 48KB 共享内存。此外，GPU 全局内存与系统 DRAM 共享物理空间。

数据集
实验使用 4 个真实数据集：3 个高维向量嵌入数据集和 1 个低维向量嵌入数据集。高维数据集包括：1536 维和 3072 维的 OpenAI 文本嵌入向量，以及 960 维的 GIST 数据集。低维数据集使用 128 维的 SIFT 数据集。所有数据集均采用单精度浮点数（4 字节）存储。我们从每个原始数据集中随机抽取50,000 条向量用于构建图索引结构。

随机采样的数据集可能与原始数据集分布存在差异。因此，为验证采样数据集是否保留了与原始数据集相似的分布，我们采用 弗雷歇起始距离（FID） 进行评估。FID 是一种常用于评估生成对抗网络（GAN）等图像生成模型性能的指标，用于衡量生成图像与原始图像特征分布之间的距离。由于图像数据本身具有高维特性，该指标同样适用于评估高维嵌入数据的分布相似性。FID 值越接近 0，表示两个数据集的分布越相似。我们计算了所有采样数据集的 FID 值，结果均极小（低于 0.01），证明采样数据集有效保留了原始数据的分布特征。

对于每条查询，我们通过计算其与所有采样向量的距离并选取前 1000 个数据 ID 构建真实近邻（ground truth），并按与查询向量的距离升序排列。

基线算法。
对比实验采用hnswlib提供的原始 HNSW 实现，并在 NVIDIA Jetson 设备上运行。
原始 HNSW 与 ON-NSW 使用完全相同的索引构建参数：M=16，efConstruction=100

遵循 HNSW 原始设计，第 0 层维护的邻居数量为上层的两倍，因此两种实现中第 0 层的 M 值均设置为32。由于第 0 层包含全部向量数据，保留更多连接有助于维持邻近节点间的局部连通性，提升检索效率。

实验使用L2（欧氏）距离作为相似度度量标准。

此外，我们修改了 HNSW 原始的距离计算函数：将原本针对 x86_64 SIMD 指令优化的实现，替换为适配 ARM 架构的NEON 指令集优化。

4.2 图索引检索效果

本节在固定 Recall@10 的前提下，对比两种图索引的检索吞吐量。原始 HNSW 采用 6 个 CPU 线程执行检索；ON-NSW 使用 32 个 GPU 线程块开展检索，每个线程块包含 1024 个线程，检索结束后额外启用 6 个 CPU 线程完成内部节点 ID 向外部标签的映射工作。

如图 4a~4c 所示，在高维向量数据集上，同等召回指标下 ON-NSW 性能优于原版 HNSW：在两款 OpenAI 数据集与 GIST 数据集上吞吐量分别提升 1.44 倍、1.41 倍、1.42 倍。性能优势来源于 ON-NSW 依托 GPU 并行能力批量处理查询、并行完成高维向量距离运算；单线程块内部借助线程束并行批量计算候选节点距离。原版 HNSW 依靠 CPU 串行算距，检索跳转步数多；而 ON-NSW 单次跳转并行遍历大量邻居节点，整体跳转次数更少。以 1536 维 OpenAI 数据集为例：传统 HNSW 在下沉至第 0 层前，高层平均需要 79 次距离运算，进入 0 层后还要经过 33 轮跳转，每轮平均 16 次距离计算。ON-NSW 直接从 0 层固定入口启动检索，平均仅需 7 轮跳转，单轮要执行约 193 次距离运算。尽管单轮算量更大，但依托 GPU 线程束并行架构，整体检索效率大幅领先原始 HNSW。

ON-NSW 为释放 GPU 并行潜力舍弃多层导航结构，检索结果无法和分层 HNSW 完全一致，但可通过调整召回参数控制结果精度，实验统一固定召回指标，保证吞吐量与硬件开销对比的公平性。

由图 4d 可见，低维数据集场景中，相同召回条件下 ON-NSW 吞吐量最高比原生 HNSW 低 2.8 倍。原因是低维向量距离计算耗时很短，GPU 并行收益无法体现，CPU 即可快速完成距离运算；同时 ON-NSW 每轮计算后必须额外执行排序操作，该开销在低维场景被进一步放大。以 SIFT 数据集为例：距离计算仅占总耗时 14%，排序开销占比超 60%；而原生 HNSW 无集中排序步骤，CPU 运算效率更高、吞吐量领先。

整体耗时对比：全 Recall@10 区间内，高维数据集上 ON-NSW 总耗时均优于 HNSW。OpenAI-1536D（Recall@10=0.974）、OpenAI-3072D（Recall@10=0.971）场景下，ON-NSW 耗时 0.305 秒，HNSW 为 0.417 秒，前者缩短约 27%；GIST 数据集 Recall@10=0.9 时，ON-NSW 耗时 0.210s、HNSW 耗时 0.280s。低维 SIFT 数据集 Recall@10=0.981 时，HNSW 耗时 0.054s，优于 ON-NSW 的 0.103s。

距离运算量对比：图 5a~5d 表明，同等召回下 ON-NSW 的总距离计算次数最高可达 HNSW 的 3.5 倍。该现象源于 ON-NSW 依靠 GPU 并行在单轮跳转中批量核算海量向量距离；虽然总运算量上升，但 GPU 的并行特性使其能更快搜到查询近邻。

4.3 不同 GPU 线程块数量下的吞吐量测试

本节通过更改 GPU 线程块数量开展吞吐量对照实验，ON‑NSW 中单个线程块固定配置 1024 个线程（单块支持的线程上限），评价指标统一采用 Recall@10。

如图 6a~6d 所示，全部数据集下，吞吐量随线程块数量增多整体上升。但线程块从 1 增加至 8 时的性能涨幅，明显高于 8 提升至 32 的阶段。究其原因：寄存器、共享内存等硬件资源限制了可同时运行的线程块上限，后续实验统一选用 32 块作为最优配置。

在本文内核配置下，单个流式多处理器（SM）最多同时运行 2 个线程块，整卡至多 16 个线程块并行执行。当启动块数超过 16 时，多余线程块进入调度队列阻塞等待资源释放。并且 16 个活跃块对应的线程束数目已足以掩盖向量加载时 DRAM 的访存延迟；继续扩充块数只会加剧线程束阻塞占比，无法带来性能提升。

4.4 各阶段耗时拆解分析

本节在全部数据集上拆解原生 HNSW 与 ON‑NSW 各检索环节的耗时占比。原生 HNSW 执行流程包含高层遍历、第 0 层近邻检索、内部 ID 与外部标签映射三步；ON‑NSW 省去高层遍历，直接将查询向量拷贝至 GPU 内核执行检索，其余步骤保持一致。

原生 HNSW 采用 6 线程完成检索，ON‑NSW 启动由 32 个线程块（每块 1024 线程）构成的 GPU 内核，检索完毕后同样启用 6 个 CPU 线程做标签映射。ON‑NSW 的检索参数 ef 固定设为最大值 30，原生 HNSW 则微调参数以对齐 Recall@10 指标。

由图 7a~7d 可见：无论哪种算法，约 95% 的总耗时都消耗在第 0 层检索环节。向量间距离需要反复计算，成为耗时主体；同时邻居节点的访问以随机访存居多，频繁造成缓存缺失，进一步拉长耗时。

对 ON‑NSW 而言，查询向量向 GPU 拷贝的耗时仅占总时长 3.5%，得益于批量查询机制大幅减少cudaMemcpy()调用次数。原生 HNSW 的高层遍历开销占总耗时 5.5%，ON‑NSW 因取消分层结构无此项开销。

标签映射环节中，原生 HNSW 开销仅 0.16%，ON‑NSW 约占 1%。原因是 ON‑NSW 将 Top‑k 编号与距离存放在页锁定内存，该内存访问延迟高于普通 DRAM，带来少量额外开销，但该部分对整体耗时影响微乎其微。

4.5 能耗效率

本节对比 ON‑NSW 与原生 HNSW 执行近似近邻检索过程中的功耗。功耗采集采用英伟达自带的tegrastats工具（集成于 2024 年 2 月发布的 JetPack5.1.4、L4T R35.5.0 系统固件），采样间隔为 100 毫秒。针对各数据集，统一为两种算法设定相同的 Recall@10 指标，指标取值与 4.2 节总耗时实验保持一致，以此测定对应功耗。实验对每组数据集连续运行 10 轮、每轮执行 1000 次查询，取全部运行结果的平均功耗。和前文实验配置一致：HNSW 启用 6 条 CPU 线程，ON‑NSW 配置 32 个 GPU 线程块，每个线程块内含 1024 个线程。

如图 8 所示，在高维数据集上，ON‑NSW 整体耗电量相比原始 HNSW 最多可降低 13%。GPU 拥有数千个 CUDA 核心，瞬时峰值功耗高于 CPU，但 ON‑NSW 检索耗时大幅缩短，最终总能耗更低。反观低维 SIFT 数据集场景，ON‑NSW 的总耗电量约为 HNSW 的两倍。

4.6 边缘设备上的 GPU 资源利用率

本文采用屋顶线（Roofline）模型分析 ON‑NSW 执行近似近邻检索时的 GPU 硬件利用率。本项实验选用 1536 维 OpenAI 数据集，采用 32 个 GPU 线程块（每块 1024 线程），总共运行 1000 次查询。结合 1024 个 CUDA 核心、625MHz 的 GPU 主频、单周期 2 次 FP32 浮点运算吞吐，可算出该 GPU 理论峰值算力为 1.28 TFLOP/s；硬件理论内存带宽为 68GB/s。

如图 9 所示，受访存延迟制约，ON‑NSW 检索阶段 GPU 算力无法打满：向量数据存放于页锁定内存，该存储区域本身访问延迟偏高。表征流式多处理器平均繁忙程度的 SM Busy 指标仅为 22%，意味着约 78% 的流式多处理器因等待内存数据陷入阻塞。除此之外，近似近邻检索以随机访存为主，造成缓存命中率偏低，GPU 一级缓存命中率 41%、二级缓存命中率仅 1.8%，绝大多数数据访问需要直接从主存读取。

五、相关工作

5.1 GPU 加速的近似近邻检索

近些年已有大量研究围绕 GPU 加速近似近邻检索展开。SONG 算法通过精简 GPU 内存分配逻辑、降低显存占用提升检索性能；CAGRA 依托 GPU 并行计算同时加速图索引构建与在线检索；ParaGraph 采用 CPU-GPU 协同运算架构，高效完成图构建与跨模态近似近邻查找；Tagore 提出一套 CPU-GPU - 磁盘异步协同框架，缓解多部件调度带来的性能瓶颈；FAISS 库同样实现了包含 HNSW 在内的多种检索算法的 GPU 版本。

上述研究虽验证了 GPU 加速的有效性，但大多面向搭载独立显卡的高端服务器开发。而英伟达 Jetson 这类边缘硬件具备独特架构，最典型特征就是 CPU 与 GPU 统一内存体系，现有文献很少针对该特性做定制优化。已有一篇近期文献虽在边缘设备上实测了近似近邻算法，但仅对现有算法做简单移植，没有结合硬件架构做针对性改良。ON-NSW 面向边缘端 GPU 重构 HNSW，在最大化并行效率的同时充分适配嵌入式硬件的独有特性。

5.2 HNSW 衍生改进算法

依托高召回率与高效检索性能，基于图结构的近似近邻算法广受研究，HNSW 更是衍生出诸多改进方案：Dynamic HNSW 通过动态调优索引参数优化运行效率；HNSW++ 引入双分支结构加速检索；d-HNSW 面向内存解耦系统，采用适配 RDMA 的布局与缓存策略改造 HNSW；DIGRA 将多叉树与 NSW 图融合，实现带范围筛选条件的检索。

以上各类改进方案均以高端服务器 CPU 为运行载体，与之不同，ON-NSW 面向边缘设备 GPU 重构 HNSW，聚焦并行优化与内存层级调度优化。

六、总结

本文提出了以 GPU 为核心的 NSW 改进算法 ON-NSW，针对边缘设备重构 HNSW。算法扁平化 HNSW 多层拓扑以充分释放 GPU 并行能力，同时结合 Jetson 的 CPU-GPU 统一内存架构优化各类数据的存储分区。实验结果表明，在保证召回指标基本持平的前提下，针对高维数据集，ON-NSW 相较原生 HNSW 能够大幅提升检索吞吐量。
