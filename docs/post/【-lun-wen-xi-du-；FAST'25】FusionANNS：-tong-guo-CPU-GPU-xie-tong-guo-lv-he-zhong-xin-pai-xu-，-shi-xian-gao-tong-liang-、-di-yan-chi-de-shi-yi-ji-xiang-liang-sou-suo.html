【论文细读；FAST'25】FusionANNS：通过CPU-GPU协同过滤和重新排序，实现高通量、低延迟的十亿级向量搜索

FlashANNS: GPU-Driven Asynchronous I/O Pipelining for Eliminating Storage-Compute Bottlenecks in Billion-Scale Similarity Search

摘要

近似最近邻搜索（ANNS）已成为数据库与人工智能基础设施的关键组成部分。向量数据集规模持续增长，给近似最近邻搜索服务在性能、成本与精度方面带来了严峻挑战。当前主流的近似最近邻搜索系统均无法同时兼顾这些问题。

本文提出FusionANNS系统：面向十亿级向量数据集，仅借助固态硬盘（SSD）与单块入门级图形处理器（GPU），即可实现高吞吐、低延迟、低成本、高精度的近似最近邻搜索。

FusionANNS 的核心思想是CPU/GPU 协同过滤与重排序机制，该机制大幅减少 CPU、GPU 与 SSD 之间的 I/O 操作，进而突破 I/O 性能瓶颈。具体来说，本文提出三项创新设计：

  1. 多层索引，避免 CPU 与 GPU 之间的数据交换；
  2. 启发式重排序，在保证高精度的同时消除不必要的 I/O 与计算开销；
  3. 冗余感知 I/O 去重，进一步提升 I/O 效率。

我们实现了 FusionANNS 系统，并将其与当前最先进的基于 SSD 的 ANNS 系统 SPANN、GPU 加速的内存式 ANNS 系统 RUMMY进行对比。实验结果表明：

  1. 相较于 SPANN，FusionANNS 的每秒查询数（QPS）提升9.4–13.1 倍，成本效率提升5.7–8.8 倍；
  2. 相较于 RUMMY，FusionANNS 的 QPS 提升2–4.9 倍，成本效率提升2.3–6.8 倍，同时保持低延迟与高精度。

一、引言

高维空间中的近似最近邻搜索（ANNS），是指为给定查询向量找出最相似的 Top-K 向量。它在数据挖掘 [1]、搜索引擎 [2] 以及人工智能驱动的推荐系统 [3,4] 等诸多领域有着广泛应用。特别是在大语言模型（LLM）蓬勃发展的推动下 [5-8]，ANNS 系统已成为现代人工智能基础设施的核心组件。图 1 展示了检索增强生成（RAG）的典型框架：领域专用知识首先被编码为高维向量并存储在向量数据库中；当聊天机器人接收到查询时，会通过 ANNS 引擎从向量数据库中检索最相关的知识，让大语言模型将这些知识作为额外上下文，从而实现更精准的推理。

ANNS 是典型的强依赖内存、高计算密集型应用。主流 ANNS 系统 [9-13] 大多采用倒排文件（IVF）[14-16] 或基于图 [12,13,17] 的索引来加速检索。对于十亿级数据集，这类索引通常需要巨大的内存资源。例如，当前最先进的基于 IVF 的 RUMMY [9] 和基于图的 Bang [10] 均需要TB 级内存才能容纳十亿级向量及其索引。巨大的内存需求显著抬高了总拥有成本（TCO），阻碍了 ANNS 向超大规模数据集（如千亿级向量）扩展。除内存需求极高外，ANNS 还存在计算密集的问题，需要在向量间执行海量距离计算，在高维大规模数据集场景下尤为突出。随着向量数据库规模快速增长，ANNS 已成为 RAG 场景下新的性能瓶颈[9]，在大语言模型查询中可能占据总延迟的 50% 左右[18]。

为降低 ANNS 的内存成本，目前主要有两类技术路线：分层索引（HI）[15,19] 与乘积量化（PQ）[20,21]。

第一，分层索引通过将索引存放在 SSD 上降低内存占用 [15,22]。典型代表是微软的商用 ANNS 系统 SPANN [14,15]，它将所有基于 IVF 的索引（即倒排列表）存储在 SSD 上，仅在内存中用导航图维护这些列表的聚类中心。尽管 SPANN 延迟较低，但我们发现其并发查询吞吐量严重受限，即使在高端 SSD 上也仅能在 4 个 CPU 线程达到峰值（见 2.1 节）。这种有限的扩展性使其难以满足高吞吐需求的 AI 应用。

第二，乘积量化是另一种节省内存的有效方案。这种向量压缩技术可将高维向量的内存占用最高降低 95%，并能将 ANNS 速度提升数倍 [20]。但由于 PQ 是有损压缩，压缩率越高，查询精度通常越低，在召回率≥90% 等高精度要求场景下往往无法接受 [23]。

为解决计算瓶颈，GPU 被越来越多地用于加速 ANNS 中的海量距离计算。近期基于 GPU 的 ANNS 方案 [9,10,20,24-26] 在可放入 GPU 高带宽内存（HBM）的小规模数据集上表现高效。但在十亿级数据集下，基于 GPU 的方案会出现显著性能下降。我们的实验表明，直接在 SPANN 中使用 GPU 做距离计算，性能反而会下降约 10%（见 2.3 节）。根本原因是GPU 显存容量有限，导致 GPU 显存、主机内存与 SSD 之间产生大量数据搬运。

尽管上述方法能在一定程度上解决性能、成本或精度问题，但没有一种方案能在十亿级 ANNS 服务中同时实现高吞吐、低延迟、低成本与高精度。直观上，将分层索引、乘积量化与 GPU 加速结合似乎能得到最优 ANNS 方案。然而我们发现，这种组合的性能反而不如仅使用分层索引的 SPANN（见 2.3 节）。总体而言，在 GPU 加速的 ANNS 系统中协同融合分层索引与乘积量化，仍面临诸多关键挑战。

挑战 1：为提升查询精度与效率，多数 ANNS 系统 [14,15] 采用向量复制策略构建高质量倒排文件（IVF）索引，即将边界向量复制到相邻的倒排列表中。这会使索引体积显著膨胀，达到原始向量大小的8 倍[14,15]。即便使用乘积量化（PQ）压缩索引，GPU 的高带宽内存（HBM）仍无法容纳全部压缩索引，进而导致GPU 与 CPU 之间频繁的数据交换。

挑战 2：由于乘积量化会造成不可忽视的精度损失，通常需要搭配向量重排序流程来提升查询精度。但不同压缩向量的精度损失差异极大，因此在给定精度约束下，为每个查询确定需要重排序的最小向量数量极具挑战性。

挑战 3：原始向量大小仅为128～384 字节，远小于现代 NVMe SSD 的最小读取粒度（4KB），每次读取原始向量都会引发严重的读放大，导致重排序阶段的 I/O 效率极低。

系统设计与方案
本文提出FusionANNS：一种面向十亿级向量检索的CPU+GPU 协同处理架构。该系统仅使用一块入门级 GPU，即可同时实现高吞吐、低延迟、低成本、高精度。FusionANNS 的核心思路是通过CPU/GPU 协同过滤与重排序，最大限度减少 GPU、CPU 与 SSD 之间的数据搬运。针对上述三大挑战，本文提出三项创新设计：

第一，创新多层索引结构，支撑 CPU/GPU 协同过滤
FusionANNS 采用三级存储：

  1. SSD 存储原始向量；
  2. GPU 高带宽内存存储PQ 压缩向量；
  3. 主机内存仅保留各倒排列表的向量 ID与导航图。

由于 GPU 显存仅存储高度压缩的 PQ 向量，而非压缩后的倒排列表，因此可容纳十亿级数据集的全部压缩向量。查询时，主机 CPU 先遍历内存中的导航图，找到 top-m 个最近倒排列表，随后仅传输向量 ID（不传输向量内容） 到 GPU 进行距离计算，大幅降低 CPU 与 GPU 间的数据传输量。

第二，启发式重排序，在保证精度的同时避免无效 I/O 与计算
将重排序流程拆分为多个小批量并顺序执行。每完成一个小批量，就通过轻量反馈控制模型判断后续批次是否对提升精度有效；若连续多批无明显增益，立即终止重排序。

第三，冗余感知 I/O 去重，进一步提升重排序阶段的 I/O 效率
将高相似度向量紧凑存储，提升 SSD 上的空间局部性。基于优化后的存储布局，实现两种 I/O 去重机制：

  1. 同一小批量内，合并映射到 SSD 同一页的多个 I/O 请求，缓解读放大；
  2. 充分利用 DRAM 缓存，消除后续小批量中的重复 I/O。

本文贡献
本文的主要贡献如下：

  1. 设计 FusionANNS 系统，它是首个面向十亿级数据集、同时实现高吞吐、低延迟、低成本与高精度的 GPU 加速 + SSD 架构 ANNS 系统。
  2. 针对挑战 1，提出多层索引，实现 CPU/GPU 协同过滤，显著降低 CPU 与 GPU 间的数据传输。
  3. 针对挑战 2，提出启发式重排序，消除重排序阶段的无效 I/O 与计算。
  4. 针对挑战 3，基于优化存储布局提出冗余感知 I/O 去重，进一步提升 I/O 效率。

二、背景和动机

本节首先介绍两种ANNS索引技术和向量的乘积量化（PQ）。接着，我们介绍主要思想并分析其主要挑战，这些挑战推动了FusionANNS的设计。

2.1 ANNS的索引技术

多数近似最近邻搜索（ANNS）算法会采用欧氏距离等度量方式，为给定查询向量找出Top-K 最近邻向量。在高维大规模数据集上，这类计算会因维度灾难而代价极高 [27]。为解决这一问题，主流 ANNS 算法 [16,17,28–32] 均采用索引技术，剪枝掉不可能包含最近邻的数据区域。这类索引能通过缩小搜索空间显著提升查询性能，但会大幅增加内存占用，在大数据集场景下尤为明显。在各类索引技术中，倒排文件索引（IVF）[15,16,31,32] 与图索引[17,33] 因效率突出而被广泛使用。
图索引
图索引通常将向量组织为近邻图结构：顶点代表向量，边代表顶点间的距离。查询时，ANNS 引擎从指定顶点出发遍历图，以找到 Top-K 最近邻。DiskANN[22] 是典型的基于图索引的 ANNS 方案：它用 SSD 存储十亿级数据集的图索引，同时将高频访问的顶点保留在主存中。尽管 DiskANN 是一种内存高效的 ANNS 方案，但在大规模数据集下，因其迭代路径过长，会导致查询延迟偏高。

倒排文件索引（IVF）
IVF 索引是面向 SSD 存储的大规模数据集的主流索引方案。构建 IVF 索引时，通常通过聚类算法 [34] 将数据集划分为多个倒排列表，每个列表用其聚类中心表示。近期研究 [9,35] 表明，在十亿级数据集上，IVF 索引 [15] 比当前最优的图索引 [22] 效率更高。

SPANN[15] 是基于 IVF 索引的顶尖十亿级 ANNS 系统。与传统 IVF 不同，SPANN 通过将聚类边界向量复制到相邻倒排列表，构建了增强型 IVF 索引。这种复制机制使索引体积膨胀至原始向量的 8 倍，但显著提升了查询精度与效率。如图 2 所示，SPANN 将所有倒排列表存于 SSD，仅在内存中用图索引维护这些列表的聚类中心。查询时，SPANN 先遍历内存中的图，找到 Top-M 个最近倒排列表并加载到主机内存，再在这 M 个列表中通过距离计算找出最终的 Top-K（K<M）最近邻。

SPANN 的瓶颈

尽管 SPANN 能达到与内存式 ANNS 相当的低延迟，但我们发现其并发查询吞吐量严重受限。如图 3a 所示，SPANN 仅在 4 个 CPU 线程时达到 QPS 峰值，增加线程无法继续提升吞吐量。我们将查询延迟分为两个阶段：(1) 内存中图遍历；(2) 从 SSD 读取并处理倒排列表。图 3b 显示，查询延迟随线程数近乎线性上升，其中图遍历延迟基本不变，而SSD 倒排列表处理延迟随线程数急剧增加。原因是：多查询并发读取大量、大尺寸的倒排列表，导致严重的 I/O 竞争与高延迟。

2.2 乘积量化PQ

为缩减大规模数据集的向量索引体积并降低计算开销，乘积量化（PQ）[32] 近年来被广泛用于高维向量压缩。

假设一个包含 N 个向量的数据集使用 PQ 进行压缩：首先将所有向量均匀划分为 M 个子空间，每个子空间包含 N 个子向量；随后对这些子向量做聚类，生成码本（codebook），码本中存放所有簇的聚类中心。借助码本，每个子向量都可以用距离最近的聚类中心来近似表示。每个子空间的聚类数量通常设为 256，使得每个簇 ID 仅需1 字节即可表示。所有码本生成完毕后，每个向量便可被压缩为长度为 M 字节的 PQ 编码。

查询时，系统会先生成一张距离查询表，表中存放查询子向量与各子空间聚类中心之间的所有距离。随后，查询向量 q 与压缩向量 v 之间的近似距离可按如下公式计算：

  • M 为子空间总数
  • qi 为查询向量的第 i 个子向量
  • ci(vi) 为压缩向量在第 i 个子空间对应的聚类中心

因此，qi 与 ci (vi) 之间的距离可直接以 PQ 编码为地址，通过查表快速获取。最终，q 与 v 之间的距离即为所有子空间距离 dist(qi, ci(vi)) 之和。

从本质上讲，PQ 将向量间的距离计算转化为多次内存访问操作。由于传统 CPU 架构的 DRAM 带宽较低、并行度有限，这一特性使其难以高效运行 PQ。因此，PQ 通常由 GPU 加速[21,32]，以充分利用 GPU 大量计算核心与高带宽内存，从而提升查询性能。

2.3 主要思想和挑战

为了规避十亿级数据集带来的巨大计算与内存资源挑战，我们的目标是设计一套依托固态硬盘（SSD）与单块入门级 GPU、同时具备高吞吐、低延迟、低成本、高精度的近似最近邻搜索（ANNS）系统。然而，设计 GPU 加速的 ANNS 系统面临一个关键难题：GPU 高带宽内存（HBM）容量有限，会导致 GPU 与 CPU 之间产生大量数据交换，显著降低大规模数据集下的 ANNS 性能。

基于乘积量化（PQ）与分层索引（HI）的朴素方案
幸运的是，乘积量化（PQ）技术可大幅降低向量的内存占用，从而缓解 CPU 与 GPU 之间数据传输带来的性能瓶颈。因此，PQ 有潜力充分发挥 GPU 的算力，加速 ANNS 中的距离计算 [20]。本文首先讨论一种直接组合 PQ 与分层索引（HI）的朴素 GPU 加速 ANNS 方案。

该方案除了用 PQ 压缩所有向量外，其余分层索引结构与 SPANN 完全一致。查询流程如下：

  1. ANNS 引擎先遍历导航图，找到 top-m 个最近的倒排列表；
  2. 将这些压缩后的倒排列表加载到 GPU 的 HBM 中进行距离计算；
  3. GPU 在 top-m 个列表中计算查询向量与各压缩向量的距离，得到 top-n 个候选向量；
  4. 由于 PQ 会损失精度，需对 GPU 输出的中间结果进行重排序以提升精度；
  5. 重排序阶段读取 top-n 候选向量的原始数据，与查询向量比对，得到最终 top-k 近邻。

实验观察
令人遗憾的是，上述朴素方案并未达到预期的高性能。为探究根本原因，我们设计了四组实验，评估HI、PQ、GPU 加速三种技术的三种组合方式，所有实验均保证相同的查询精度。

如图 4a 所示，与 SPANN 的纯 HI 方案相比，单纯加入 PQ 或 GPU 都无法降低端到端查询延迟：

  • HI+GPU：虽大幅降低距离计算延迟，但 CPU 与 GPU 间传输倒排列表的开销（即 CudaMemcpy）抵消了 GPU 加速收益；
  • HI+PQ：仍用 CPU 处理 PQ 压缩的倒排列表。虽然从 SSD 加载到内存的 I/O 延迟降低，但 CPU 在计算查询向量与压缩向量距离时面临密集内存访问，导致端到端延迟显著上升；
  • HI+PQ+GPU：距离计算延迟极低，但数据拷贝开销与额外重排序流程带来巨大成本，完全抵消 GPU 收益。

此外，如图 4b 所示，所有组合方案的吞吐量均低于仅使用 HI 的原始 SPANN。直接将 PQ 加入 SPANN，甚至使 QPS 大幅下降65%。

为揭示性能下降的根源，我们统计了单次 ANNS 查询平均产生的I/O 次数，以及在 SSD、主存、GPU HBM 之间传输的数据量。
如图 4c 所示：

  • 尽管 PQ 将倒排列表的 I/O 大小从 12~48KB 降至 SSD 页粒度（4KB），但重排序流程使 I/O 次数增加了 70%；
  • 最终，I/O 性能瓶颈从SSD 带宽瓶颈转为IOPS（每秒 I/O 操作数）瓶颈。

同时，如图 4d 所示：大量倒排列表在 CPU 与 GPU 之间反复传输，进一步抵消了 GPU 加速的优势。

挑战 1：总体而言，分层索引（HI）、乘积量化（PQ）与 GPU 加速三者结合后，延迟反而高于仅使用 HI 的 SPANN，吞吐量也更低。
根本原因在于：GPU 的高带宽内存（HBM）仍然无法容纳所有经 PQ 压缩的倒排列表，导致 GPU 与 CPU 之间的大量数据传输成为新的性能瓶颈。如果不对跨设备的数据布局进行精细设计，不实现三种技术的深度协同，就不可能充分发挥 GPU 对 ANNS 的加速潜力。

挑战 2：为达到同等查询精度，通常需要通过重排序对 GPU 输出的中间结果进行精修。需要重排序的 top-n 向量数量（即重排序数）通常是最终 top-k 近邻数的数倍。
为评估重排序数量对精度的影响，我们线性增加重排序数并执行 10,000 次查询。如图 5a 所示，当重排序数设为 40 时，约 42% 的查询可在Recall@10=1.0下得到精确的 top-10 近邻；所有查询的平均精度可达到 Recall@10=0.9。在这种情况下，仅对少数滞后查询增加重排序数才有意义。
此外，如图 5b 所示，不同 ANNS 查询所需的最小重排序数差异极大。如果对所有查询使用固定重排序数，会产生大量不必要的 I/O 操作与距离计算。但在给定精度约束下，为每个查询动态确定最小重排序数极具挑战性。

挑战 3：重排序流程会向 SSD 上的原始向量发起大量 I/O 请求。原始向量大小通常为128~384 字节，而现代 NVMe SSD 的最小操作单元为一页（4KB）。这种粒度不匹配会导致严重的读放大，使重排序阶段的 I/O 效率极低。
幸运的是，我们发现需要重排序的向量通常彼此高度相似。这一特性为通过重构 SSD 上的数据布局缓解读放大提供了可行思路。

三、FusionANNS设计

本节将介绍FusionANNS的核心设计，即多层索引、CPU/GPU 协同过滤、启发式重排序以及冗余感知 I/O 去重。

3.1 多层索引


图 6 展示了多层索引在主机内存、GPU 高带宽内存（HBM）和固态硬盘（SSD）中的存储结构。我们采用离线方式构建多层索引。
首先，使用分层平衡聚类算法[34] 将数据集迭代划分为若干倒排列表。每个倒排列表包含多个向量 ID（vector-ID）及其对应的向量内容。
我们沿用 SPANN 的配置方式设置倒排列表数量（约为数据集总向量数的 10%），并采用相同的向量复制机制解决边界向量问题 [15]。具体而言，当一个向量位于多个聚类的边界时，我们按照公式（2）将该边界向量分配至对应聚类。

式中，v 表示待分配的向量，Ci 表示第 i 个簇。特别地，C1 表示距离向量 v 最近的簇。参数 ε 决定了一个向量可被同时分配到多个簇的最大距离阈值。为平衡查询精度与效率，每个向量最多被分配到 8 个簇[15]。

内存索引
数据集完成聚类后，我们基于所有倒排列表的聚类中心，采用 SPTAG [36] 构建图索引并存放于主存。借助这张导航图，FusionANNS 可高效为查询向量找到 top-m 个最近倒排列表。该图的构建方式为持续向空图中添加新向量：当一个向量作为新顶点加入时，会为其创建边，连接到 top-k（通常为 64）个最近邻顶点；随后其邻接顶点会更新自身邻居列表，以限制最大边数。

与 SPANN 将全部倒排列表存于 SSD 不同，我们仅提取每个倒排列表的向量 ID 作为元数据（不含向量内容）存入内存（如图 6 所示）。图索引与元数据生成后，中间倒排列表即可丢弃。由于图与元数据的内存占用很小，FusionANNS 可在通用服务器上以低成本内存方式支持十亿级 ANNS。

GPU 显存中的 PQ 压缩向量
乘积量化（PQ）通过有损压缩大幅降低高维向量的内存占用，即使是 NVIDIA V100（32GB HBM）这类入门级 GPU，其显存也可容纳十亿级数据集的全部压缩向量。在 FusionANNS 中，我们将所有压缩向量常驻锁定在 GPU 显存，避免了以往 GPU 加速 ANNS 系统中常见的CPU–GPU 频繁数据交换[9,20,24]。

由于内存索引保留了边界向量复制机制的优势，FusionANNS 可高效获取所有候选向量的 ID，仅传输向量 ID（不传输向量内容） 到 GPU 进行距离计算。这一设计彻底消除了因 CPU–GPU 间 PCIe 带宽有限带来的性能瓶颈。

SSD 上的原始向量
与基于 IVF 的 SPANN 将全部倒排列表存于 SSD 不同，FusionANNS 仅需在 SSD 上存放原始向量用于重排序。原始向量的体积仅约为倒排列表的 1/8，因此可显著降低存储开销。对每次查询而言，只有重排序阶段会产生少量 I/O 请求，从而缓解了并发查询下 SSD 的 I/O 瓶颈。

小结
以往基于 SSD 的 ANNS 系统（如 SPANN [15]、DiskANN [22]）无法有效利用 GPU 加速，因为 GPU 不仅不能减少单次查询的 I/O，反而会引入大量数据搬运开销。与之相对，本文提出的多层索引架构可显著降低 GPU 显存、主存与 SSD 的存储占用，并大幅削减 SSD–CPU–GPU 之间的数据传输量。

比较SPANN与FusionANNS的索引存储方式：
一、SPANN
1.内存（DRAM）：聚类中心+聚类导航图
2.SSD：完整倒排列表（聚类中心向量+属于这个聚类的原始向量（包含8倍的复制数据）+向量ID）
3.数据流向：查询 → 内存图遍历 → 从 SSD 加载整个倒排列表 → CPU 计算距离 → 返回结果

二、FusionANNS
1.内存（DRAM）：仅从完整倒排列表中提取出聚类中心--向量ID的映射关系+聚类导航图
2.显存：PQ压缩的全部向量
3.SSD：仅完整向量（ID-向量数据，无重复向量）
4.数据流向：查询 → CPU 遍历内存图 → 发送向量 ID 到 GPU → GPU 算距离 → 返回候选 ID → 仅重排序时从 SSD 读少量原始向量

3.2 CPU/GPU协同过滤


多层索引为 ANNS 查询提供了CPU/GPU 协同过滤能力。图 7 展示了 FusionANNS 的系统架构以及 ANNS 查询的完整流程。

当收到向量查询时，FusionANNS 首先利用 GPU 生成查询向量的距离表，用于后续的乘积量化（PQ）距离计算（步骤①）。与此同时，CPU 遍历内存中的导航图，找到与查询向量最近的 top-m 个倒排列表（步骤②）。随后，CPU 查询元数据，收集这些候选倒排列表内的所有向量 ID（步骤③）。之后，CPU 将这些向量 ID 发送给 GPU，并调用 GPU 核函数进行后续处理（步骤④）。

当 GPU 接收到向量 ID 后，首先通过并行哈希模块对 ID 进行去重（步骤⑤）。对每个向量 ID，GPU 从高带宽内存（HBM）中读取对应的 PQ 压缩向量，并计算它与查询向量之间的 PQ 距离。在这一步中，GPU 为 PQ 向量的每个维度分配一个线程，用于访问距离表中预先计算好的对应值；再由一个协调线程将这些值累加，得到每个候选向量的 PQ 距离（步骤⑥）。随后，GPU 将所有距离按升序排序，并将 top-n 个向量的 ID 返回给 CPU（步骤⑦）。

由于使用 PQ 向量得到的中间结果精度有限，CPU 根据这些向量 ID从 SSD 中读取原始向量，进行进一步的重排序（步骤⑧）。最终，CPU 返回最终的 top-k 近邻结果。

3.3 启发式重排序

由于乘积量化（PQ）在距离计算中会造成精度损失，通常需要执行重排序流程来精修 GPU 输出的中间结果。如 2.3 节所述，为达到相同的查询精度，不同 ANNS 查询所需的最小重排序数量通常差异极大。因此，固定设置重排序数量可能会引发不必要的 I/O 操作与距离计算，或是导致精度下降。

早期的内存式 ANNS 系统（如 LanceDB [37]）为简便起见，会对所有查询重排序相同数量的候选向量，因为高速内存可以容忍部分无效的数据访问。然而，这种简单策略在基于 SSD 的 ANNS 系统中效率很低：在 SSD 的 IOPS（每秒 I/O 操作数）有限的情况下，并发查询产生的大量小 I/O 请求会导致性能显著下降。

为解决这一问题，我们提出启发式重排序机制，以最小化 I/O 操作与距离计算。其核心思想是：保守地设置一个较大的重排序数量以保证高精度，一旦后续搜索对提升查询精度不再有益，就立即终止重排序流程。

为实现这一目标，我们将重排序过程划分为多个小批量（mini-batch） 并顺序执行，每个小批量包含相同数量的候选向量。由于所有候选向量已按距离升序排列，先执行的小批量更有可能找到属于最终 top-k 的近邻。每完成一个小批量，我们就通过一个轻量反馈控制模型判断后续小批量是否对提升精度有帮助。

为简化问题，我们使用优先队列 Q（大顶堆 max-heap） 维护当前的 top-k 近邻。初始时，大顶队列为空。对每个小批量，我们计算查询向量与该批量内向量的距离，并将距离小于当前堆内最大距离的向量插入堆中。

如果连续 β 次检测到相邻小批量计算后大顶堆的变化率都小于设定阈值 ε，我们就立即终止重排序过程。

当一个小批量处理完成后，我们从 Q 中获取更新后的 top-k 向量 ID，并计算相邻两个小批量之间大顶堆的变化率 ∆（第 9–10 行）。

  • 如果变化率低于设定阈值 ε，稳定计数器加 1；
  • 一旦稳定计数器超过阈值 β，立即终止重排序；
  • 如果变化率超过阈值 ε，则重置稳定计数器并继续处理后续小批量。
    最终，返回 Q 中最终的 top-k 向量。
    为达到最优性能，我们根据实验将 ε、β、BatchSize 分别设为 0.1、1、k。

小结
本文提出的启发式重排序算法，能够在保证高精度的前提下，最小化对 SSD 的 I/O 请求与距离计算的 CPU 资源消耗，最终显著降低重排序延迟。

总结：启发式重排序的作用就是为了减少每次查询的重排序阶段需要从ssd中随机读的全精度向量个数

3.4 冗余感知的I/O去重

FusionANNS 依靠 SSD 上的原始向量，对 GPU 返回的中间结果进行重排序。一种朴素方式是将所有原始向量按顺序存放在 SSD 页中。但原始向量大小仅 128～384 字节，远小于 SSD 页粒度（4KB），单独读取单个原始向量往往会造成严重的读放大。此外，重排序会产生大量随机小 I/O，使得 I/O 延迟成为影响端到端查询延迟的关键因素。

与大多数基于 SSD 的 ANNS 系统 [15] 一致，本文采用直接 I/O（Direct I/O），以充分发挥现代 NVMe SSD 的低延迟特性。为进一步提升 I/O 效率，首先优化数据布局以增强 SSD 上的空间局部性；再通过冗余感知 I/O 去重缓解读放大问题。

优化存储布局
尽管需要重排序的向量是依据 PQ 距离筛选得到的，但这类向量与查询向量相似度很高，彼此之间也具备强相似性，天然在特征空间上聚集。利用这一相似性，可通过合理组织 SSD 数据布局缓解读放大。

具体做法：离线构建内存索引时，对导航图中的每个聚类中心，建立一个桶（bucket），存放距离该中心最近的一批原始向量，且桶与桶之间无重复向量。

若单个桶的大小无法与 SSD 页对齐，则采用最大最小算法[40] 对未对齐部分进行桶合并，最大限度减少 SSD 页内的空闲空间。最后将所有桶合并为一个整体文件存入 SSD，并在内存维护一张向量 —SSD 页映射表。

小批量内与小批量间 I/O 去重
依托优化后的存储布局，本文设计两类 I/O 去重机制：

  1. 同小批量内：合并映射到同一 SSD 页的多个 I/O 请求；
  2. 跨小批量间：利用 DRAM 缓存，消除后续小批量中的重复 I/O。
    结合图 8 举例说明：
    假设重排序分为两个小批量：
  • 小批量 0 需要重排向量：V2、V4、V6
  • 小批量 1 需要重排向量：V5、V8、V9

执行小批量 0 时，先查询映射表，得到各向量对应的 SSD 页编号。由于 V2、V6 存放在同一 SSD 页 P0，可合并两次 I/O 请求，只需读取一次 P0 就能同时获取 V2、V6。P0、P2 不在 DRAM 缓存中，因此直接发起两次 I/O 读入缓存。

执行小批量 1 时，V5、V8、V9 虽分属不同 SSD 页，但包含 V5 的 P2 已在 DRAM 缓存中，无需重复读取；只需额外发起两次 I/O 读取 P1、P3 即可。

小结
通过优化 SSD 存储布局，系统得以实现小批量内与小批量间双层 I/O 去重，最终有效缓解读放大、提升整体 I/O 效率。

四、实施方案

我们使用 C++ 和 CUDA 编写了 22000 行代码，实现了 FusionANNS 的系统原型。FusionANNS 可广泛部署在配备入门级 GPU的通用服务器上。与大多数 ANNS 系统一致，我们采用每个 CPU 线程处理一个独立查询的模式。

无竞争 GPU 内存管理
FusionANNS 专为并发 ANNS 查询实现了一套 GPU 内存管理器。系统初始化阶段，我们先将压缩向量加载到 GPU 的高带宽内存（HBM）中，并将剩余空间作为内存池。随后，将内存池划分为若干独立内存块，每个块分配给单个查询作为工作内存使用。查询完成后，其占用的内存块可重新分配给其他等待执行的查询。

该方案可避免频繁的内存分配与查询间的锁竞争，从而提升系统性能。

高效 GPU 核函数
对每个向量，我们根据其维度分配多个 GPU 线程，实现距离的并行计算。此外，我们设计了专用核函数，基于哈希算法实现向量 ID 的并行去重。

对于候选向量 ID 列表，我们为每个向量 ID 分配一个 GPU 线程，并使用自旋锁确保同一时刻只有一个线程能够访问并更新哈希表项。

该方案可充分利用 GPU 的高并行性加速去重过程。

与 GPU 加速的内存型方案 RUMMY 相比，在不同数据集下 FusionANNS 的 QPS 提升 2–4.9 倍，且始终保持低延迟。

值得注意的是，RUMMY 在 DEEP1B 数据集上性能显著下滑：数据集越大，从主机内存向 GPU 显存传输数据占用的 PCIe 带宽越多，CPU–GPU 间的带宽瓶颈被进一步放大。

而 FusionANNS 在 CPU 与 GPU 之间仅传输轻量的向量 ID，不传输向量原始数据，因此相对 RUMMY 取得显著性能优势。

不同精度等级下的性能表现
为评估 FusionANNS 在不同精度下的表现，将 Recall@10 从 90% 提升至 98%，基于 SIFT1B 数据集测得不同精度下的 QPS 与延迟如图 10a、图 10b 所示，所有结果均以 SPANN 为基准做归一化处理。

FusionANNS 相对 SPANN、DiskANN 的 QPS 分别提升约 9.4–11.7 倍、3.2 倍。

随着查询精度要求提高，FusionANNS 相对 SPANN 的 QPS 增益进一步扩大；且在高精度约束下，其延迟甚至远低于内存型方案 RUMMY。

根本原因在于：CPU/GPU 协同过滤机制有效消除了主存与 GPU 显存之间的数据交换，因此可以扩大搜索空间以满足更高精度，同时仍保持高性能。

反观其他 ANNS 方案，当搜索空间增大时，会产生更多 I/O 操作与距离计算，性能明显劣化。

5.2 扩展性

ANNS 系统的吞吐量与 CPU 线程数高度相关。为评估各 ANNS 系统的性能可扩展性，我们以2 的指数倍逐步增加线程数量。

图 11 展示了不同线程数下各 ANNS 系统的 QPS 与延迟表现。当线程数从 1 增加到 64 时，FusionANNS 的 QPS 呈现显著增长。

所有 ANNS 系统在单线程场景下的 QPS 几乎持平；但当线程数提升至 64 时，FusionANNS 在保持低延迟的同时，相对 SPANN、DiskANN、RUMMY 的 QPS 最高分别提升 13.2 倍、3.8 倍、5.1 倍。

SPANN 仅用 4 个线程 就达到 QPS 峰值，继续增加线程会导致延迟大幅上升。

值得注意的是，在 SIFT1B 和 SPACEV1B 数据集上，RUMMY 的 QPS 在 16 线程 达到峰值后明显下降；同时线程数超过 16 后，RUMMY 的延迟也显著增大。

可扩展性受限的原因是：并发查询增多会引发 CPU 与 GPU 之间海量数据传输，造成严重的 PCIe 带宽争抢。

由于 DEEP1B 数据集的向量维度更大，RUMMY 面临的带宽竞争更为激烈，如图 11f 所示，其 QPS 甚至低于 DiskANN。

即便在内存资源受限的条件下，FusionANNS 仍具备远优于其他方案的可扩展性。其根本原因是：消除了 CPU 与 GPU 之间的数据交换开销，同时大幅提升了 SSD 的 I/O 效率。

5.3 单个技术的有效性

本小节基于 SIFT1B 数据集，对 FusionANNS 中各项核心技术的有效性进行消融实验评估。

首先基于本文多层索引实现一个仅 CPU 版本变体 MI(CPU)：将原本由 GPU 处理压缩向量的逻辑替换为主机 CPU 执行。随后在多层索引基础上，逐步叠加 FusionANNS 的其他关键技术，分别评估各项技术对系统性能与 I/O 访问量的影响。

如图 12a、12b 所示，仅采用 CPU 实现的多层索引版本 MI (CPU)，相较 SPANN 的 QPS 提升 1.5–4.2 倍，但查询延迟偏高。

而搭载 GPU 的多层索引版本 MI(GPU)，不仅相比 MI (CPU) 大幅降低延迟，相对 SPANN 的 QPS 也提升 5.9–6.8 倍。

原因在于：CPU 内存带宽有限，而 GPU 高带宽、高并行的 HBM 显存，能让 MI (GPU) 更高效地完成 PQ 距离计算；同时多层索引架构使 CPU 仅向 GPU 传输向量 ID，有效缓解了 PCIe 带宽瓶颈。

在多层索引基础上，叠加启发式重排序（HR）与冗余感知 I/O 去重后，可进一步分别将延迟最高降低 39%、QPS 最高提升 17%。

图 12c 展示了单次查询产生的平均 I/O 次数。

当要求更高查询精度时，SPANN 的 I/O 请求数会显著增加；而多层索引相较 SPANN 可将 I/O 次数降低 3.2–3.8 倍。

在此基础上，启发式重排序可再减少最高 30% 的 I/O，冗余感知 I/O 去重可再减少最高 23% 的 I/O。

此外，SPANN 的每次 I/O 往往需要读取多个 SSD 页，而本文及其他对比方案每次 I/O 仅访问单个 SSD 页。

FusionANNS 不仅大幅降低单次查询的 I/O 次数，还减小了单次 I/O 的数据粒度，从而实现显著的性能提升。

六、相关工作

内存型 ANNS 方案
近几十年来，近似最近邻检索（ANNS）已得到广泛研究，相关工作主要聚焦内存索引技术[16,17,31,33,36]。
层次化可导航小世界图（HNSW）[17] 在内存中构建多层可导航图结构，实现快速近似最近邻检索；
空间划分树与图（SPTAG）[36] 结合相对邻域图与空间划分树，选取若干检索种子节点，加速图遍历过程。
但现有内存型 ANNS 算法需要占用大量内存资源来存储原始向量与索引结构，高昂的内存开销大幅提升部署总成本，难以支撑 ANNS 向大规模数据集扩展。
FusionANNS 借助 SSD 承载大规模向量数据，通过多层索引与系统性优化，同时实现高性能与低成本优势。

基于 SSD 的 ANNS 方案
近年来多项工作针对十亿级数据集设计分层索引以降低内存占用，代表性工作包括 DiskANN [22]、SPANN [15]、Starling [41]、BBANN [45]、GRIP [46] 等。这类方案充分利用现代 SSD 硬件特性优化向量索引结构。
SmartANNS [19] 面向智能固态硬盘（SmartSSD）设计分层索引方案，利用多块智能 SSD 内部的 PCIe 带宽缓解 I/O 瓶颈，在十亿级 ANNS 场景下实现近似线性的可扩展性。
然而，并发查询之间严重的 I/O 竞争，导致传统基于 SSD 的 ANNS 系统吞吐量普遍受限，难以满足高吞吐业务需求。
FusionANNS 依托创新的多层索引结构与 CPU/GPU 协同检索，大幅降低单次查询的 I/O 次数，兼顾高吞吐量与低查询延迟。

基于硬件加速器的 ANNS 方案
近年部分研究 [24–26,47] 利用 GPU 加速图基 ANNS 中的图遍历流程；
同时还有大量工作结合 PQ 量化技术[32,48]、GPU [20,21]、FPGA [49–51] 加速倒排（IVF）类 ANNS 检索。
但这类方案大多受限于 GPU、FPGA 片上内存容量，仅能支撑小规模数据集 [24–26,47,49]，或无法保证检索精度 [49–51]。
当前最优工作 RUMMY [9] 虽通过主机内存扩容 GPU 显存，并设计重排序流水线技术支持十亿级数据集，但 CPU 与 GPU 之间海量的数据传输 仍严重制约其性能。
FusionANNS 融合分层索引、乘积量化（PQ）与 GPU 加速技术，从架构上彻底消除 CPU 与 GPU 之间的频繁数据搬移，在保证高检索精度的前提下，实现高吞吐、低延迟与低成本的综合优势。

七、结论

本文提出 FusionANNS，一种面向十亿级近似最近邻检索（ANNS）的CPU+GPU 协同处理架构。

FusionANNS 利用 CPU/GPU 协同过滤与重排序机制，在保证高检索精度的同时，显著提升 ANNS 的检索性能与成本效益。

相较于当前最优的基于 SSD 的 ANNS 方案 SPANN，FusionANNS 在保持低查询延迟的同时，大幅提升了并发查询的吞吐量。

此外，相较于 GPU 加速的内存型 ANNS 方案 RUMMY，FusionANNS 也实现了更高的吞吐量与更优的成本效率。
