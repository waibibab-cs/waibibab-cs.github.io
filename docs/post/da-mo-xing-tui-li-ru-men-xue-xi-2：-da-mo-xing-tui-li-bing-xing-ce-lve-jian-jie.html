大模型推理入门学习2：大模型推理并行策略简介

大模型推理的并行策略，核心在解决三个事情：1.模型太大，单卡放不下；2.单卡算的慢，吞吐不够；3.请求太多，延迟太高。因此不同的优化策略优化的目标也各不一样，常见的可以分为以下三类，（还有部分比较复杂的并行策略，本文不做介绍）
a.解决“模型放不下”的问题
并行策略：Tensor Parallel（TP）、Pipeline Parallel（PP）
b.解决“请求太多”的问题
并行策略：Data Parallel（DP）
c.解决上下文太长“的问题
并行策略：Sequence Parallel（SP）

也可以按照这样方式理解：假设Transformer的激活张量的形状为(B,S,H)，其中B=Batchsize表示有单个批次的请求数量；S=序列长度，表示单个请求有S个token；H=隐藏维度，表示每个token是H维度的向量。那么：
切batchsize--->DP
切序列长度--->SP
且隐藏层尺寸--->TP

一、数据并行（DP）

DP(Data Parallel)数据并行，是解决数据并发量大时使用的策略，DP方法在不同的GPU运行LLM模型的多个副本，并在每个模型副本上独立处理用户请求组。

假设激活形状为：(8,4096,8192)，如果有4张GPU，每个GPU上保存一份完整的模型副本，那么DP的切法可以是：

GPU0:[2,4096,8192]
GPU1:[2,4096,8192]
GPU2:[2,4096,8192]
GPU3:[2,4096,8192]

例如有四个用户请求：A（帮我总结文章）、B（帮我写邮件）、C（帮我解释代码）、D（帮我翻译文本），DP的做法就是：GPU0跑请求A、GPU1跑请求B、GPU2跑请求C、GPU3跑请求D

在推理时，GPU0不需要知道GPU1在干什么，多个GPU间无需通信，然而在训练时，每张GPU拿一批不同数据，算梯度：

GPU0:batch0->forward->backward->gradient0
GPU1:batch1->forward->backward->gradient1
GPU2:batch2->forward->backward->gradient2
GPU3:batch3->forward->backward->gradient3

最后需要同步梯度（All Reduce gradients）再更新模型参数，因此训练DP需要通信

总之，DP的优点是：1.简单 2.通信少 3.吞吐扩展好 4.故障隔离好
缺点是：1.显存浪费 2.单卡放不下大模型

在实际推理系统（如VLLM）里，DP通常叫replica、model replica、woker replica

二、张量并行（TP）

Tensor Parallelism (TP)张量并行，一句话理解即：把同一层里的大矩阵计算拆分到多张GPU上一起算

最常见的一种是列并行，假设Y=XW，其中

X:[B*S,H]
W:[H,4H]
Y:[B*S,4H]

然后把W按输出维度切成4份：W=[W0,W1,W2,W3]，每张GPU存一块：

GPU0:W0
GPU1:W1
GPU2:W2
GPU3:W3

每张GPU拿完整输入X分别计算：GPU0:Y0=XW0、GPU1:Y1=XW1......
然后合并：Y=[Y0,Y1,Y2,Y3]，示意图可以表示为：

                    输入激活（完整复制到所有GPU）
                            X [B*S, H]
          ┌──────────────────┼────────────────----------------──┐
          │                  │                  │               |
          ▼                  ▼                  ▼               ▼
      ┌────────┐        ┌────────┐        ┌────────┐        ┌────────┐
      │ GPU 0  │        │ GPU 1  │        │ GPU 2  │        │ GPU 3  │
      │        │        │        │        │        │        │        │
      │ W0     │        │ W1     │        │ W2     │        │ W3     │
      │[H, H]  │        │[H, H]  │        │[H, H]  │        │[H, H]  │
      └───┬────┘        └───┬────┘        └───┬────┘        └───┬────┘
          │                 │                 │                 │
     Y0=XW0            Y1=XW1           Y2=XW2            Y3=XW3
   [B*S,H]           [B*S,H]          [B*S,H]           [B*S,H]
          └───────────────┬───────────────┬───────────────┘
                          │
                    AllGather / Concat
                          │
                          ▼
                   Y = [Y0,Y1,Y2,Y3]
                       [B*S, 4H]

还有一种是行并行
对于Y=XW，假设：

X:[B*S,4H]
W:[4H,H]
Y:[B*S,H]

此时把W按输入维度（行）切分成4个[H,H]形状的权重分别放到各个GPU中，对应地输入也需要切分成四个[B*S,H]的形状分别送入4个GPU中计算，与列并行不同的是，在最后不是做一个简单的拼接，而是直接进行求和，示意图如下：

            输入激活 X [B*S, 4H]
        ┌────────┬────────┬────────┬────────┐
        │   X0   │   X1   │   X2   │   X3   │
        └────────┴────────┴────────┴────────┘
             │         │         │         │
             ▼         ▼         ▼         ▼
        ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐
        │ GPU 0  │ │ GPU 1  │ │ GPU 2  │ │ GPU 3  │
        │        │ │        │ │        │ │        │
        │ X0     │ │ X1     │ │ X2     │ │ X3     │
        │ W0     │ │ W1     │ │ W2     │ │ W3     │
        │[H,H]   │ │[H,H]   │ │[H,H]   │ │[H,H]   │
        └───┬────┘ └───┬────┘ └───┬────┘ └───┬────┘
            │          │          │          │
       P0=X0W0    P1=X1W1    P2=X2W2    P3=X3W3
         [B*S,H]   [B*S,H]   [B*S,H]   [B*S,H]
            └─────────┬─────────┬─────────┘
                      │
                 AllReduce (求和)
                      │
                      ▼
              Y = P0 + P1 + P2 + P3
                   [B*S, H]

综上，TP的优点是：1.缓解部分模型放不下的情况；2.单层计算更快；3.对但请求延迟更友好
缺点包含：1.通信频繁，每一层都要通信；2.强依赖高速互联（如NVLink、NVSwitch）

TP在prefill和decode中的差异：prefill阶段处理整个prompt，（B,S,H）中的S很大，此时TP很有用，但在Decode阶段一次生成一个token，S=1，通信开销相对更严重，因此decode阶段的TP瓶颈从纯计算转移为通信延迟、KV cache访问、带宽等

三、流水并行（PP）

PP(Pipeline Parallel)并行核心思想很简单：把一个大模型按“层”切成几段，每段放到不同 GPU 上，让请求像流水线一样依次经过这些 GPU。

PP的优点有：1.能放下更大的模型；2.通信比TP更少，TP每层都需要通信，而PP只需要在部分层之间通信；
缺点：1.最大的问题是pipeline bubble；2.负载不均衡，因此切层时尽量让各层计算量接近；3.单请求延迟较高

适用场景：1.模型单卡放不下；2.模型层数很多；3.多级多卡；4.追求吞吐而非极低单请求延迟
不适用场景：1.batch很小；2.decode为主；3.希望极低单请求延迟

四、序列并行（SP）

SP(Seqeunce Parallel)序列并行是将长序列切分成多个片段，分配到不同GPU设备上并行处理的模型并行策略。

主要方法是对于输入激活(B,S,H)在S维度上进行切分，例如：
原始激活：

[B, S, H] = [8, 8192, 12288]

SP 切分后：

GPU0: [8, 2048, 12288]
GPU1: [8, 2048, 12288]
GPU2: [8, 2048, 12288]
GPU3: [8, 2048, 12288]

示意图：

完整序列：t0 t1 t2 t3 t4 t5 t6 t7
切分后：
GPU0: t0 t1
GPU1: t2 t3
GPU2: t4 t5
GPU3: t6 t7

每张 GPU 只负责一部分 Token。
因此，激活显存从：

O(B × S × H)

降低为：
$$O\left(B \times \frac{S}{P} \times H\right)$$
其中P为并行度。

SP部分能工作的原因是Transformer 中有很多操作只依赖单个Token，例如LayerNorm、Dropout、Residual Add、Bias Add等，无需依赖其它token，但不是所有计算都能本地完成，例如attention的计算，因此需要一些通信机制：
1.All-Gather：将所有 GPU 上的序列片段拼接起来
通信前：

GPU0: seq0
GPU1: seq1
GPU2: seq2
GPU3: seq3

通信后：

GPU0: seq0 seq1 seq2 seq3
GPU1: seq0 seq1 seq2 seq3
GPU2: seq0 seq1 seq2 seq3
GPU3: seq0 seq1 seq2 seq3

每张卡重新获得完整序列。

2.reduce-scatter：计算结束后，将结果重新切回 Sequence 维度

因此SP的典型流程一般是：
Sequence Shard
↓
LayerNorm（本地计算）
↓
All-Gather
↓
Attention / MLP（TP计算）
↓
Reduce-Scatter
↓
Sequence Shard
这种：分片 → 聚合 → 计算 → 再分片的流程

综上，SP的优点包含：1.对于可序列并行的模块（layernorm等）显著降低输入激活的显存；2.与TP高度兼容
缺点：1.增加通信开销；2.并非所有算子适用，例如attention不适合，对于这种算子可能需要CP(Context Parallel)等其他策略

五、总结

对比项 DP 数据并行 TP 张量并行 PP 流水并行 SP 序列并行 CP 上下文并行
切分维度 Batch / 请求维度 Hidden / Tensor 维度 Layer / 模型层维度 Sequence 维度，但主要针对激活 Sequence / Context 维度，主要针对 Attention
切分粒度 每张 GPU 保存完整模型副本，处理不同 batch 或不同请求 同一层的大矩阵被切到多张 GPU 上，例如列并行、行并行 Transformer 层被分成多个 stage，不同 GPU 负责不同层段 将 [B,S,H] 沿 S 切成 [B,S/P,H] 将长上下文 token 切到不同 GPU，每张 GPU 负责一段 context 的 Q/K/V 或 KV cache
通信方法 训练：All-Reduce 梯度；推理：通常无通信，只是多副本服务 列并行常用 All-Gather；行并行常用 All-Reduce / Reduce-Scatter 相邻 pipeline stage 之间发送 activation / hidden states All-Gather 聚合序列片段；Reduce-Scatter 再切回序列分片 Ring Attention、All-Gather K/V、P2P 传递 K/V、All-to-All 等，取决于实现
通信开销 训练中每 step 同步梯度，开销中等；推理几乎无通信 高，每层都可能通信，强依赖 NVLink / NVSwitch 中等，只在 stage 边界通信，但有流水线调度成本 中高，需要在部分算子前后做 All-Gather / Reduce-Scatter 高，长上下文下通信量大，常常受带宽和延迟影响
优点 简单；吞吐扩展好；推理多副本天然适合高并发；故障隔离好 能让单卡放不下的大模型跑起来；单层计算并行；对单请求延迟较友好 能放下更深、更大的模型；相比 TP 通信频率低；适合超大模型 显著降低训练时激活显存；与 TP 兼容；适合大模型训练 能分摊长上下文 Attention 和 KV cache 压力；适合超长上下文训练/推理
缺点 每张 GPU 都要存完整模型，显存浪费；单卡放不下模型时无法使用 通信频繁；对高速互联要求高；decode 阶段通信占比容易变大 有 pipeline bubble；单请求延迟增加；stage 负载不均会拖慢整体 基本只对训练有价值；不能直接解决 Attention 的 O(S²) 问题；增加通信 实现复杂；通信重；需要专门 Attention 算法配合；对硬件互联要求高
适用场景 训练中扩大 batch；推理中多副本服务高并发请求 训练和推理都常用；模型参数单卡放不下；需要降低单层计算时间 训练和推理都可用；模型层数多、单卡放不下完整模型；追求吞吐 主要用于训练；TP 下激活显存成为瓶颈；长序列、大 hidden、大 batch 长上下文训练和推理；KV cache 或 Attention 显存/计算压力过大，例如 32K、128K、1M context
