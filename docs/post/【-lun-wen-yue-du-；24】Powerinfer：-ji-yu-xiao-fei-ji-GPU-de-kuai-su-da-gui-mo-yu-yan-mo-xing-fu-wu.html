【论文阅读；24】Powerinfer：基于消费级GPU的快速大规模语言模型服务

PowerInfer: Fast Large Language Model Serving with a Consumer-grade GPU

摘要

本文介绍了PowerInfer——一款可在搭载单块消费级显卡的个人电脑上实现大语言模型（LLM）高速推理的推理引擎。PowerInfer的核心设计思路，是利用大语言模型推理过程中天然存在的强局部性特征，该特征体现为神经元激活状态服从幂律分布：仅有一小部分神经元会在各类输入下持续被激活，这类神经元被称作热神经元；而绝大多数冷神经元的激活情况则随输入内容变化。 基于这一特性，PowerInfer打造了GPU-CPU混合推理架构：将高频激活的热神经元预加载至GPU中以实现快速调用，冷神经元则交由CPU运算，大幅降低GPU显存占用与CPU、GPU之间的数据传输开销。此外，该引擎还融合了自适应预测器与面向神经元的稀疏算子，进一步优化神经元激活效率与计算稀疏性。 实测结果表明，在单块英伟达RTX 4090消费级显卡上，针对多款大语言模型（包括OPT-175B）进行测试时，PowerInfer的推理速度较llama.cpp最高提升11.69倍，且全程保持模型原有精度。在运行OPT-30B模型时，单张RTX 4090显卡搭载PowerInfer的表现可媲美高端服务器级A100显卡，其令牌生成速率达到A100的82%。

一、引言

生成式大语言模型（LLM）凭借在各类复杂自然语言处理任务中展现出的优异能力，受到了广泛关注 [6,51,58]，在创意写作、高阶代码生成等领域表现尤为突出。这类模型目前大多部署在配备高端高价服务器级显卡的数据中心中，深刻改变着人们的日常工作与生活。与此同时，将大语言模型运行在门槛更低的本地平台也逐渐成为新趋势 [10,56]，搭载消费级显卡的个人电脑便是主流选择。推动这一转变的核心诉求包括强化数据隐私保护 [32]、实现模型定制化 [29] 以及降低推理成本 [51]。数据中心场景下的部署以高吞吐率为首要目标 [23,43,57]，而本地部署则更注重小批量任务处理的低延迟特性。

然而，大语言模型庞大的内存占用，使其在消费级显卡上落地面临巨大挑战。大语言模型基于自回归 Transformer 架构构建，逐词元顺序生成文本，每一轮推理都需要调用包含数千亿参数的完整模型，因此显存容量成为制约推理过程的核心瓶颈。本地部署场景通常仅处理单个请求 [7]，几乎无法开展并行计算，这一问题也随之愈发突出。

量化 [15,55]、剪枝 [30] 等模型压缩技术虽能缩减模型体积，但即便经过深度压缩，模型规模依旧超出消费级显卡的承载能力。例如，采用 4 位精度的 OPT-66B 模型，仅加载参数就需要约 40GB 内存空间 [27]，这一数值甚至超过了英伟达 RTX 4090 这类高端消费级显卡的显存上限。

模型分片卸载是另一种主流方案，该方案以 Transformer 层为粒度，将模型拆分后分别存放于显卡与内存中 [4,17,43]。llama.cpp 等当前主流系统便采用这一思路，把网络层分配至显卡与 CPU 内存协同完成推理，以此降低显卡资源消耗。但受限于 PCIe 总线的传输速率以及 CPU 本身有限的运算能力，该方案最终会导致推理延迟居高不下。

本文认为，大语言模型推理出现显存难题的根本原因，是硬件架构与模型推理特征之间存在局部性失配。现有硬件的存储层级体系，均围绕数据局部性设计：显卡拥有更高的显存带宽，但容量有限，理论上适合存放访问频繁、体量较小的常用数据；CPU 内存容量更大，但带宽偏低，更适合存储体量庞大、访问频次较低的数据。但大语言模型的每一轮推理都需要调用全部模型参数，整体参数规模远超单块显卡的承载能力，完全不具备数据局部性，硬件的局部性优化优势也就无从发挥。

已有相关研究证实，大语言模型推理过程存在激活稀疏性[28,34,61]：在每一轮推理中，仅有少量神经元被激活，而这些激活神经元对最终词元输出起到决定性作用。这类稀疏激活行为可在推理过程中被精准预测，因此只需计算被激活的神经元，就能实现推理加速。但不同输入对应的激活神经元组合各不相同，且只能在运行阶段确定，这就要求将完整模型全部载入显存，使得该技术难以应用在显存资源紧张的本地部署场景中。

本研究在大量推理实验中发现，大语言模型的神经元激活状态呈现出偏态幂律分布：一小部分神经元在各类输入场景下持续处于激活状态，承担了超过 80% 的激活运算，这类神经元被定义为热神经元；其余绝大多数神经元则为冷神经元，其激活与否由实时输入内容决定，仅参与剩余部分的运算。这一发现证明，高激活稀疏性的大语言模型本身具备天然的运算局部性，可用于解决前文所述的软硬件局部性失配问题。

基于这一局部性特征，本文提出了 PowerInfer—— 一款针对单张消费级显卡本地部署场景优化的高效大语言模型推理系统。该系统的核心思路是利用模型推理的局部性特征，将数量较少的热神经元部署在显卡上运算，占绝大多数的冷神经元则交由 CPU 处理。具体而言，PowerInfer 通过两大阶段发挥局部性优势：第一阶段为离线预处理，系统根据神经元的激活频率统计结果，区分热神经元与冷神经元，并分别预加载至显卡与 CPU；第二阶段为在线推理，系统借助预测器，针对每一份输入数据预判本轮可能被激活的热、冷神经元。依靠该机制，显卡与 CPU 可独立完成各自负责的激活神经元运算，大幅减少开销高昂的 PCIe 数据传输。

不过，PowerInfer 的设计落地也面临多项棘手难题。首先，用于识别网络层激活神经元的在线预测器通常部署在显卡中，会占用大量显存，挤占模型本身的运行空间。为此，PowerInfer 设计了一种自适应优化方法：针对激活稀疏度与分布偏态性更高的网络层，构建轻量化预测器。该迭代优化方案能够在保证预测精度的前提下缩减预测器体积，为模型推理释放显存资源。

其次，利用模型稀疏特性开展计算，必须依托稀疏算子。cuSPARSE 等传统通用稀疏函数库并非最优选择，这类库需要逐一记录非零元素，并完成稠密矩阵到稀疏格式的转换 [54,63]。对此，PowerInfer 专门设计了神经元感知型稀疏算子，可直接对单个神经元进行运算，跳过完整矩阵的处理流程。该算子不仅能在神经元粒度下高效实现矩阵向量乘法，还省去了繁琐的稀疏格式转换步骤。

最后，如何在显卡与 CPU 之间合理分配激活神经元，也是一项复杂的优化工作。分配策略需要综合考量各神经元的激活频率、层内数据交互开销以及显存大小等硬件资源条件。为解决该问题，PowerInfer 在离线阶段生成一套神经元部署策略：该策略以神经元对模型推理结果的影响程度作为评估指标，并将分配问题转化为整数线性规划问题求解，同时统筹神经元激活频率、CPU 与显卡的带宽层级等多重因素。

PowerInfer 的在线推理引擎基于 llama.cpp 扩展开发，新增代码共计 4200 行 C++ 与 CUDA 代码；由性能分析器和求解器组成的离线模块，依托 transformers 框架实现，代码量约 400 行 Python 代码。该系统兼容多款主流大语言模型系列，包括参数规模 70 亿至 1750 亿的 OPT 模型、70 亿至 700 亿参数的 LLaMA2 模型以及 400 亿参数的 Falcon 模型，同时支持英伟达 RTX 4090、RTX 2080Ti 等主流消费级显卡。

性能测试结果显示，在搭载单张英伟达 RTX 4090 显卡的个人电脑上运行时，PowerInfer 可在保证模型精度不变的前提下，使量化模型的平均生成速度达到 13.20 词元 / 秒，非量化模型达到 8.32 词元 / 秒。相较 llama.cpp，其量化模型推理速度最高提升 8 倍，非量化模型最高提升 11.69 倍。值得一提的是，售价约 2000 美元的 RTX 4090 显卡，借助 PowerInfer 实现的推理速度，仅比可完整加载模型、售价约 20000 美元的顶级服务器显卡 A100 低 18%。目前，PowerInfer 代码已全部开源。

二、背景与动机

2.1 LLM推理与架构

大语言模型属于自回归模型，其推理过程会依据已生成的词元逐一生成新内容。整个流程以提示词为起点，分为两个阶段：首先是提示词处理阶段，输出首个词元；随后进入生成阶段，持续逐词生成内容，直至达到最大生成长度或输出序列结束符（<EOS>）。每生成一个词元即为一轮推理迭代，该过程都需要完整运行大语言模型。

大语言模型由多层 Transformer 堆叠构成，每一层均包含自注意力模块与多层感知机（MLP）模块（见图 1 左侧）。自注意力模块通过捕捉输入词元之间的关联关系生成嵌入向量，该模块内的多个注意力头会分别提取不同维度的特征信息。各个注意力头的计算结果会被整合，随后输入至多层感知机模块。多层感知机模块借助全连接层与激活函数完成非线性变换，进一步优化输入序列的表征结果，其输出数据会送入下一层网络，或是作为整个大语言模型的最终输出。

图 1 右侧展示了多层感知机模块的两层全连接层 FC1 与 FC2，二者均通过矩阵乘法运算生成向量。输出向量中的每一个元素，都由输入向量与单个神经元（对应权重矩阵中的行或列）做点积运算得到。ReLU [2]、SiLU [40] 等激活函数相当于门控单元，会对向量中的数值进行筛选，决定保留或舍弃部分数值，进而影响 FC1 与 FC2 中的神经元激活状态。例如图中的 ReLU 函数会剔除所有负值，仅让 FC1 中输出为正值的神经元参与后续计算。本文中将这类对最终输出产生贡献的神经元定义为激活神经元。同理，上述运算结果也会决定 FC2 中哪些神经元被激活，并参与该层的输出计算。

激活稀疏性：大语言模型推理过程中存在明显的神经元激活稀疏现象，自注意力模块与多层感知机（MLP）模块中均能观测到该特征 [26,28,59]。在自注意力模块里，近半数注意力头的贡献度极低；而 MLP 模块的稀疏性主要由激活函数的特性导致。基于 ReLU 激活函数构建的模型，其 MLP 稀疏表现尤为突出，DejaVu 相关研究 [28] 指出，OPT-30B 模型在推理时约有 80% 的神经元处于未激活状态。该现象并非 ReLU 类模型独有，采用 SwiGLU 等其他激活函数的模型同样存在激活稀疏性。

例如表 1 数据显示，使用 SwiGLU 激活函数的 LLaMA2-13B 与 Yi-34B 模型，神经元稀疏度分别达到 43% 和 53%。上述结论与 CATS [24]、ReLU2 [61] 等以往研究的成果一致。

此外，在模型单次迭代运行过程中，我们可以提前预测后续数层的神经元激活情况。基于这一发现，DejaVu [28] 在推理阶段采用基于多层感知机的在线预测器，仅对已激活神经元开展计算，最终推理速度提升 6 倍以上，同时神经元激活状态的预测准确率仍能保持在 93% 及以上。但需要注意的是，每一轮推理对应的神经元激活稀疏性都与输入内容强相关，即特定神经元是否被激活直接取决于当前输入，无法在本轮推理开始之前预先判定。

以 GPU 为核心的卸载方案：该方案将超出显卡显存容量的部分模型参数存储至 CPU 内存。如图 2a 所示，每一轮推理迭代中，系统先运算显存内的参数，再根据需求从 CPU 内存调取其余参数。只要 CPU 内存与硬盘存储空间总量充足，该方案便可支持不同规模的大语言模型完成推理。FlexGen [43] 是这类方案的典型代表，它采用交错调度策略，优先保障吞吐率而非降低延迟，按层依次处理批量数据。但在对延迟敏感的场景中，该方案会造成单词元推理延迟居高不下（见图 3a），主要原因是显卡与 CPU 之间的数据传输过于频繁。如图 3b 所示，将模型权重从 CPU 传输至 GPU 的耗时占总处理时长的 99.5% 以上，这极大地拖累了整体推理延迟。

DejaVu [28] 虽利用神经元激活稀疏性加速大语言模型推理，但该方案原本面向数据中心场景设计，当应用于无法完整加载大语言模型的消费级显卡时，便会暴露出诸多问题。在这类硬件环境下，DejaVu 的核心瓶颈在于运行过程中需要频繁将激活神经元从 CPU 传输至 GPU。对于 OPT-30B 这类超出显卡显存上限的模型而言，即便 DejaVu 能够降低 GPU 的计算负载，其性能依旧受限于数据传输环节（见图 3a）。最终如图 3a 数据所示，DejaVu 的推理延迟问题十分突出，表现与 FlexGen 处于同一水平。

混合卸载方案：该方案以 Transformer 网络层为粒度，将模型参数拆分并分别存放于 GPU 与 CPU 中（见图 2b），llama.cpp [17] 便是此类方案的代表。运算时由 CPU 先完成对应网络层的计算，再将中间结果传输至 GPU，由 GPU 完成词元生成。这种卸载方式减少了数据传输量，缓解了 PCIe 总线带宽不足带来的问题，可将推理延迟降至 600 毫秒左右（见图 3a）。但对比 OPT-30B 模型在 A100 显卡上仅 45 毫秒的推理延迟，该速度依旧存在较大差距。

混合卸载方案同样存在软硬件局部性失配的问题，因此推理延迟难以进一步优化。每一轮推理迭代都需要调用完整模型参数，无法适配 GPU 与 CPU 构成的分层内存架构，数据局部性表现极差。GPU 虽具备强大的算力，却受限于显存容量。例如在 24GB 显存的英伟达 RTX 4090 显卡上运行 300 亿参数的模型时，仅有 37% 的模型能部署在 GPU 中，绝大部分计算任务都会转移至 CPU 端。CPU 内存容量更大，但算力薄弱，最终承担了整体 98% 的计算耗时（见图 3b）。

三、LLM推理的局部性洞察

3.1 洞察1：幂律激活机制

大语言模型推理过程具备显著的局部性特征，表现为固定的一组神经元会被频繁激活。尽管大语言模型的激活稀疏性与输入内容相关，但神经元的激活状态整体呈现出幂律分布。本文基于维基百科数据集 [13] 开展性能剖析，采集了约 100 万个词元对应的激活稀疏性统计数据。

图 4a 数据显示，在采用 ReLU 激活函数的 OPT-30B 模型、采用 ReGLU 激活函数的 LLaMA2-70B 模型以及采用 SwiGLU 激活函数的 LLaMA2-70B 模型的多层感知机模块中，分别有 26%、43% 和 69% 的神经元承载了总计 80% 的激活运算。这类被频繁激活的神经元，我们将其定义为热神经元。与之相对，剩余 74%、57%、31% 的神经元是否激活则由输入内容决定，这类神经元被划分为冷神经元。由于 ReLU 函数会带来更高的稀疏度，相较于使用其他激活函数的模型，基于 ReLU 构建的模型中该分布特征体现得更为明显。

此外，这种强局部性并非仅存在于单个多层感知机模块内，而是贯穿于整个模型。如图 4b 所示，放眼模型所有网络层，OPT-30B、采用 ReGLU 的 LLaMA2-70B、采用 SwiGLU 的 LLaMA2-70B 这三款模型中，分别仅有约 17%、26%、75% 的神经元完成了全部激活运算的 80%。图 5a 清晰表明，OPT-30B 模型的每一个 Transformer 层内，都只有小部分热神经元，其余绝大部分均为冷神经元。值得注意的是，OPT-30B 模型的前 24 层整体激活程度极低，仅有不到 1% 的神经元会被激活，因此这些网络层中的热神经元数量也极少。

有趣的是，同一批热神经元在各类下游任务中始终保持激活状态。为验证这一结论，我们选取多类数据集开展神经元激活特征分析，涵盖知识问答、真实性评估、逻辑推理等任务场景。图 5b 表明，在不同任务下，热神经元的激活状态始终稳定，神经元激活的幂律分布规律并未发生改变。分析数据显示，在不同任务领域中，激活频率排名前 20% 的神经元重合度超过 90%。这一高度稳定的特征证明，神经元激活所呈现的幂律分布是模型架构本身的固有属性，并非由特定数据集所导致。

3.2 洞察2：快速的CPU内计算


若待计算的激活神经元存放在 CPU 内存中，直接在 CPU 上完成运算，比将其传输至 GPU 再计算效率更高。本地部署场景下，激活神经元数量本就偏少，且普遍采用小批量处理模式，这一优势会更加突出。搭载向量扩展指令集的现代 CPU，能够高效完成这类小规模矩阵运算。

我们基于 OPT-30B 模型开展对比实验：分别将多层感知机模块中稀疏度为 10% 的神经元、自注意力模块中 60% 的神经元放在 GPU 运算，以及直接在 CPU 端运算，统计两种方式的耗时。图 6 结果表明，当批处理大小小于 32 时，即便使用英伟达 RTX 4090 显卡，将上述神经元的权重数据传输至 GPU 并完成计算的总耗时，也高于 CPU 借助 AVX2 向量扩展指令集直接运算的耗时。

四、Powerinfer综述


本文提出了PowerInfer，这是一套可部署在搭载单块消费级显卡的个人电脑上的低延迟大语言模型推理系统。PowerInfer 充分利用本文第 3 节提出的神经元高局部性相关结论，设计了面向神经元的模型卸载策略与配套推理引擎。该系统采用 GPU 与 CPU 联合存储模型权重的方式，可适配不同参数量规模的大语言模型。

这套卸载策略依托第一项核心结论，充分利用了大语言模型推理过程中神经元激活的幂律分布特性。具体而言，PowerInfer 将高频激活神经元的权重预加载至 GPU，低频激活神经元的权重则存放于 CPU 中。

为进一步降低推理延迟，推理引擎仅对在线预测器判定为激活状态的神经元执行计算，跳过绝大多数未激活神经元。由于已加载至 GPU 的热神经元承担了绝大部分激活运算，这种预加载方案能将主要推理任务分配至 GPU 执行。对于显存中未存放权重的冷神经元，PowerInfer 直接在 CPU 端完成计算，无需向 GPU 传输权重数据（对应第二项核心结论）。

PowerInfer 的效果与模型的激活稀疏性直接相关。采用 ReLU 系列激活函数的大语言模型，其前馈网络的神经元激活稀疏度可达 90% 以上，是适配该系统的最优选择。使用其他激活函数的大语言模型，稀疏度通常在 50% 左右，对应的加速效果会有所减弱，但系统仍能带来一定的性能提升。

值得关注的是，当前行业正逐步兴起两类技术趋势：在不损失模型性能的前提下提升大语言模型的激活稀疏性 [44,46]，以及直接基于 ReLU 系列激活函数训练大语言模型，英伟达的 Nemotron [1] 与 MiniTron [48] 便是典型案例。这一发展趋势也将进一步拓宽 PowerInfer 在各类大语言模型中的适用范围。

4.1 整体架构与工作流程

图 7 展示了 PowerInfer 的整体架构，该系统由离线模块与在线模块两部分构成。不同大语言模型的局部性特征存在差异，因此离线模块需要对模型的神经元激活稀疏性进行分析统计，完成热神经元与冷神经元的划分。在线阶段中，推理引擎将两类神经元分别加载至 GPU 与 CPU，在运行过程中以低延迟响应大语言模型推理请求。

模型分析器与策略求解器（离线模块）
依据第一项核心结论，借助通用数据集即可充分采集神经元的激活规律。因此本系统采用离线分析器，基于通用数据集（如 C4 数据集 [39]）构造推理请求，在模型推理过程中采集神经元激活数据。该模块首先对网络所有层的神经元激活状态进行监测（步骤①），随后由策略求解器对神经元完成冷热分类。

求解器的目标是将高频激活神经元分配至 GPU，其余神经元分配至 CPU。它结合神经元影响度指标与硬件参数实现负载均衡，并通过整数线性规划算法，最大化 GPU 端神经元的综合影响度（步骤②）。

面向神经元的大语言模型推理引擎（在线模块）
在处理用户请求前，在线引擎会按照离线求解器生成的分配方案，将两类神经元部署至对应的运算单元（步骤③）。系统运行时，引擎会分别创建 GPU 执行器与 CPU 执行器（CPU 端以线程形式运行），统筹调度 CPU 与 GPU 开展并行计算（步骤④）。

同时，推理引擎会预判神经元的激活状态，跳过所有未激活神经元。预加载在显存中的激活神经元直接由 GPU 运算；CPU 完成自身负责神经元的计算后，将结果传输至 GPU 进行汇总。CPU 与 GPU 两端均采用神经元感知型稀疏算子，运算过程直接作用于矩阵中对应单个神经元的行与列。

五、面向神经元的推理引擎

5.1 自适应稀疏预测器

PowerInfer 的在线推理引擎仅对预测为激活状态的神经元执行计算，以此降低运算负载。该思路同样被 DejaVu [28] 所采用，后者提出训练一组固定规模的多层感知机预测器。在每一个 Transformer 层中，DejaVu 会使用两个相互独立的预测器，分别预测自注意力模块与多层感知机模块内的神经元激活情况，最终仅针对预判激活的神经元开展推理计算。

但在资源受限的本地部署场景中，设计实用的预测器存在不小挑战，需要在预测精度与模型体积之间做好平衡。神经元激活预测会频繁调用预测器，因此这类组件必须存入显存以保证访问速度。然而大量固定规模的预测器会占用可观显存，挤压大语言模型参数的存储空间。举例而言，适配 OPT-175B 模型的预测器约需 27GB 显存，这一数值已经超出英伟达 RTX 4090 显卡的显存上限。反之，若单纯粗暴地缩减预测器体积，又会造成预测精度下降：当预测器体积从 480MB 缩减至 320MB 时，其预测准确率从 92% 跌至 84%，还会进一步拖累大语言模型本身的精度，例如在威诺格拉德推理任务 [41] 中，模型准确率从 72.77% 下降至 67.96%。

研究发现，预测器的体积主要受两大因素影响：大语言模型网络层的激活稀疏度，以及层内神经元激活分布的偏态程度。如图 9 所示，激活稀疏度越高的网络层，识别激活神经元的难度越低，对应的预测器模型可以设计得更小；而激活稀疏度较低的网络层，精准定位激活神经元的难度显著提升，就需要参数量更大的预测器。此外，若激活分布高度偏态（即激活行为高度集中在少数神经元上），即便是轻量化预测器也能实现较高的预测准确率。

针对上述特性，本文为各个 Transformer 层设计了非固定规模预测器，并配套迭代式训练方法。训练流程首先依据网络层的激活稀疏特征，设定预测器的基准模型规模（见图 9）；随后结合层内激活分布的偏态程度，迭代调整模型大小，确保预测精度不受影响。

多层感知机类预测器一般由输入层、单层隐藏层和输出层构成。由于输入层与输出层的维度由对应 Transformer 层的结构决定，因此模型调优主要针对隐藏层开展。

本次预测器训练过程中，采用模型在 WikiText-2 数据集上的困惑度作为预测精度的评判标准。训练初期，根据网络层的稀疏等级设定预测器维度，之后反复迭代调整隐藏层规模，直至搭载该预测器的整体模型困惑度与原始基准模型基本持平，二者误差控制在 0.1% 以内。依靠该方案，PowerInfer 成功将预测器的参数量控制在大语言模型总参数量的 6% 以内。

5.2 神经元管理

离线策略求解器确定神经元部署方案后，PowerInfer 的在线推理引擎会依照该方案，将模型参数分别加载至 CPU 内存与 GPU 显存。单个网络层可能包含多个权重矩阵，系统会根据神经元属于热神经元还是冷神经元，将其逐一分配至 GPU 或 CPU 进行存储。

要保证拆分后的神经元能够按正确顺序完成运算，从而输出精准结果，这一点至关重要。为此，PowerInfer 分别在 CPU 内存与 GPU 显存中各维护一张神经元索引表。两张表格记录了每个神经元在原始权重矩阵中的位置映射关系。在与输入张量执行矩阵乘法运算时，系统会依据索引表的映射规则，让各个神经元与对应的张量数值完成计算。这类索引表占用的额外内存空间极少，以参数量达 1750 亿的 OPT 模型为例，其整体存储需求约 350GB，而神经元索引表总共仅占用约 9MB 空间。

5.3 异构协同执行（GPU-CPU 混合执行）

PowerInfer 采用 GPU-CPU 混合执行架构，两类计算单元各自独立完成所分配激活神经元的运算任务。该架构能够在 GPU 与 CPU 之间合理分配计算负载，充分发挥二者的硬件优势，同时最大限度规避数据传输带来的性能损耗。由 GPU 负责运算预加载的热神经元，CPU 处理冷神经元，最终所有计算结果统一在 GPU 端完成合并。

推理开始前，PowerInfer 会构建计算有向无环图（DAG），图中每个节点对应大语言模型推理过程中的一个计算算子，并将整张图存放于 CPU 内存的全局任务队列中。队列内的每个算子都会标记其依赖的前置算子。推理运行阶段，操作系统创建的两类 POSIX 线程执行器分别管控 CPU 与 GPU 的计算任务：执行器从全局队列中取出算子，校验依赖关系后，将算子分发至对应的计算单元。

GPU 与 CPU 均使用面向神经元的专用算子。其中，GPU 执行器通过cudaLaunchKernel等应用程序接口调用 GPU 算子；CPU 执行器则调度空闲 CPU 核心开展运算，并在执行算子前确定并行计算所需的线程数量。针对算子间的依赖关系（尤其当前置算子在 GPU 执行、后继算子在 CPU 执行的场景），系统会设置同步屏障，确保 GPU 运算全部完成后，CPU 才会启动对应算子的计算。

当激活神经元分散在 GPU 与 CPU 两端时，设备间的同步机制同样不可或缺。任意一端完成神经元计算后，会等待另一端执行完毕，再统一合并运算结果。由于 GPU 侧神经元的激活频次更高，因此 PowerInfer 将结果合并操作统一交由 GPU 执行。

5.4 神经元感知算子

大语言模型存在激活稀疏性，矩阵乘法运算可跳过未激活神经元及其对应的权重，这就需要借助稀疏算子实现。但目前主流的稀疏矩阵乘法工具均无法很好适配该场景，其中既包括 SparTA [64]、FlashLLM [54] 等当下先进的稀疏感知编译器，也涵盖 cuSPARSE [37] 等函数库。这类工具要么仅支持稀疏内核的静态编译，要么需要在运行时将稀疏矩阵转换为稠密矩阵格式，会产生巨大性能开销，在本文所面对的动态稀疏场景下该问题尤为突出。此外，动态即时编译器 PIT [63] 虽能高效实现 GPU 上的通用稀疏矩阵乘法，却并不适用于 CPU 算力有限的 GPU-CPU 混合执行架构。

为解决上述问题，PowerInfer 设计了神经元感知算子。该算子可直接在 GPU 与 CPU 上对激活神经元及其权重开展运算，无需在运行时将稀疏数据转为稠密格式。区别于传统算子，神经元感知算子不以完整矩阵为运算单位，而是直接作用于矩阵内独立的行向量与列向量。其执行逻辑为：先判定神经元的激活状态，若预测该神经元处于激活状态，则结合参数矩阵中对应的行或列完成计算。

面向 GPU 的神经元感知算子
在 GPU 平台上，向量间运算的效率通常低于矩阵与向量的运算，但在小批量场景下，基于向量间运算实现的神经元感知算子反而具备优势。该算子会规避未激活神经元带来的无效计算与内存操作，同时省去开销高昂的矩阵格式转换步骤。不仅如此，所有线程块可并行完成神经元激活状态检测，一旦判定神经元激活，便立即执行对应向量运算。具体而言，PowerInfer 为 GPU 上不同线程块分配相互独立的计算任务，每个线程块负责处理一组专属神经元，线程块之间无需同步，也不会出现执行分支异常的问题。

面向 CPU 的神经元感知算子
CPU 的并行计算能力整体偏弱，而神经元感知算子恰好能充分发挥其优势。CPU 执行器将神经元感知算子调度至多个 CPU 核心运行，并把神经元划分为多个小批次，由各核心并行完成激活状态检测。每个核心仅处理自身批次内的激活神经元，同时依托 AVX2 等硬件向量扩展指令集，对向量间运算进行深度优化。

六、神经元部署策略

想要充分发挥 GPU 与 CPU 的算力优势，依托局部性特征设计部署策略对性能优化至关重要。倘若随机将神经元分配至 GPU 或 CPU，部分热神经元会被部署在 CPU 端，导致 GPU 强大的计算能力无法得到充分利用。此外，即便单纯将激活频率最高的神经元全部交由 GPU 处理，也可能造成 CPU 与 GPU 之间的数据传输量剧增，进而拉低整体运行效率。为解决该问题，PowerInfer 的离线模块借助求解器制定神经元部署策略，明确每一个神经元应当分配至 GPU 还是 CPU。

6.1 离线特征采集

在确定神经元的部署方案前，首先需要采集所有神经元的激活相关数据。根据前文得出的结论一：通用语料中筛选出的热神经元，在各类应用场景下同样会保持高频激活。基于该特性，本文采用离线分析器完成数据采集工作。该分析器加载大语言模型，处理由 C4 数据集 [39]、维基百科数据集 [13] 等多个通用数据集生成的推理请求。为精准统计激活数据，分析器在 Transformer 层的每一个模块后都嵌入监测内核，同时在 GPU 上搭建神经元信息表，用于记录每个神经元的激活次数。

6.2 神经元影响度指标

神经元影响度指标用于衡量单个神经元对大语言模型整体推理结果的贡献程度，该指标是 GPU 神经元分配工作的核心依据。在采集足量输入数据的前提下，统计得到的神经元激活频率能够真实反映其实际运行状态，本文便基于这一特性计算该指标。如公式 1 所示，单个神经元的影响度指标，由离线采集阶段统计得出的神经元激活频率定义。

6.3.3 整数线性规划优化

求解器采用整数线性规划（ILP），在满足公式 (3) 至不等式 (8) 全部约束条件的前提下，求解目标函数的最大值。由于整数线性规划问题本身属于 NP 完全问题，直接对千亿级参数的大语言模型求解该问题，会带来极大的计算压力。

为加快运算速度并得到近似最优解，本文的核心优化思路是将每层神经元划分为批次，以批次为单位统一分析布局方案。具体做法为：在同一层中，将64 个影响度相近的神经元合并为一个批次。该分批策略可将神经元总数从数百万锐减至数万，最终把整数线性规划问题的求解时长缩短至约 10 秒。

七、实现方案

PowerInfer 的在线推理引擎基于llama.cpp[17] 开发，共计新增 4200 行 C++ 与 CUDA 代码。llama.cpp 是面向个人电脑、业界领先的开源大语言模型推理框架。结合离线求解器的输出结果，PowerInfer 对模型加载器进行改造，实现大语言模型在 GPU 与 CPU 之间的分布式部署；同时针对 CPU-GPU 混合执行场景优化推理引擎，并为两类计算单元新增 10 个面向神经元特性的算子。llama.cpp 其余组件与功能均保持不变。例如，键值缓存（KV 缓存）仍存放于 CPU 内存中，从而为高频激活神经元腾出更多 GPU 显存。此外，项目还在 transformers 框架 [53] 中添加约 400 行 Python 代码，使其可作为 PowerInfer 的离线性能分析工具与求解器使用。

当前版本的 PowerInfer 可支持多款主流、不同参数量级的大语言模型系列。针对这些模型，团队采用维基百科 [13] 等通用语料库训练在线激活预测器，且在训练过程中刻意未使用任何下游任务数据集。激活预测器的训练虽耗时较长（通常为数小时），但仅需执行一次，且可借助多块 GPU 进一步提升训练速度。

本次实验的测试负载取自对话机器人竞技场数据集 [62]、ChatGPT 提示词数据集 [35] 以及 Alpaca 数据集 [50]，覆盖大语言模型的各类典型应用场景，也是真实大语言模型服务场景中最具代表性的数据集。其中 ChatGPT 提示词数据集收录了用户与 ChatGPT 的交互对话内容 [38]，Alpaca 数据集则由 GPT-3.5 通过自指令方式生成指令样本构成。

对比基准系统
本文将 PowerInfer 与两款当前主流的本地大语言模型推理框架 llama.cpp [17] 和 SpecInfer [33] 进行性能对比。llama.cpp 是目前本地部署场景中应用最为广泛的大语言模型推理框架，由于其原生并不支持 OPT 模型，为保证对比的公平性，本文对 llama.cpp 进行扩展，使其能够兼容 OPT 模型。SpecInfer 则是典型的推测解码框架，该框架先借助轻量化的草稿模型生成文本词元，再利用原始大模型对批量词元进行校验。

本文在 2.2 节中分析指出，FlexGen [43]、DejaVu [28] 等其他同类方案，在本文所研究的低延迟敏感场景下会产生更高的推理时延，因此未将其纳入对比范围。

8.2 端到端性能测试

本次实验首先在批大小为 1（本地部署的典型配置 [7]）的条件下，对比 PowerInfer、llama.cpp 与 SpecInfer 的端到端推理性能。由于真实场景中对话文本的输入、输出长度存在差异 [23]，实验从 Alpaca 数据集与 ChatGPT 提示词数据集中抽取词元长度在 8 至 128 区间的样本，测试模型的文本生成速度。

图 10 展示了搭载英伟达 RTX 4090 的高端主机（PC-High）上，不同模型在各类输入输出配置下的生成速度。整体来看，PowerInfer 的平均生成速度可达 8.32 词元 / 秒，峰值速度为 16.06 词元 / 秒，性能大幅领先 llama.cpp 与 SpecInfer：相较二者平均提速分别达到 7.23 倍和 6.19 倍。针对 Falcon-40B 模型，其相对 llama.cpp 的最高提速比更是达到 11.69 倍。尽管 SpecInfer 采用了推测解码技术，但由于模型体量超出 GPU 承载能力，在校验阶段会产生频繁的数据交换，该过程耗时占比超过 95%，因此 SpecInfer 的整体性能并未明显优于 llama.cpp。随着输出词元数量增加，PowerInfer 的性能优势会进一步凸显，原因在于文本生成阶段在整体推理耗时中的占比持续提升。在该阶段，CPU 与 GPU 上仅有少量神经元被激活，相比 llama.cpp 能够减少大量无效计算。以 OPT-30B 模型为例，其每生成一个词元仅会激活约 20% 的神经元，且绝大部分激活神经元都在 GPU 上完成运算，这也是 PowerInfer 神经元感知推理引擎带来的优势。

图 11 数据表明，在低端主机（PC-Low）环境中，PowerInfer 依旧能够显著提升性能，相对 llama.cpp 平均提速 4.71 倍、最高提速 7.06 倍；相对 SpecInfer 平均提速 5.97 倍、最高提速 7.47 倍。不过其性能提升幅度不及高端主机，核心原因是低端主机仅配备 11GB 显存，该限制约束了可部署至 GPU 的神经元数量。对于参数量达到 300 亿及以上的模型，这一问题尤为突出，大量激活神经元不得不交由 CPU 处理。

图 12 统计了 PowerInfer 与 llama.cpp 架构下，CPU 和 GPU 的神经元负载分布情况。神经元负载指各计算单元承担的激活神经元运算量占比。在高端主机环境中，PowerInfer 将 GPU 的神经元运算负载占比从原本的平均 20% 提升至 70%，即由 GPU 处理七成的激活神经元。但当模型内存需求远超 GPU 显存上限时（例如在 11GB 显存的 RTX 2080Ti 上运行显存占用 60GB 的模型），GPU 的神经元负载占比会降至 42%。这是因为显存不足以容纳所有高频激活神经元，一部分相关运算只能转移至 CPU 执行。

在 输入文本较长、输出词元极少（不足 8 个） 的低频场景 [35] 中，PowerInfer 的性能提升效果有限，提速区间仅为 1.07 倍（见图 11）至 4 倍（见图 10）。该场景下，一次性处理大量词元的提示词处理阶段成为决定推理速度的关键。此时每个词元会激活不同的神经元集合，模型激活稀疏度大幅下降，CPU 因需要处理大量低频激活神经元且自身算力存在瓶颈，进而成为整个推理流程的性能短板。

8.3 消融实验

8.3.1 性能拆解分析


图 15 拆解了 PowerInfer 各模块对整体加速效果的贡献。本实验采用逐步集成的方式，将 PowerInfer 的各项功能依次接入 llama.cpp。
第一步，在 llama.cpp 中新增预测器与神经元感知算子（标记为 + PO），使 CPU 与 GPU 仅对已激活神经元执行计算。该版本仍沿用按层分配计算任务的模式，每一层的运算全程由 GPU 或 CPU 单独承担。
在 + PO 的基础上，继续接入 PowerInfer 混合推理引擎（标记为 + Engine）。借助该引擎，神经元感知算子可在 GPU 与 CPU 上同步处理同一层内的神经元。此版本采用简易神经元划分策略，随机将神经元分配至 GPU。最后一步，将本文第 6 节设计、由离线求解器生成的优化分配策略（标记为 + Policy）集成至 + Engine 架构中，构成功能完整的 PowerInfer。
仅接入 + PO 时，Bamboo-7B 与 Falcon-40B 的推理性能分别提升至原有的 1.87 倍、3.32 倍，优化效果主要来自剔除无效的未激活神经元计算。叠加 + Engine 后，两款模型的加速比进一步升至 2.60 倍与 7.80 倍。该模块实现了精细化的神经元分配与层内并行计算，大幅提升了 GPU 的运算占比。最终集成 + Policy 后，二者加速比达到 3.62 倍与 11.69 倍。
本文设计的分配策略核心优势在于能够精细平衡层内通信开销。+Engine 所用的随机划分策略未考虑神经元激活热度以及 CPU、GPU 之间的层内通信损耗，常会抵消将高频激活神经元部署至 GPU 带来的性能收益。而 PowerInfer 的优化策略可更好地均衡 CPU 与 GPU 之间的计算负载和通信开销。

图 18a 结果表明，在输入长度为 1 的纯生成任务中，PowerInfer 大幅缩小了 RTX 4090 与 A100 之间的性能差距。在 PC-High 上，使用原生 llama.cpp 运行 OPT-30B、Falcon-40B 时，性能相比 A100 上的 vLLM 分别落后 93%、92%；接入 PowerInfer 后，性能差距收窄至 18%、23%。图 18b 对应输入长度为 64 的场景，尽管提示词处理阶段整体稀疏度有所下降，但 PowerInfer 仍将二者的性能差距控制在 28%、29%。剩余的性能差距主要源于 CPU 负载过高，此时 CPU 已成为整体推理的性能瓶颈。

实验结果表明，无论模型规模大小、任务类型如何，PowerInfer 造成的精度损失都微乎其微，该结论也与过往相关研究成果一致 [28]。尽管 Transformer 每一层的预测准确率均高于 95%，但仍会出现个别激活神经元被漏判的情况，这使得模型在部分下游任务上的表现出现小幅波动。

本文进一步分析了预测错误神经元的特性及其对推理精度的影响。研究发现，被预测器误判的神经元，通常对所在网络层的输出影响极小。这类神经元无论处于激活还是未激活状态，都几乎不会改变整体计算结果，因此也更容易出现预测偏差。

为量化该影响，本文基于 OPT-7B 模型开展实验：对比原始稠密计算结果与基于预测器的计算结果之间的余弦相似度，最终测得神经元预测错误对输出结果的影响仅约 0.4%。这一微小差异进一步证明，即便偶尔出现神经元预测失误，PowerInfer 依旧能够保持原模型的推理精度。

九、相关研究

大语言模型权重稀疏化
模型剪枝技术 [19,20,31] 通过将权重置零来削减参数量，典型方案包括 SparseGPT [14] 与 Wanda [49]，二者可实现 50% 的权重稀疏度。SparTA [64] 对稀疏矩阵进行拆分，同时利用稀疏张量单元与单指令多线程核心完成计算。Flash-LLM [54] 针对张量核心上的稀疏矩阵 - 矩阵乘法，提出按稀疏形式加载、按稠密形式计算的处理思路。但上述方法与大语言模型原生的激活稀疏特性相互独立，不仅通常会造成精度损失，也难以实现理想的实际加速效果 [34]。与之不同，PowerInfer 依托模型天然的激活稀疏特性，在保障推理效果的同时兼顾运行效率。

大语言模型注意力层稀疏化
PowerInfer 主要利用多层感知机模块的激活稀疏特性。值得一提的是，注意力模块同样存在稀疏性，其来源主要分为两类：一是注意力头层面，处理当前词元时仅需调用部分注意力头参与计算 [28]；二是注意力头内部，通过裁剪或转储无效键值缓存缓解大语言模型服务场景下的内存瓶颈，H2o [60]、InfiniGen [25] 均采用了该思路。PowerInfer 所使用的激活稀疏方案与注意力稀疏相互独立，二者结合可进一步降低推理时延。

大语言模型推测推理
推测推理技术 [7,8,16,52] 也可用于部署显存超出 GPU 容量的大模型。该技术先由轻量化模型预生成词元，再交由主模型批量校验。SpecInfer [33] 借助该思路有效减少了大语言模型的解码步数。推测推理与本文研究方向不同，但将其与 PowerInfer 相结合，有望进一步提升推理速度。

面向大语言模型的部署优化
现有大量面向通用机器学习模型的推理服务系统 [18]。随着 Transformer 架构的普及，多款专为该架构设计的部署系统相继问世 [9,12,22,42,65]。Orca [57] 提出迭代级调度策略；vLLM [23] 采用分页注意力机制，将词元存储在 GPU 内存的非连续地址中，突破了键值缓存必须连续存储的限制。但这类方案均未解决整机模型无法完整载入 GPU 显存的个人终端部署难题。

十、总结

PowerInfer 是一款针对大语言模型优化的高速推理系统，充分挖掘大模型推理过程中的局部性特征。该系统结合自适应预测器与神经元感知算子，有效利用神经元激活与计算环节的稀疏性。在不损失模型精度的前提下，PowerInfer 相较 llama.cpp 等主流框架，最高可将大语言模型推理速度提升至原来的 11.69 倍。
