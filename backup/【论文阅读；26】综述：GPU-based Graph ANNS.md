GPU-Accelerated Algorithms for Graph Vector Search: Taxonomy, Empirical Study, and Research Directions
# 摘要
近似最近邻检索（ANNS）是支撑各类大规模数据挖掘与机器学习应用的基础技术。随着数据集规模不断扩增，借助 GPU 实现向量高效检索已成为必然趋势。目前，基于图的检索方法是近似最近邻检索领域的最优方案，但针对这类算法如何适配现代 GPU 架构进行优化、以及其在真实场景下的端到端表现，学界仍缺乏系统性研究。
本文针对**GPU 加速的图向量检索算法**开展全面综述与实证分析。首先，本文构建了一套完备的 GPU 优化策略分类体系，并厘清了算法任务与 GPU 硬件执行单元之间的对应关系。随后，基于八大主流大规模基准数据集，对六种代表性算法展开评测，全面分析图索引构建与查询检索两大环节的性能表现。
实验表明，距离计算是该类算法最核心的计算瓶颈；而在大规模部署场景中，**CPU 主机与 GPU 设备间的数据传输**是造成检索高时延的首要因素。本文同时剖析了不同系统方案在可扩展性与内存占用之间的权衡关系。
本研究成果可为设计高可扩展、高稳定性的 GPU 近似最近邻检索系统提供明确指导，也可为知识发现与数据挖掘领域提供一套完整的评测基准。
# 一、引言
k 近邻搜索（KNNS）是数据挖掘、信息检索与向量检索领域的基础核心技术。具体而言，给定包含n条数据样本的数据集X={x1​,x2​,…,xn​}（其中样本xi​属于d维实数空间Rd），以及同样位于d维实数空间Rd内的查询点q，该任务要求依据指定距离度量方式，从数据集X中检索出与查询点q距离最近的k个数据点。传统精确解法需要逐一计算查询点与全部n个数据点之间的距离，单次查询的时间复杂度为O(nd)。在小规模数据场景下，O(nd)的时间复杂度看似具备可行性，但在推荐系统、检索增强生成等诸多实际应用场景中，数据集样本量n与向量维度d往往极大（通常满足n≥106、d≥100）。这类业务场景对用户查询的实时响应能力有着严苛要求，在此前提下，即便是线性时间复杂度算法也难以落地，无法满足低延迟的使用需求，也不能适配当下海量、高维的数据集规模 [7,29]。

为解决这类可扩展性难题，近似最近邻搜索（ANNS）放宽了精确匹配的约束条件。该技术以损失微小的检索精度为代价，大幅提升检索效率。围绕高性能近似最近邻搜索技术的研究推动了相关方法的不断迭代发展。早期出现的**基于树**的检索方法 [3,8,9,14,16,44,50,53]，通过将搜索空间划分为多层级区域，实现对数级别的搜索复杂度。但在高维空间中，不同区域必然出现重叠问题，导致这类方法的性能大幅下降。为改善这一缺陷，**基于哈希**的方法 [1,2,10,11,27,40,41,52,57,65] 应运而生：该类方法将高维向量映射为紧凑的二进制编码，以此实现次线性复杂度检索，然而其普遍存在检索精度损失问题，且检索效果对哈希函数的参数设置高度敏感。此外，**基于量化**的方法 [6,19,24,32,35,54,61] 会将向量压缩为有限码本集合中的码字，既能减少存储开销，也可加快距离计算速度，不过码本训练过程通常会产生巨大的计算开销。

为在保证高召回率的同时实现更优异的检索效率，**基于图**的方法 [13,21,28,47,59,63,64,66] 被相继提出，并已成为当前近似最近邻搜索领域的主流技术。这类方法的核心思路是构建近邻关系图，借助贪心搜索算法在图节点间遍历，不断探索邻接节点，最终找到查询点对应的k个最近邻数据点。但早期的图检索算法 [12,17,18,25,26,42,43] 主要针对中央处理器（CPU）进行优化。图形处理器（GPU）虽可依托多线程技术并行处理大量查询请求，但针对单个查询任务，图的遍历过程基本仍是串行执行 —— 邻接节点扩展、距离计算、结果比对等操作均按顺序完成。为突破这一局限，多种面向 GPU 的加速方案被陆续提出 [24,31]。这类方案充分利用 GPU 强大的并行计算能力，并行执行距离运算与节点扩展操作，显著提升了大规模向量检索的整体吞吐率。

尽管基于 GPU 加速的图结构近似最近邻搜索算法已取得长足发展，但仍存在若干关键问题，制约了该类技术的系统化评测与实际落地应用。现有研究中，多数工作聚焦于 CPU 端检索方法 [5,56]，而针对 GPU 端方案的相关研究 [30,31]，在方法分析与实际性能评估两方面均存在明显短板，具体问题如下：
1. 缺陷一：缺乏系统化分类体系。目前尚无统一的分类标准对各类 GPU 加速技术进行归纳梳理。GPU 拥有强大的并行计算能力，但想要充分发挥其性能，需要配套的专用优化手段，而相关优化策略既未得到系统性研究，也缺少完善的文献记载。过往多数研究仅关注算法最终效果，对于线程调度、内存管理等 GPU 核心优化策略的探讨十分有限。
2. 缺陷二：缺少统一的评测框架。目前没有一套完整的评测体系，能够覆盖 GPU 端近似最近邻搜索技术的全运行流程，也无法全面衡量其多维度性能表现。现有研究大多只关注检索环节的加速效果，却忽视了图构建、数据迁移等在大规模部署场景中至关重要的额外开销。
3. 缺陷三：全链路评测不完整。当前基准测试的方式较为零散，性能指标往往仅基于独立计算核心的运行耗时统计。这种评测方式忽略了内存分配、高速串行计算机扩展总线（PCIe）数据传输、主机与设备间同步等环节带来的显著影响，最终得出的结论可能无法真实反映系统在实际场景中的运行性能。

为填补 GPU 加速图结构近似最近邻搜索在算法理论设计与硬件实际运行效率之间的鸿沟，本文对该领域现有研究成果展开全面、多维度的分析。鉴于当前学界尚未理清现代 GPU 架构与高维检索基础运算之间的适配关系，本文首先构建一套完整的 GPU 加速技术分类体系，明确算法任务与硬件执行单元之间的对应关系。在此基础上，选取六种主流算法，基于八个大规模基准数据集开展大量实证实验，逐层拆解全链路运行性能，挖掘潜藏的性能瓶颈。结合理论分析与严谨的基准测试结果，本文厘清了计算吞吐率、数据迁移开销与内存可扩展性三者之间复杂的权衡关系。

本文的主要研究贡献总结如下：
1. 构建结构化的 GPU 优化策略分类体系，为相关研究者选取适配 GPU 近似最近邻搜索场景的加速技术提供系统性参考。
2. 阐明近似最近邻搜索基础运算与现代 GPU 硬件单元的映射机制，解析如何利用硬件架构特性最大化运行效率。
3. 基于不同规模的数据集，评测主流算法的可扩展性，并拆解各功能模块的运行耗时。
4. 证实主机与设备间的数据传输延迟，是大规模检索场景下制约整体性能的关键瓶颈。

如附录 A.1 中的图 8 所示，图形处理器（GPU）是专为大规模并行计算设计的专用处理器。在近似最近邻搜索场景中，GPU 强大的并行计算能力与高内存带宽，能够有效加速高维距离计算、大规模图遍历等计算密集型操作。

GPU 硬件体系中，与可扩展近似最近邻搜索密切相关的核心组件包括高速串行计算机扩展总线（PCIe）、流式多处理器（SM）以及线程，具体说明如下：
- **PCIe 总线**：用于连接主机端 CPU 与设备端 GPU 的高速总线。在近似最近邻搜索任务中，PCIe 负责将海量向量数据集与批量查询请求传输至 GPU 全局内存。采用异步传输方式降低 PCIe 传输开销，是掩盖传输延迟的关键手段。
- **流式多处理器（SM）**：GPU 执行运算的基础硬件单元。单块 GPU 包含多个流式多处理器，每个处理器均配备 CUDA 核心、张量核心，以及寄存器、共享内存等片上资源。
- **线程**：GPU 执行层级中最小的调度单元。每个线程在一个 CUDA 核心上执行指令序列。在并行化的近似最近邻搜索实现方案中，单个线程可负责处理向量的指定维度计算，也可与其他线程协同维护候选优先队列。

# 三、模块拆解分析
本章对**图索引构建阶段**进行分类梳理，并从模块与技术层面对**检索阶段**展开详细解析。索引构建阶段分为批量插入、迭代优化、分治三类方法，同时分析各类方法在 GPU 架构下的并行加速策略。检索阶段则拆解为候选节点查找、批量距离计算、数据结构维护三大环节，并深入剖析不同算法所采用的并行加速原理与实现技术。
## 3.1 图索引构建阶段
受硬件架构差异影响，目前尚未形成一套通用的图索引构建框架。实际应用中主要存在三类主流构建方案：**并行迭代优化（第一类）**、**批量增量插入（第二类）** 与**并行分治（第三类）**。三类方案可组合使用，以进一步提升整体性能。各类方法的详细介绍见附录 A.2。
**第一类：并行迭代优化**
该方法遵循迭代式图优化思路：先构建初始的近似近邻图，再通过反复执行邻域挖掘、重排序与边剪枝操作，持续迭代优化图结构。

这类方案通常先依托 NN-Descent 算法 [15] 生成初始 k 近邻图，再将图数据迁移至 GPU 完成后续优化。执行过程中，图中的每个顶点单独分配至一个 GPU 线程块，各线程块相互独立、并行挖掘该顶点的 N 跳邻接节点，并开展迭代优化。多轮迭代完成后，优化得到的高质量近似 k 近邻图会回传至 CPU 并持久化存储至磁盘，供后续检索使用。

相较于 NSG [18] 等传统 CPU 端算法，GPU 导向的迭代优化具备两大核心优势：其一，距离计算可实现全批量处理，能够充分发挥 GPU 单指令多数据流单元与内存带宽的性能潜力；其二，各个顶点的优化过程相互独立，支持在 GPU 上大规模并行执行与更新，大幅提升图索引的构建效率。

具体来看，CAGRA [48] 采用**基于排名的重排序**方式统计 N 跳邻接节点数量，不再单纯依据欧氏距离排序邻接点，而是按照节点间的连通性排名完成重排。该方式不仅显著加快建图速度、降低内存占用，大量实验也证实，其检索精度与基于距离的优化方案基本持平。

Tagore [39] 则基于**两阶段 GNN-Descent 算法**[55] 对传统 NN-Descent [15] 进行改进：第一阶段沿用原始 NN-Descent 的采样策略完成图结构初步优化；第二阶段中，每个节点从自身最近的 m 个邻接点的邻接列表中采样，进一步精简自身邻接节点集合。
**第二类：批量增量插入**
该方案由传统 CPU 增量插入方法演变而来，核心是在 GPU 上以批次为单位，并行完成数据点的图插入操作。每个数据点对应一个独立 GPU 线程块，线程块自主挖掘并匹配该点的邻接顶点。整批数据全部插入完成后，生成的图结构回传至 CPU 并保存至磁盘。

与传统 CPU 增量建图方式相比，该方案充分释放 GPU 的大规模并行能力，可对大量待插入数据同时开展邻域检索，有效降低建图延迟、提升吞吐率，在海量数据集场景下优势尤为突出。此外，批量插入的执行模式让内存访问模式更规整、更适配缓存机制，进一步提升 GPU 利用率与整体计算效率。

GANNS [60] 是该类方法的代表性算法，其通过在每一批待插入数据内部先行构建局部 k 近邻图来优化建图质量。批次内局部图能够刻画新增节点之间的邻接关系，避免并行插入过程中遗漏同批次内距离相近的节点。将批次级 k 近邻结构融合至全局图后，GANNS [60] 在保障检索精度的同时，维持了 GPU 端优异的建图吞吐能力。

**第三类：并行分治**
该方法采用分治思想，将大规模数据集划分为多个子数据空间，在 GPU 中并行构建各个子空间对应的局部 k 近邻图，最终合并所有子图得到完整的全局图。这种分层并行模式可高效利用 GPU 资源，在保证图连通性的前提下，大幅加快海量数据的建图速度。

GGNN [22] 是该类算法的典型代表，采用分层架构：首先并行构建多个局部子图，再自上而下完成子图合并。其构建出的图质量与串行方法相当，同时在 GPU 上实现显著加速。此外，顶层图还可作为初始种子结构，为后续检索流程提速。
## 3.2 检索阶段
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260610113947.png)
如图 2 所示，基于图的近似最近邻检索拥有一套标准化执行流程，该流程将原本串行的图遍历操作改造为适配 GPU 架构的可并行任务。常规实现中，一组线程束（包含 32 个 GPU 线程）负责处理单个查询点，各线程束相互独立运行、互不干扰。整套流程划分为三大核心环节，协同完成高效近邻查找，并充分发挥 GPU 的并行特性。附录 A.3 中的算法 1 展示了适配 GPU 的标准贪心检索流程，清晰体现三大环节在并行执行模型中的调度逻辑。

**环节一：并行候选节点查找（S1）**
在 GPU 架构中，由一组线程协同处理单个候选顶点。线程配合提取尚未访问的最优候选节点，并并行加载其所有邻接节点。由于邻接节点存储在全局内存的不同地址偏移处，并发加载不会产生写冲突，加载完成的邻接节点可快速暂存至共享内存。

传统 CPU 图检索中，候选节点提取与邻域扩展多为串行执行，即便开启并行也仅能做到有限的核级并行，且存在内存访问不规则的问题。而 GPU 方案依托细粒度的线程束级并行与高内存带宽，大幅加快候选节点查找速度，提升检索整体吞吐率。

早期 GPU 图检索算法 SONG [62] 仅使用单个线程加载候选节点的邻接点，线程块内其余线程在此阶段处于空闲状态。后续 GGNN [22]、GANNS [60] 等算法对此进行了深度优化，让线程块内所有线程并行加载不同邻接节点。这种并行邻接节点加载策略有效提升内存吞吐、降低访问延迟，进一步优化 GPU 端候选节点查找效率。

BANG [33] 则是特例：该算法的图数据并未全部常驻 GPU 显存，邻接节点加载操作在 CPU 端完成，加载后的邻接信息通过异步传输发送至 GPU 进行后续处理。该设计使 BANG [33] 能够处理显存容量无法容纳的超大规模图。

主流算法普遍采用布隆过滤器剔除已完成距离计算的邻接节点，但 GANNS [60] 提出了不同思路：该算法认为距离计算可依托 GPU 细粒度线程级并行实现高效加速，因此完全舍弃布隆过滤器。原因在于布隆过滤器会引发随机内存访问，严重损耗 GPU 内存带宽。

PathWeaver [34] 采用**方向引导筛选**策略，在保证检索精度的同时减少距离计算量。算法先计算查询点与节点的方向向量，对照预设统计表统计所有邻接节点的符号匹配位数，优先选择匹配位数最多的邻接点作为候选节点。

**环节二：批量距离计算（S2）**
批量距离计算通过任务划分实现并行化：将欧氏距离计算任务拆分给多个线程，每个线程负责计算查询点与数据点的部分维度，各线程算出的局部结果最终通过归约运算整合，得到完整欧氏距离。相较于 CPU 实现方案，该环节是基于图的近似最近邻检索中加速效果最显著的部分，充分发挥了 GPU 的线程级并行能力与高内存带宽优势。

SONG [62]、GGNN [22] 等多数算法采用**一线程束对应一个数据点**的模式，由包含 32 个线程的线程束计算单个数据点与查询点之间的距离。CAGRA [48] 则提出**线程束拆分**技术：将一个线程束划分为多个 8 线程小组，每个小组独立计算不同数据点的距离。该设计提升了 GPU 线程占用率与并行度，进而提高整体吞吐。附录 A.4 通过实验对比了不同分组尺寸的性能，结果表明，在召回率相同的前提下，线程束拆分方案的每秒查询量高于传统的一线程束对应一个数据点的方案。

**环节三：数据结构维护（S3）**
本环节根据上一阶段的计算结果更新候选节点列表。通常情况下，同一线程束或线程块内的不同线程负责向优先队列、候选列表等数据结构的不同位置写入数据，彻底规避写冲突，最大化 GPU 并行执行效率。得益于该设计，GPU 可并发更新多条候选记录，相较传统 CPU 方案，本环节的运算速度得到大幅提升。

SONG [62] 与 PilotANN [23] 使用堆结构维护候选列表，完成单次距离计算后，仅由指定的首个线程将结果写入堆中。GGNN [22] 选用循环队列，插入新节点时由不同线程分别完成数组内单个元素的移位操作，极大加快数据结构的更新速度。

GANNS [60]、CAGRA [48] 等算法则采用**延迟校验**策略：距离计算完成后不立即将节点加入候选列表，而是先将所有距离值存入未访问距离数组，待全部邻接节点的距离计算完毕后，利用 GPU 专用的双调排序算法对该数组排序，再合并至候选列表。与逐个插入节点的方式相比，双调排序能够充分调动所有线程的计算能力。

## 4.2 图索引构建效果评估
### 4.2.1 构建效率评估
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260610114911.png)
在近似最近邻搜索场景中，图索引构建效率至关重要，它不仅决定系统初始部署速度，也直接影响系统更新数据的频次。本次基于 8 个中大型数据集，对 7 款代表性算法的构建效率开展评测，核心观测四项指标：构建耗时、平均度数、连通分量数、峰值内存占用，同时对比 CPU 与 GPU 端表现。注：BANG 算法与 VAMANA [28] 采用相同的图构建机制，因此表格中不再单独列出 BANG。

附录 A.6 的表 2 与表 4 结果显示，GPU 加速算法的构建速度大幅领先传统 CPU 算法。实验充分证明，依托 GPU 的大规模并行计算能力，是处理海量、高维数据集的必要条件。CAGRA [48]、GGNN [22] 等算法充分调用 GPU 资源，并行完成距离计算与图结构更新，相较于 CPU 串行执行方案，速度提升数个数量级。但提速的同时，这类算法的 GPU 显存占用也显著增加，部分算法在超大数据集场景下甚至出现显存溢出问题。

**分析结论**：后续优化需重点研究 CPU-GPU 混合架构方案，平衡构建速度与内存限制。利用主机内存存储海量图数据，同时保留 GPU 负责并行计算的优势，可有效解决显存溢出问题，保障算法在十亿级数据规模下的可扩展性。
### 4.2.2 图结构质量评估
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260610115032.png)
以往相关综述 [56] 多以图结构与标准 k 近邻图的拓扑差异作为图质量评判依据，本文则采用**面向检索**的评价指标：达到目标检索精度所需的平均查询路径长度。高质量的图结构可让贪心检索以更短的遍历跳数快速找到近邻节点。

实验固定不同数据集的图节点度数：GIST [29] 设为 128、DEEP10M [7] 设为 64、DEEP1M [7] 与 SIFT1M [29] 设为 32，统计召回率 @10 达到 90% 时的平均路径长度。

由图 3 可知，各算法的图遍历效率存在明显差异。CAGRA [48] 与 BANG [33] 在所有数据集上均表现出更优的图质量，查询路径长度始终最短。实验结果表明，图质量的核心在于结构的**可遍历性**，而非单纯的连通性。CAGRA [48] 等采用高级边剪枝与邻接点筛选策略的算法，能够在高维空间中构建出高效 “检索通路”，让贪心检索只需更少跳转即可收敛到目标节点。

**分析结论**：后续研究应结合硬件特性优化图剪枝策略，让拓扑结构中的 “检索通路” 适配 GPU 内存访问规则，进一步减少遍历跳数与随机访问延迟。
## 4.3 检索性能分析
### 4.3.1 含数据传输的检索测试
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260610115059.png)
本部分将端到端检索延迟拆分为**主机向设备传输（HtoD）** 与**设备端检索**两个环节开展评测。由于查询得到的 Top-K 结果数据量极小，设备向主机传输（DtoH）产生的延迟可忽略且各算法表现基本一致，因此该环节不纳入统计。

实验选用数据规模在 29 万至 1 亿条之间的多组数据集，统一测试 10000 条查询请求在召回率 @10 为 90% 时的总处理耗时。

表 3 结果表明，SONG [62]、GGNN [22]、GANNS [60]、CAGRA [48] 等纯 GPU 算法面临严重的数据传输瓶颈。随着数据集规模增大，主机向设备的数据传输耗时呈非线性增长：在 DEEP1M [7] 数据集上，传输耗时占总延迟的 40%~80%；在 DEEP100M [7] 数据集上，该占比更是超过 97%，导致 GPU 计算核心的高速优势难以发挥。

反观 BANG [33]、PilotANN [23] 等混合架构算法，借助 CPU 分担部分任务有效降低了传输耗时，但主机端的计算开销又造成检索整体延迟上升。

以上现象说明，在大规模近似最近邻搜索场景中，**PCIe 总线带宽**而非 GPU 计算能力，才是制约性能的核心因素。当数据传输耗时占总运行时长 90% 以上时，单纯优化检索计算核心的收益会持续递减。

**分析结论**：后续研究需着力攻克 “内存墙” 难题，可采用传输与计算异步重叠、传输过程中对向量数据深度压缩、全索引常驻 GPU 显存等方案，从根源上削减数据传输开销。
### 4.3.2 传输开销拆解分析
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260610115219.png)
为深入剖析数据传输瓶颈的构成，本实验将传输耗时划分为三部分：向量数据传输、图结构传输、查询向量传输，并统计各部分的耗时占比。

由图 4 可见，SONG [62]、GGNN [22]、GANNS [60]、CAGRA [48] 等纯 GPU 算法的传输开销构成几乎一致，这类算法需要将全部向量数据与图索引完整迁移至 GPU。下文选取当前最优算法 CAGRA [48] 作为该类算法代表，其余纯 GPU 算法的详细结果见附录 A.7 图 11。

混合架构算法的开销构成则差异明显，这源于 BANG [33] 与 PilotANN [23] 采用了不同的软硬件协同执行策略。

纯 GPU 算法的传输瓶颈与数据总量强相关；而 BANG [33]、PilotANN [23] 等混合算法虽然通过任务下放至 CPU 免去了部分传输开销，但往往只是将瓶颈转移，同时还会新增软硬件同步延迟。

**分析结论**：后续系统优化需提升计算与传输的时间占比，一方面对占比最高的向量数据进行压缩，另一方面充分利用时间重叠技术，让数据传输与 GPU 计算并行执行。
### 4.3.3 无数据传输的检索测试
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260610115249.png)
为探究算法的计算效率上限，本实验设定**全图结构常驻 GPU 显存**的静态场景，剔除数据传输带来的干扰，单独评测计算核心的检索性能，体现算法在 GPU 设备上的原生加速能力。

结合图 5 与附录 A.8 图 10 可知，纯 GPU 算法相较传统 CPU 基准算法，提速幅度普遍超过 10 倍。其中在小规模数据集、召回率 @10 为 90% 的条件下，CAGRA [48] 的每秒查询数始终位居第一，且召回率提升时，其吞吐率依旧保持稳定。

BANG [33]、PilotANN [23] 等混合架构算法虽然具备良好的可扩展性，但由于必须在 CPU 端执行部分逻辑产生额外延迟，纯检索吞吐率有时甚至不及 CPU 基准算法。

实验证明，实现设备端最大化并行能力，是常驻式图检索取得高性能的关键。

**分析结论**：下一代高性能检索计算核心，需尽可能减少分支跳转与主机设备同步操作，充分释放 GPU 的海量计算吞吐潜力。
### 4.3.4 检索子模块耗时拆解
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260610115306.png)
为定位影响检索延迟的关键操作，本实验将计算核心运行耗时细分为三大模块：并行候选节点查找（CL）、批量距离计算（DC）、数据结构维护（DM）。选取四类特征各异的数据集，统一将所有算法的召回率 @10 设定为 90%，统计各模块的时间占比。

结合图 6 与附录 A.9 图 12 可得：对于 GGNN [22]、GANNS [60]、CAGRA [48] 等经过深度优化的原生 GPU 算法，**距离计算**是最主要的耗时模块，占总运行时长的 60%~80%。

SONG [62] 等早期实现方案因其余模块优化不足，距离计算的耗时占比相对更低。现代混合算法则呈现不同特征：BANG [33] 引入乘积量化技术，将距离计算耗时占比降至 30%~40%；PilotANN [23] 受 CPU-GPU 协同机制限制，候选节点查找模块产生大量额外延迟。

高维向量距离计算是绝大多数算法的核心瓶颈，而量化技术能够有效缓解这一计算压力。但需要注意的是，在 CPU-GPU 混合架构中，这类优化会将瓶颈转移至 CPU 端，复杂的检索逻辑与数据管理任务会造成 CPU 负载饱和。

**分析结论**：设计极致高速的检索计算核心时，需在 “通过量化减少运算量” 与 “合理分配软硬件负载” 之间取得平衡，避免 CPU 过载成为新的性能短板。
## 4.4 可扩展性测试
本章节统计的每秒查询数均基于计算核心运行时长计算，彻底剥离数据传输的影响，单独验证算法本身的可扩展能力。
### 4.4.1 数据规模可扩展性
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260610115337.png)
基于 DEEP [7] 系列数据集开展测试，数据规模从 100 万条逐步提升至 1 亿条，评测各 GPU 加速算法在不同数据量下的性能表现。

如图 7 (a) 所示，随着数据集规模增大，所有算法的性能均出现自然下降，但 CAGRA [48] 的性能衰减幅度最小。即便数据量达到 1 亿条，该算法仍能维持较高处理速度；其余对比算法则因图遍历深度增加、内存压力上升，吞吐率出现大幅下滑。

CAGRA [48] 的优异表现证明，经过优化的线程束级并行机制，能够有效掩盖大图结构带来的延迟突增问题。

**分析结论**：算法的大规模可扩展性，核心在于屏蔽大图结构引发的内存延迟。后续研究需持续提升线程束占用率，确保数据规模拓展至十亿级时，图遍历复杂度不会制约整体吞吐能力。
### 4.4.2 吞吐能力测试
基于 SIFT1M 数据集，通过调整查询批次大小，测试各类基于图的近似最近邻搜索算法的吞吐特性，实验统一保持召回率 @10 为 90%。

如图 7 (b) 所示，增大查询批次可显著提升所有算法的每秒查询数。由于每条查询任务都对应独立的 GPU 线程，更大的批次能够更好地掩盖内存访问延迟，硬件吞吐能力会随负载增加近似线性提升，直至硬件资源达到饱和状态。

**分析结论**：未来架构设计应重点实现动态批处理与多流并发执行，最大化硬件资源利用率。脱离固定批次大小的限制，让系统无论输入负载高低，都能充分发挥线程并行能力，并平摊任务启动带来的固定开销。
# 五、设计指导原则
现有基于 GPU 加速的图结构近似最近邻搜索算法仍存在明显短板。早期算法 SONG [62] 对检索环节的优化不足，整体性能表现一般。GGNN [22]、GANNS [60] 与 CAGRA [48] 相较传统 CPU 图索引算法虽有大幅提升，但随着数据集规模不断扩大，这类纯 GPU 算法会面临两大核心难题：
- 主机向设备的数据传输开销持续走高：在 DEEP1M 数据集上，该开销占总耗时的 40% 至 80%；而在 DEEP100M 数据集上，占比更是超过 97%。
- GPU 显存需求呈指数级增长，极大制约了算法在超大规模数据集下的可扩展性。
BANG [33]、PilotANN [23] 这类 CPU-GPU 混合架构算法选择将大部分检索操作交由 CPU 执行，以此缓解 GPU 显存压力。但该方案只是将性能瓶颈转移至主机端：大量距离计算任务落在 CPU 上，支撑十亿级数据集仍需主机配备大容量内存，此时系统整体性能会受限于主机内存带宽与访问延迟，而非 GPU 的计算能力。

据此，下一代基于 GPU 加速的图结构近似最近邻搜索算法需要在两大方向实现突破：
- 针对百万至千万级数据集：减少向 GPU 传输的数据量，降低算法对显存的依赖。
- 针对十亿级数据集：在削减主机内存占用的同时，充分发挥 GPU 的并行计算能力，进一步提升查询吞吐率。
# 六、总结
本文对基于图结构的 GPU 近似最近邻搜索算法开展了系统性分析。实验选用 8 个规模介于 29 万至 1 亿条样本的数据集，评测 6 款 GPU 算法与 2 款 CPU 算法，全面考察检索性能（每秒查询数、查询路径长度、距离计算耗时、批处理吞吐率及参数敏感度）与索引构建效率（构建耗时、内存占用、图连通性）。

研究证实，距离计算是检索阶段的主要性能开销。本文还首次量化分析了 CPU 与 GPU 之间的数据传输成本，发现数据传输耗时远高于计算核心的执行耗时。同时，本文提出将平均查询路径长度作为衡量图结构质量的指标，该指标能更直观地反映图的遍历效率，也为后续算法优化指明了明确方向。

**架构概述**
英伟达新一代架构依次历经安培架构、霍普架构至布莱克韦尔架构迭代，内存与通信带宽性能持续提升。以安培架构为例，据英伟达技术文档 [46] 介绍，该架构采用分层设计，包含多个图形处理集群（GPC）；每个图形处理集群又由纹理处理集群（TPC）与流式多处理器（SM）组成。标准安培架构 GPU 集成数十个流式多处理器，每个处理器内部搭载多组 CUDA 核心、张量核心以及共享内存单元。该架构可同时调度数千个线程并发运行，十分适配近似最近邻搜索任务的并行计算需求。

## 图类型
据现有综述文献 [56] 所述，目前各类基于图的近似最近邻搜索算法，本质上均由以下四类基础图结构衍生而来。其中，最小生成树的连通性较弱，检索平均路径偏长，因此仅用于保障图的基础连通性，不会直接作为最终使用的检索图。
**德劳内图（DG）**
在d维欧氏空间Rd中，基于点集S构建的德劳内图G(V,E)需满足如下条件：对于任意一条连接顶点u与v的边e∈E，必然存在一个同时经过u、v两点的圆，该圆内部不含其他顶点，圆的边界上最多可存在三个顶点。

**相对邻域图（RNG）**
基于点集S构建的相对邻域图G(V,E)具备以下特性：任取图中连接顶点u、v的边e∈E，对于集合中除u、v以外的任意顶点w，均满足dist(u,v)<dist(u,w) 或 dist(u,v)<dist(v,w)。

**k 近邻图（KNNG）**
k 近邻图G(V,E)的构建规则为：将点集S中的每个数据点，与其距离最近的k个邻点相连。该结构不具备对称性，即若顶点u属于v的邻域集合N(v)，无法推出顶点v也属于u的邻域集合N(u)。

**最小生成树（MST）**
针对点集S构建的最小生成树G(V,E)，是使所有边的权重之和∑i=1∣E∣​w(ei​)达到最小值的图结构。其中每条边ei​连接顶点u和v，边的权重w(ei​)定义为两点间的距离dist(u,v)。

## 面向GPU的图基anns算法
**SONG [62]**
SONG 是首款专为 GPU 设计的基于图的近似最近邻搜索算法，依托硬件架构优化实现高效检索。该算法将传统图检索流程拆解为**候选节点查找、批量距离计算、数据结构维护**三个独立阶段，消除流程间的串行依赖。算法采用固定度数邻接表、开放寻址哈希表与布隆过滤器，规避动态内存分配问题并优化内存访问模式。借助有界优先队列与选择性插入机制，SONG 实现对 GPU 内存的精细化管控；同时在并行距离计算过程中，保证数据表更新以串行方式执行。

**GGNN [22]**
GGNN 采用**自底向上的分层建图**方案，同时提升索引构建效率与查询性能。建图阶段先将数据集划分为多个小批次，逐层递归合并子图，并通过双向连边强化图的连通性，每个节点的出边数量保持固定。检索阶段，该算法基于线程块级并行机制，搭配多功能共享内存缓存与自适应停止条件，快速定位近邻节点。
针对十亿级数据集，GGNN 支持**多 GPU 分片架构**：每个分片单独构建检索图以适配单块 GPU 的显存容量，统一查询请求会在所有分片上并行执行，最终由 CPU 通过高效多路归并算法汇总各分片结果，输出全局 k 近邻。

**GANNS [60]**
GANNS 是一款基于近邻图的高性能 GPU 加速算法，图构建与检索全流程均使用 GPU 加速。建图阶段采用分治策略，将整体点集划分为多个子集，并行构建局部近邻图，再逐步合并得到全局图；图中同时保留正向边与反向边，保障图结构质量。
检索阶段以批次形式处理候选节点并完成距离计算，结合**延迟更新**与**延迟校验**机制，既规避动态数据结构带来的额外开销，又进一步提升并行执行能力。

**CAGRA [48]**
CAGRA 的图构建与检索环节均基于 GPU 加速。建图时先通过 NN-Descent [15] 算法快速生成初始 k 近邻图，再对边进行重排序并新增反向边完成优化，以此增强图的连通性与检索性能。优化后的图采用固定节点出度设计，可充分发挥 GPU 的并行计算能力，提升硬件资源利用率。
检索阶段通过随机采样初始化候选节点列表，动态更新内部 Top-M 列表与候选列表，实现高效迭代检索。

**BANG [33]**
BANG 是面向十亿级数据集设计的 GPU 加速近似最近邻搜索方法，其图结构基于经典 DiskANN [28] 算法构建。检索阶段将图结构与原始数据存放于主机内存，以此降低 GPU 显存占用；由 CPU 加载邻接节点，并将节点编号与对应数据依次传输至 GPU。
GPU 端采用分阶段执行策略，将任务划分为邻接节点过滤、等待邻接数据、距离计算、预取扩展候选节点、结果重排序多个环节。该算法结合异步数据传输与数据预取技术，在保障并行度的同时，有效缓解 CPU 与 GPU 之间的数据传输瓶颈。

**PilotANN [23]**
PilotANN 是面向大规模近似最近邻搜索的**CPU-GPU 混合架构**框架。索引构建阶段，利用奇异值分解（SVD）[20,36] 将高维向量拆分为主分量与残差分量；通过节点采样构建子图，并采用压缩稀疏行（CSR）[51] 格式将子图存入 GPU 显存，完整图则保留在 CPU 内存中。
检索采用多阶段执行策略：首先依托 GPU、子图与向量主分量完成初步检索，筛选候选节点；再由 CPU 结合残差分量并遍历更多子图做进一步优化；最后在完整图上执行贪心检索，保障检索精度。该算法结合快速入口选择（FES）与高效数据管理方案，可在超大规模数据集上实现高吞吐检索。
