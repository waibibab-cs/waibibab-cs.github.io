SVFusion: A CPU-GPU Co-Processing Architecture for  Large-Scale Real-Time Vector Search
# 摘要
近似最近邻搜索（ANNS）是信息检索、推荐系统等现代应用的底层支撑。随着向量数据规模飞速增长，面向实时向量检索的高效索引构建已成为基础刚需。现有基于 CPU 的方案虽支持数据更新，但吞吐性能低下；而 GPU 加速系统虽性能强劲，却面临动态更新困难、显存容量受限等问题，致使**兼顾精度与速度**的大规模持续向量检索场景存在严重性能短板。

本文提出**SVFusion**—— 一种面向实时向量检索的 CPU-GPU 协同框架，实现高性能 GPU 计算与在线数据更新的有机融合。SVFusion 采用分层向量索引架构，依托 CPU-GPU 协同计算，并引入自适应两阶段向量缓存机制，充分挖掘有限 GPU 显存的利用效率；同时通过流水线优化任务调度，实现数据传输与计算任务重叠执行，搭配轻量级删除处理策略，可在不阻塞现有业务任务的前提下支持并发操作。

实验结果表明，SVFusion 在查询延迟与吞吐性能上均实现显著提升，在各类负载的大规模向量数据集上，相较基线方法平均加速**20.9 倍**，且始终保持高召回率。
# 一、引言
深度学习与大语言模型的兴起，推动高维向量嵌入在文本、图像、音频等非结构化数据中得到广泛应用。高效检索这类嵌入向量，对个性化推荐、网页搜索以及大模型相关应用等任务至关重要。为在服务等级目标约束下满足低延迟、高准确率的严苛需求，现代向量检索系统越来越多地采用**近似最近邻搜索（ANNS）**，尤其是高维空间中兼顾精度与计算效率的**基于图结构**的检索方案。

在此基础上，近年来研究涌现出诸多 GPU 加速优化工作，如 SONG、GANNS、CAGRA 等。这类方案依托 GPU 大规模并行计算能力与高内存带宽，相比纯 CPU 方案实现显著加速，支撑起高吞吐、可扩展的向量检索业务。
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260507100327.png)
尽管高性能向量检索技术已取得长足进展，但现有绝大多数系统均面向**静态或缓慢更新**的数据集设计，难以适配需要实时为高速流入向量构建索引的动态业务场景。如图 1 所示，大规模流式业务场景要求检索、插入、删除等操作均具备低延迟能力。

纯 CPU 检索方法具备实时更新能力，但存在计算开销大、内存带宽受限的问题，随着数据集扩容，可扩展性急剧变差；反观 GPU 检索方案，虽能为静态索引提供超高吞吐，但处理大规模更新时会产生巨大的数据传输开销（最高可达 20 秒），整体检索吞吐甚至降至每秒 500 次查询以下。同时，针对高流速的流式负载，对索引进行全量重建也完全不具备可行性。

CPU 方案**易于更新但性能受限**、GPU 方案**性能强劲但适配静态数据**，二者存在明显割裂，亟需一套融合双方架构优势的统一解决方案。现有部分混合架构方法尝试通过在显存中存储压缩向量来弥补这一差距，但会造成检索精度损失，无法满足高召回业务需求；且依赖静态量化码本，若要适配流式数据更新，还需付出高昂的重新训练成本。

综上，大规模流式近似最近邻搜索主要面临两大核心挑战：

**挑战一：GPU 显存容量受限**
现代向量业务需处理数百万至数十亿级别的高维嵌入向量，数据规模往往超出单卡显存上限，导致性能严重下滑。例如存储 3500 万条 768 维向量至少需要 120GB 存储空间，远超 NVIDIA A100 等高端加速卡约 40GB 的常规显存容量。这种硬件限制迫使数据需频繁在显存与主机内存间迁移，高昂的传输开销严重抵消了 GPU 并行计算的性能优势。虽然向量压缩技术可降低存储占用，但通常会牺牲检索精度，或为维持精度而增加查询延迟。因此亟需设计高效的 CPU-GPU 协同框架，合理管控数据布局与存储位置。
>通过向量压缩（PQ量化）解决GPU显存不足的工作：FlashANNS、FusionANNS等，它们将压缩后的所有向量存储在GPU显存中，但是会导致两个问题：1.直接导致检索精度的下降2.在动态更新场景下，需要PQ码本的重训练，开销极大，因此只适合静态场景

**挑战二：频繁向量更新引发性能衰减**
大规模高频向量更新会带来两大性能难题：其一，现有基于图结构的检索方案在频繁插入、删除场景下更新吞吐能力不足，普遍采用同步删除标记与周期性图重建策略，易引发查询延迟突增、系统响应变慢；在 GPU 架构下，图拓扑结构修改需要频繁内核同步，进一步放大该问题，同时 CPU-GPU 混合架构还会产生额外跨设备数据传输开销。其二，频繁更新会破坏原始图结构，导致检索遍历路径劣化，随时间推移检索精度持续下降。

针对上述难题，本文提出研究问题：能否设计一种混合架构方案，**无需向量压缩**，同时高效支撑高频向量更新？

本文提出 SVFusion，面向大规模流式向量检索的 CPU-GPU 协同框架，实现 GPU 高性能计算与索引实时更新的深度融合。SVFusion 包含三大核心创新点：
1. **分层向量索引结构**：在 CPU 与 GPU 多级内存中分层组织向量数据与图拓扑，最大化计算资源利用率。设计流水线优化的任务协同机制，将高频访问向量的并行检索计算部署在 GPU 层，由 CPU 层负责动态更新与低频访问数据计算，充分发挥两类计算单元各自优势。
> 在 PilotANN、SVFusion、GustANN、FusionANNS、FlashANNS 这五个系统中，仅有 SVFusion 明确设计并实现了对向量数据的冷热划分与分层布局
2. **负载感知型向量布局机制**：提出全新两阶段调度策略，自适应决策向量是否在 CPU 与 GPU 间迁移；若目标向量不在显存中，则直接在 CPU 本地完成计算。该机制结合访问模式与图结构特征做综合评估，在数据局部性与传输开销之间实现更优权衡，保证动态负载下的高吞吐与性能稳定。
3. **并发更新处理策略**：在持续更新场景下同时保障更新吞吐与检索质量。设计轻量级图修复机制，对受删除操作影响严重的图顶点进行局部重建；提出多版本图合并算法，实现后台索引规整与并发插入操作的同步协同，在不牺牲索引新鲜度的前提下最大化更新效率。

依托以上核心技术，SVFusion 在 CPU-GPU 异构架构下实现高性能、可更新的实时向量检索。

本文主要贡献如下：
- 提出新型流式向量检索 CPU-GPU 协同框架 SVFusion，融合分层索引与流水线任务协同，提升异构内存层级下的整体计算效率。
- 设计负载感知的向量布局机制，智能平衡数据缓存与计算调度决策，优化动态负载场景下的检索性能。
- 提出并发更新处理策略，通过轻量级图修复与多版本同步技术，在持续更新过程中维持索引检索质量。
- 完整实现 SVFusion 系统，并在多样化流式负载下开展全面评测。实验结果表明，相较于 FreshDiskANN，本框架检索吞吐最高提升 9.5 倍，插入吞吐最高提升 71.8 倍，同时始终保持更优的召回率表现。

## 2.1 近似最近邻搜索
设集合 X={x1​,x2​,…,xN​}⊂RD 为由 N 个向量构成的数据集，其中每个数据点 xi​∈RD 代表 D 维实数空间中的向量。任意两个向量 u,v∈RD 之间的距离记为 dist(u,v)，本文采用**欧氏距离**（即 L2 范数）。

给定查询向量 q∈RD 与正整数 k (0<k<N)，**精确最近邻搜索（NNS）** 旨在从数据集 X 中找出距离查询 q 最近的 k 个向量构成子集 UNNS​⊂X，满足：对任意 xi​∈UNNS​、任意 xj​∈X∖UNNS​，均有 dist(xi​,q)≤dist(xj​,q)。

精确最近邻搜索需要计算查询向量与全体数据向量的距离，在大规模数据集下计算开销极高、难以落地。为此，现有多数研究转而**近似求解最近邻**，在检索精度与计算效率之间做权衡，即**近似最近邻搜索（ANNS）**。

给定近似误差因子 ε>0，ANNS 算法返回按与查询 q 距离升序排列的 k 个有序向量集合 UANNS​={x1​,x2​,…,xk​}。设 xi∗​ 为查询 q 在数据集 X 中的第 i 个精确最近邻，则算法需满足：对所有 i=1,2,…,k，有

dist(xi​,q)≤(1+ε)⋅dist(xi∗​,q)

近似检索的效果通常采用**召回率**指标进行评估，其定义为：
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260507103636.png)
其中 UNNS​ 表示**k 个最近邻的真实基准集合**。召回率越高，对应近似误差因子 ε 越小，近似程度越高，代表检索精度越好。
## 2.2 流式近似最近邻搜索（SANNS）
流式近似最近邻搜索（SANNS）在传统 ANNS 框架基础上进行扩展，适配**持续动态变化**的数据集，支持一系列序列操作。

设初始向量数据集 X0​={x1​,x2​,…,xN0​​}⊂RD 包含 N0​ 个向量，每个向量拥有唯一标识。

SANNS 随时间处理有序操作序列 O={o1​,o2​,…,oT​}，其中 T 代表流式任务的时间步总数或操作总数。

在每个时间步 t∈[1,T]，对当前数据集状态 Xt−1​ 执行唯一操作 ot​，生成新的数据集状态 Xt​

SANNS 支持以下四类操作：
- **Build(Xinit​)**：Xt​=Xinit​，初始化数据集状态，其中 Xinit​⊂RD。该操作基于给定静态数据集构建初始索引结构，也可用于索引周期性重建。
- **Search(q,k)**：Xt​=Xt−1​。给定查询向量 q 与正整数 k，从当前数据集 Xt−1​ 中检索出近似真实 k 近邻的 k 个向量集合。
- **Insert(xt​)**：Xt​=Xt−1​∪{xt​}，向数据集中新增一个向量。该操作动态扩充可检索向量空间，适配新增数据接入。
- **Delete(xt​)**：Xt​=Xt−1​∖{xt​}，从当前数据集中移除已有向量 xt​。该操作剔除过期或无效向量，缩减检索空间。

传统 ANNS 面向基准固定的静态数据集，与之不同，SANNS 需要在**数据集持续演化、真实近邻基准不断变化**的场景下，维持检索精度，同时保证各类动态操作的执行效率。

为全面评估 SANNS 算法性能，主要采用两类评价指标：

**精度**通过**召回率**量化，将检索结果与当前数据集状态下的真实最近邻做比对，衡量近似检索质量；

**效率**以**吞吐率**评估，定义为每秒可处理的查询与更新操作数量（QPS）。
## 2.3 基于图的索引
我们定义有向图 G=(V,E)，其中 V 为顶点集合，顶点总数为 N；每个顶点 vi​∈V 对应数据集中的一个向量。边集合 E⊆V×V 用于刻画顶点间的关联关系：若两个顶点对应的向量在向量空间中距离相近，则二者之间建立连边。

对任意顶点 vi​∈V，记其**出邻域集合**为

Nout​(vi​)={vj​∈V∣(vi​,vj​)∈E}，

**入邻域集合**为

Nin​(vi​)={vj​∈V∣(vj​,vi​)∈E}。

**基于图的 ANNS 与 SANNS**
基于图的近似最近邻检索通过**贪心遍历近邻图**完成查询处理。给定查询向量 q，检索从一个或多个入口顶点出发，不断向与查询距离更近的邻接节点迭代探索，直至满足停止条件。该方法相较于暴力全量搜索大幅降低检索复杂度，同时能够保持高召回率。

为适配流式检索场景（SANNS），索引必须支持持续的向量插入与删除操作。设时刻 t 的当前近邻图为 Gt​=(Vt​,Et​)，给定待插入集合 It​ 与待删除集合 Dt​⊆Vt​，更新后的图为 Gt+1​=(Vt+1​,Et+1​)，其中顶点集合满足 Vt+1​=(Vt​∪It​)∖Dt​，边集合满足 Et+1​⊆Vt+1​×Vt+1​。

**面向图结构 ANNS 的 GPU 加速**
借助 GPU 可大幅提升近似最近邻检索的性能。实际应用中，基于 GPU 的检索方法通常采用**固定出度的 k 近邻图**，以充分发挥 GPU 大规模并行计算能力；若顶点出度不固定，会造成负载不均衡，导致计算资源利用率低下。

其核心加速思路为：将图结构等关键元数据存放于 GPU 高带宽显存中，并采用**每个查询分配一个线程块**（而非每个查询分配一个线程）的调度方式，实现核心检索操作的大规模并行化。该设计相比纯 CPU 方案可取得显著性能提升，更多细节详见附录 A。
# 三、总览
本文提出 **SVFusion**，面向大规模流式近似最近邻搜索（SANNS）的 CPU-GPU 协同框架。该框架通过以下核心设计，解决实时向量检索所面临的根本性难题：

为突破 GPU 显存容量限制：（1）提出**分层图结构向量索引**，在保证检索性能的同时，于 CPU 内存与 GPU 显存间维护同步的图结构；（2）设计**负载感知的向量放置策略**，合理调度向量在 CPU 与 GPU 间的存储分布，最大限度降低昂贵的 PCIe 数据传输开销。

为进一步提升高频更新场景下的性能：（3）提出**轻量级图修复与多版本合并机制**，通过高效的索引图维护策略，以极小的精度损失实现高吞吐并发更新。

>一、CAGRA介绍
>Hiroyuki Ootomo, Akira Naruse, Corey Nolet, Ray Wang, Tamas Feher, and Yong  Wang. 2024. CAGRA: Highly Parallel Graph Construction and Approximate  Nearest Neighbor Search for GPUs. In IEEE International Conference on Data Engineering (ICDE). 4236–4247.
>- CAGRA的定位，正是“**专为极致GPU并行而生的下一代图索引算法**”。它并非Faiss或Milvus那样的全栈引擎，而是作为NVIDIA cuVS加速库的核心组件，专攻图索引的构建与检索，可供各类引擎直接集成调用。
>- CAGRA摒弃HNSW复杂的多层结构，采用**单层图设计**和**固定出度**，让显存访问和线程调度更规整。检索时采用**全批次并发扩展**，极大提升GPU并行效率。相比CPU方案，其索引构建快**12.3倍**，延迟降低**4.7倍**。
>- **NVIDIA加速库的核心组件**：CAGRA是NVIDIA **cuVS**（前身为RAPIDS RAFT）的核心算法。cuVS库提供IVF-Flat、IVF-PQ等多种算法，CAGRA专攻小批次、高并发搜索场景，三者共同提供了完整的GPU加速方案栈
>- **索引（Index）而非引擎（Engine）**：ANNS引擎是全栈软件系统（实现数据管理、分布式等完整自闭环），而CAGRA只是一个需要被集成的加速算法。它专注于“图索引”加速，能被广泛地集成到各类引擎之中
>- 腾讯微信团队在核心线上推荐业务中大规模应用了CAGRA。他们基于CAGRA构建了高并发、高可用的GPU向量检索服务，并进行了动态准入控制、多流并发优化等底层优化，应用成熟度很高
>
>二、这里的“热点子图”是根据下面的WAVP算法动态形成的，当决定将一个热点向量搬运到GPU的时候会搬运热向量本身以及其邻居ID（不包含原始数据，无论冷热）从而自动形成拓扑关系

**CPU 模块**：在内存中维护完整索引图及多版本副本，存储全部向量数据；负责调度 GPU 计算与多端同步，并通过版本控制实现并发访问。该设计利用系统大容量主存承载海量向量数据集的 SANNS 各类操作，以极低的同步开销实现高吞吐处理。
>需要说明的是，当向量数据集规模超出可用内存容量时，SVFusion 引入了与 FreshDiskANN 类似的磁盘扩展机制，将冷数据及低频访问向量下放到磁盘存储中。

**专用缓存管理器**：负责调度向量数据在 CPU 与 GPU 内存间的放置策略，优化混合负载下的查询性能。区别于传统仅依赖时间局部性的缓存策略，本缓存管理器**同时利用向量访问模式与图拓扑结构**，动态加载与淘汰向量条目，尽可能让图遍历过程中频繁访问的向量常驻 GPU 显存。

三大组件可实现高带宽显存、内存乃至磁盘之间的灵活内存管理，为流式向量操作奠定可扩展基础。系统通过**存储与计算解耦**，支持数据在多级存储间自适应放置与高效访问。这种混合架构既突破了纯 GPU 系统的显存容量限制，又解决了纯 CPU 系统的性能瓶颈，可广泛适用于大规模实时向量检索场景。

**SANNS 工作流程**
图 2b–2e 展示了 SVFusion 端到端的 SANNS 完整流程，依托 CPU-GPU 协同处理，实现构建、检索、插入、删除四类核心操作：

**索引构建（Build）**
采用适配 GPU 可扩展性的批量式索引构建策略。若数据集可完全放入 GPU 显存，SVFusion 沿用 CAGRA 的构建流程，充分发挥 GPU 算力；对于超大规模数据集，则对数据进行划分，为每个分区在GPU中并行构建子图，再在CPU中合并为全局统一索引并保留分区间连通性。该设计在充分利用 GPU 加速能力的同时，可扩展支撑超出显存容量的海量数据集。

**向量检索（Search）**
收到查询请求后，首先在 GPU 端启动图遍历。缓存管理器检查所需向量与子图是否已缓存于 GPU 显存：若已缓存，则全程在 GPU 完成检索；若未缓存，SVFusion 既可按需从 CPU 拉取数据，也可自适应切换至 CPU 端计算向量距离，以此减少传输开销。最终对结果排序得到 Top-k 近邻并返回给用户。
>对于缓存未命中的向量，具体会在CPU中直接计算还是先将其从CPU传输到GPU中计算取决于下面的WAVP算法

**向量插入（Insertion）**
在新增向量的同时保持图结构完整性与检索质量。每条新向量首先在 GPU 端检索候选近邻，必要时按需进行数据传输；随后 CPU 通过启发式重排序优化近邻选择，并插入反向边以维持图的双向连通性。完成一批次插入后，将更新后的子图从内存同步至高带宽显存以保证多端一致性，通过大批量批量传输平摊单次传输开销。同时为保障容错能力与管控存储容量，SVFusion 会定期将图结构与向量数据持久化至磁盘。

**向量删除（Deletion）**
采用**轻量级删除机制**处理大规模向量删除请求，包含两套互补策略：
（1）**惰性删除**：仅将目标向量标记为已删除，不立即修改图拓扑结构；
（2）**异步修复**：通过局部拓扑感知修复与周期性全局规整，恢复图的连通性。
整套删除流程采用**非阻塞设计**，检索与插入操作可并发正常执行、不受阻塞干扰。详细的删除处理机制将在 5.2 节展开阐述。
# 四、设计
大规模动态向量检索场景要求图索引能够承载海量数据集，并适配**持续变化的索引结构与访问模式**。尽管 CPU-GPU 协同计算具备可观的性能提升潜力，但如何高效管理内存分配、调度任务执行、实现动态数据布局，仍然存在诸多挑战。为此，SVFusion 提出分层索引架构、负载自适应缓存机制以及流水线优化协同策略，以最大化系统吞吐性能。
## 4.1 研究动机
**分层内存管理**
在大规模业务场景下，GPU 显存容量十分有限。现有方案多通过在 GPU 端对向量进行压缩以降低内存占用，但这不可避免地会造成检索精度下降。本文的核心观察是：真实业务的向量访问模式具有**访问偏斜性**，部分向量会被高频访问。基于这一特性，本文设计分层内存架构，**将热点向量无压缩缓存至 GPU 显存**，同时兼顾高检索精度与内存利用效率。

**动态缓存管理**
流式近似最近邻检索场景中，访问模式会持续变化，原本的热点向量会逐渐转变为冷数据。传统缓存策略多在计算发起前预取数据，而在 CPU-GPU 协同检索中，向量的访问位置同时决定**计算执行位置**与**是否需要跨设备数据传输**。这一特点催生了**负载感知缓存机制**，可在动态变化的负载下做出最优的向量布局决策。

**混合计算协同调度**
CPU-GPU 协同 ANNS 检索需要在异构计算单元上完成多阶段计算流程。但同步开销大、资源利用率不足等问题，给高效任务协同带来了困难。因此亟需设计流水线执行架构，在最小化同步代价的同时支持多任务并发执行。
## 4.2 基于图结构的分层向量索引
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260507112855.png)
图 3 展示了本文跨 CPU、GPU 内存层级的分层索引结构。现有 GPU 加速 ANNS 方法受限于显存容量，难以支撑超大规模数据集；而本文设计充分利用 CPU 内存与 GPU 显存的互补特性：GPU 显存**带宽极高（1 TB/s）但容量有限**，CPU 内存**容量可达数百 GB 但带宽较低（100 GB/s）**。为弥合带宽与容量之间的矛盾，本文分层架构将向量数据分布在多级存储层级中，并配套协同检索算法，动态调度 CPU 与 GPU 共同参与计算，实现海量数据集下的高吞吐近似最近邻检索。

**分层索引结构**
本索引架构首先依托 CPU-GPU 两级内存，承载超出 GPU 显存容量的数据集；由缓存管理器统一调度，将热点向量**无压缩常驻 GPU 显存**。对于规模超出内存上限的超大型数据集，架构可无缝扩展至磁盘持久化存储，形成三级分层体系。

设数据集总规模为 N：
CPU 内存维护**完整数据集与全局图结构** G=(V,E)，并为每个向量分配主机端唯一标识 h_id∈[1,N]；
GPU 显存仅缓存少量热点子集向量，数量满足 M≪N，并为其分配紧凑的设备端标识 d_id∈[1,M]。
通过在 GPU 常驻热点向量及其对应的局部子图，绝大多数检索计算可直接利用高带宽显存完成，且**无精度损失**。当数据集规模进一步超出系统内存上限时，框架可无缝接入磁盘存储，构建显存 — 内存 — 磁盘三级分层架构，支撑向量规模从百万级扩展至十亿级，并在动态负载下保持性能稳定。

缓存管理器部署在 GPU 端，负责在 SANNS 各类操作中协调多级存储间的数据迁移。其内部维护一张映射表，映射规则为 mapping(h_id)→{d_id∣NONE}：若向量已缓存至 GPU，则返回设备端标识；若向量驻留 CPU 内存，则返回空标识。该设计借助 GPU 高带宽内存实现大规模并行向量查表，规避了 CPU-GPU 频繁同步带来的额外开销。

## 4.3 负载感知向量放置策略
为应对流式负载下动态变化的访问模式，本文提出一种自适应向量放置策略，用于优化向量在 GPU 与 CPU 内存间的数据局部性，并提升近似最近邻检索过程中的计算效率。

该策略摒弃静态分配方式，依据**运行时访问统计信息**对向量进行优先级排序，从而在负载动态变化的场景下维持较高的查询吞吐。

设时刻 t 的完整向量集合为 Xt​={x1​,x2​,…,xN​}。受 GPU 显存容量限制，仅有子集 XGt​⊆Xt​ 常驻 GPU 显存，满足向量数量 M≪N；其余向量均存放于 CPU 内存中。

当访问某一不在 GPU 显存中的向量 x∈/XGt​ 时，CPU-GPU 协同处理面临的核心难题是：**为每次向量访问做出最优放置决策**：
（1）将向量从 CPU 内存迁移至 GPU 显存；
（2）直接在 CPU 端执行距离计算。

若对每一个缓存缺失的向量都执行迁移晋升，会产生巨大的数据传输开销，对于低频访问向量尤为浪费；而若全部依赖 CPU 计算，则无法发挥 GPU 的并行算力与高带宽优势，导致系统性能无法达到最优。

为对每一次未命中 GPU 缓存的向量访问，在上述两种策略间做出最优选择，需要**量化计算收益与数据传输开销之间的权衡关系**。为此本文定义了如下收益函数：

> LRU等传统缓存的核心逻辑是：“最近被访问过的向量更可能再次被访问，把它们留在缓存里”。但如果热点向量的数量很多（50% 的向量都是热点），且热度差异不大，LRU 的淘汰策略会频繁把未来依然会被访问的 “中度热点向量” 挤出去，导致缓存命中率下降，无法达到最优性能。

## 4.4 流水线优化任务协调
SVFusion 通过一套高效的任务协同机制，统筹 CPU 与 GPU 资源上的计算任务，最大化硬件利用率。与以往依赖串行执行模型或仅在 GPU 上进行细粒度优化的工作不同，SVFusion 基于第 4.2 节所述的分层索引结构，引入了流水线化设计，实现了异构资源间细粒度的高效负载分配。为进一步降低同步开销、提升整体吞吐，SVFusion 集成了三项关键协同优化技术：

**面向批次的向量处理**。我们在 CPU 和 GPU 计算中均采用批处理策略以最大化硬件利用率。在邻接节点扩展过程中，不立即处理检索到的向量，而是根据其被分配的计算设备，将向量累积到特定的执行批次中（即算法 1 中的 XCPUᵢ与 XGPUᵢ）。这种延迟执行模型通过在发起距离计算前聚合足够的工作负载，有效摊销了 GPU 内核启动、SIMD 加速 CPU 运算以及跨设备数据传输的开销，从而提升了资源利用率。

**CPU-GPU 并发执行**。SVFusion 通过两项关键技术充分利用 CPU 与 GPU 的异构并行性。首先，它采用基于独立 CUDA 流的异步数据传输，使计算与通信相互重叠，让两个处理器可以并发运行而不互相阻塞。其次，GPU 显存被划分为多个独立分段，每个分段由轻量级自旋锁保护；向量通过其标识符的哈希映射到不同分段，实现了细粒度同步并减少了线程竞争。该设计消除了对全局锁的需求，有效解决了大规模并行 GPU 工作负载中常见的性能瓶颈。

**自适应资源管理**。SVFusion 基于运行时状态，动态平衡 CPU 与 GPU 之间的计算负载。如第 4.3 节所述，我们使用阈值 θ 指导向量放置决策。其核心创新在于自适应阈值调优：SVFusion 持续监控数据集规模、设备利用率和查询延迟等系统指标，在运行时对阈值进行调整。该自适应策略确保了高效的资源利用，缓解了处理器侧的性能瓶颈，并在多样化的工作负载模式与硬件配置下保持稳定的吞吐性能。
# 五、索引更新
## 5.1 向量插入
SVFusion 借助 GPU 加速实现高吞吐向量插入，同时保证索引结构质量。传统基于 CPU 的方案虽采用多线程方式，但受限于并行邻域搜索与距离重排的开销，批处理规模很小，通常仅能处理约 100 个向量。相比之下，本文框架利用 GPU 的大规模并行能力，通过贪心邻域搜索可处理**大得多的批次**（如 6.4 节分析所示，常规批大小可达 10000）。

本文进一步设计了**基于排序的候选边重排策略**：不再依赖开销高昂的距离计算，而是通过候选边的序值来近似表征其重要性。该序值通过统计每条候选边的**迂回路径数**得到，即：借助已选中邻居通过两跳路径绕开当前边的可选路径数量。该方法遍历候选集合 C 仅需 O(∣C∣log∣C∣) 复杂度，大幅降低了计算开销。
>一、迂回路径数解释：当为新向量v从候选邻居中选边时，对于每个候选邻居u：
>假设此时已经选好了一个邻居a，如果存在路径v-a-u，说明不直接连边v-u，也能通过两跳路径到达u，此时，v-a-u就被称为边v-u的一条迂回路径
>二、迂回路径数排序vs传统距离排序：
>传统距离排序的典型案例是diskann中算法2那样的鲁棒裁剪算法，这种情况下，距离的总计算量为O（NR）N为候选邻居的数量，R为出度，而每次距离计算涉及D个维度的计算，因此总浮点计算量为O（NRD）
>对于迂回路径，只需要对N个候选邻居的迂回路径数（都是整数）做一次常规排序，计算量下降为O(NlogN)
>注意：在整个过程中，候选邻居的搜索发生在GPU上，而最终的重排序阶段发生在CPU上

反向边插入阶段，采用细粒度原子操作与线程局部缓冲区处理图的并发更新，相比粗粒度锁机制拥有更好的可扩展性同时保证图结构一致性。向量插入流程的详细步骤见附录算法 5。

## 5.2 向量删除
删除操作负责从索引中移除顶点，同时**保证索引一致性与系统可用**。为此，SVFusion 将删除流程解耦为三个协同步骤：
(1) 逻辑删除：将顶点标记为失效状态，不干扰正在执行的查询；
(2) 局部拓扑感知修复：以极低开销快速恢复核心图连通性；
(3) 周期性全局合并清理：彻底清理已删除顶点，同时维持索引质量。
### 5.2.1 逻辑删除
SVFusion 采用**轻量惰性删除策略**，高效支持高频删除操作，且不影响系统性能。SVFusion 不会立即修改图拓扑结构，而是通过**位集合**对待删除顶点做标记，实现逻辑层面的移除。后续的插入、删除与查询遍历过程中，会自动跳过这些被标记的失效顶点。通过延后开销高昂的结构修复与邻居更新操作，该策略最大限度降低删除延迟，同时规避同步瓶颈。

### 5.2.2 受影响顶点修复
逻辑删除虽能以极低延迟实现向量移除，但随着被删顶点不断累积，会逐步破坏图的结构完整性，导致检索性能下降。
现有删除处理方案大多面向导航图设计，删除带来的影响通常只集中在**高度数枢纽节点**上。这类方法会将修复操作延后，直到删除占比超过全局阈值（如 20%）后，再执行开销极大的全局图合并重构。
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260507162223.png)
但 kNN 图的结构特性与该设计假设存在本质差异：导航图度数分布不均衡，而 kNN 图严格遵循**均匀度数约束**，删除带来的影响会均匀扩散到所有顶点，而非集中在少数枢纽节点。
为量化这种结构差异，本文通过实验对比了两类图的删除影响分布。如图 5 所示，删除 10% 顶点后，kNN 图的邻居缺失比例显著更高：
邻居缺失率在 10%–40% 的受影响顶点占比，是导航图的**3.26 倍**；
邻居缺失率超过 40% 的受影响顶点占比，是导航图的**2.04 倍**。
因此，在达到全局修复阈值前，kNN 图的连通性就已大范围恶化，严重损害检索质量。这就要求随着删除累积，必须针对关键顶点做**持续增量修复**。

**局部拓扑感知修复**
SVFusion 采用局部拓扑感知修复策略，解决图结构快速退化问题。该策略不会立刻清理已删除顶点，而是利用删除位集合快速识别**重度受影响顶点集 VL​**—— 即邻居删除占比超过 50% 的顶点，仅对这类关键顶点执行轻量修复。

对任意顶点 v∈VL​，传统重构会为每个已失效邻居 p，把 p 的所有出邻域都与 v 重连，开销极高；本文轻量修复仅从 p 的出邻域中**最多选取 c 个顶点**与 v 建立连接。常数 c 取值很小（本文设 c=8），新增边复杂度仅为 O(cR)，远低于全局重构的 O(R2)（R 为图固定度数）。

按距离顺序完成选边添加后，采用**宽松过滤条件的鲁棒剪枝**，将 v 的邻居数量约束在度数上限 R 以内。这种增量修复方式，无需对所有受影响顶点做高代价修复，以极小开销维持了图索引质量。

**全局合并清理**
当删除占比超过预设阈值（如 20%）时，SVFusion 启动全局合并：从已删除顶点的出邻域中聚合候选，统一重构所有受影响的邻域结构。该合并过程在后台异步执行，不中断前台的插入与检索业务。

为与前台操作做到数据同步，合并基于**图快照**执行，并通过图版本合并机制，协调不同版本图之间的数据一致性。
## 5.3 同步更新
为最大化系统吞吐，SVFusion 采用**多版本设计**：后台图重构基于快照运行，前台插入操作则在**活跃主图**上持续进行。这种解耦方式实现了更新与重构并发执行，避免耗时的全局重构过程阻塞前台业务。
该多版本机制的工作逻辑为：每当启动后台任务（如全局合并重构）时，就新建一个图版本，将当前图复制为**只读快照**供后台任务处理；任务完成后，再把重构结果合并回活跃主图。后台合并会衍生出新的版本以支持并发更新，可能触发后续更多合并操作。为防止内存无限制增长，系统限制最大并发版本数，达到阈值时暂缓新的更新请求。
但若缺乏合理同步机制，版本合并容易引发数据不一致。为此本文提出一种**轻量两阶段合并算法**，在保证高更新吞吐的同时维持数据一致性。

**增量子图追加**
在时刻 t0​ 基于快照 Gt0​ 启动全局重构时，前台持续执行插入，到时刻 t1​ 已将活跃主图更新为 Gt1​。算法先筛选出新增顶点集合 Vnew​=V(Gt1​)∖V(Gt0​)，并将对应的新增子图 `Gnew​=Gt1​[Vnew​]`  **直接追加**到重构后的图 Gt0′​ 中。
>因为这些节点和边都是全新的，和后台正在改造的老图完全不冲突。所以系统直接把 $G_{new}$ 整个“粘”到后台重构好的 $G_{t_0}'$ 上。
>局限性：Gnew中的边结构全是新节点指向老节点的，没有反向的，因此需要第二阶段的合并

**反向边整合**
在时间区间 [t0​,t1​] 内，新插入顶点也会与 Gt0​ 中的原有顶点建立入边连接。这类反向边对维持图连通性至关重要，若仅简单追加新增子图，会丢失这部分关联关系。
为保留这类连接，系统维护一条**边更新日志 L**，记录三元组 (v,vnew​,d)：
其中 v∈V(Gt0​) 为原有顶点，vnew​∈Vnew​ 为新增顶点，d=dist(v,vnew​) 为二者向量距离。
对所有存在日志记录反向边的原有顶点，将这些新连接并入其邻域列表；若因此超出顶点度数约束，则采用 **α-RNG 剪枝**，基于已预先计算的距离筛选价值最高的边，在控制边数量的同时保障图的连通性。

在图 (a) 中，顶点 6 和 7 在时刻 T1 被执行逻辑删除。在后台压缩阶段（图 b），系统对已删除顶点相关的连接进行修复。基于图 (a) 的快照，顶点 8 和 9 在时刻 T2 完成插入，如图 (c) 所示。为实现图 (b) 与图 (c) 两个版本的同步，增量合并阶段（图 d）追加新增子图，并整合新增顶点到原有顶点之间的反向边。最后通过边剪枝（图 e）剔除两条冗余连接，在满足度数约束的同时保留图的连通性。

## 6.1 实验设置
**硬件配置**
实验基于一台 Ubuntu 18.04.6 LTS 服务器，使用 16 线程进行测试；配置 Intel Xeon Gold 5218 2.30GHz CPU、共 64 个 CPU 核心、376GB DDR4 主存，以及 40GB 显存的 NVIDIA A100 GPU。
**数据集**
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260507170001.png)
本文在三个特征各异的主流真实数据集上评测 SVFusion：Wikipedia、MSMARCO、MSTuring。如表 2 所示，这些数据集维度与内容各不相同，已被大量现有研究用作近似最近邻检索（ANNS）系统的基准测试集。查询集从各数据集中随机采样，真值通过对全量数据集做穷尽线性扫描计算得到。

**工作负载**
实验覆盖多种流式负载模式，每种负载由向量集合与 2.2 节定义的操作序列构成。负载差异主要体现在两点：（1）插入与删除的时间间隔；（2）单步删除向量间的空间相关性。为保证评测可对比性，本文沿用 2023 Big ANN Challenge 中的三种典型负载模式：
- **滑动窗口（SlidingWindow）**
    将数据集随机划分为 Tmax​=200 个等大小分段。从第 1 步到第Tmax​步，每步插入一个分段；从第 Tmax​/2+1 步开始，删除 Tmax​/2 步之前插入的向量。在索引规模趋于稳定的第 Tmax​/2+1 步起开始评测检索性能。该负载模拟向量索引在固定时间窗口内仅保留近期数据的场景。
    
- **过期淘汰（ExpirationTime）**
    将数据集按生命周期分为三类：短期（10 步）、长期（50 步）、永久（100 步），数量比例为 10:2:1。每步插入数据集的 1/Tmax​ 比例，随机分配生命周期类别，并删除已过期向量。该负载用于评测数据生命周期异构场景下的索引性能。
    
- **聚类删除（Clustered）**
    通过 5 轮 k-means 聚类将数据集划分为 64 个簇。每轮交替执行插入与删除阶段：先依次从每个簇按随机比例插入向量，再从每个簇的活跃点中按随机比例删除部分向量。由于空间邻近向量被同时插入和删除，该负载会造成活跃向量分布发生剧烈偏移。

**实现方式与对比基线**
本文基于 cuVS 24.02 库，参照图 2 架构扩展实现 SVFusion；具体实现细节与附录代码已开源至 GitHub 仓库。本文选取以下方法作为对比基线：
- **HNSW**：主流基于图的近似最近邻检索算法，工业界与学术界应用广泛。参数设置：M=48，构建与搜索阶段 ef_construction=ef_search=128。
- **FreshDiskANN**：当前支持内存 / SSD 动态更新的图结构 ANNS 最优方法。沿用原文配置：向量插入阶段 R=64,lb=ls=128,α=1.2，检索阶段 ld=128。
- **CAGRA**：NVIDIA 面向 GPU 设计的高性能 ANNS 算法，性能优异，但处理大规模数据集时受限于 GPU 显存容量。
- **PilotANN**：近期提出的混合架构，可有效为 CPU 型 ANNS 引入 GPU 加速，但**不支持动态更新**。

现有 GPU 检索方法要么仅支持静态建索引，要么动态更新能力受 GPU 显存约束，且缺少开源实现。为保证流式负载下对比公平，本文将 FreshDiskANN 与 HNSW 改造为 GPU 加速版本，把距离计算卸载到 GPU 执行。

**评测指标**
SVFusion 的设计目标是在保证高检索精度的同时实现高效向量更新。因此实验从三大维度展开：更新性能、检索性能、内存效率。
- **更新性能**：衡量更新吞吐，即流式负载下每秒处理的插入 / 删除向量数量。
- **检索性能**：同时评估检索效率与精度；效率用检索吞吐衡量，精度采用 Recall@k 衡量，默认 k=10。
- **内存效率**：通过缓存缺失率评估 GPU 缓存有效性；缓存缺失率定义为：ANNS 检索过程中需要从 CPU 内存而非 GPU 缓存读取向量的访问占比。缺失率越低，高频访问向量的缓存驻留效果越好。

在所有流式负载下，SVFusion 均能保持优异检索质量，Recall@10 稳定在 91%–96%，与基线持平甚至更优。相比 FreshDiskANN，SVFusion 召回率平均高出 0.4%–3.4%。

**吞吐性能分析**
图 7 表明，SVFusion 的检索与插入吞吐均为最优。相较 FreshDiskANN 和 HNSW，SVFusion 平均**检索吞吐提升 20.9 倍**、**插入吞吐提升 3.5 倍**。
一个关键现象：当数据集超出 GPU 显存容量时，GPU 加速版基线（FreshDiskANN (GPU)、HNSW (GPU)）反而比纯 CPU 版本**慢 5.4–7.2 倍**。原因是此时 CPU 与 GPU 之间频繁的数据传输开销，抵消了 GPU 计算带来的加速收益。
而 SVFusion 通过**自适应缓存策略**规避了这一问题。
此外，对于从空数据集开始的负载（过期淘汰、聚类删除），初始阶段 GPU 显存未被占满、缓存缺失率低于 1%，检索吞吐可达到 40K 峰值。

**缓存缺失率分析**
为刻画内存访问特征，本文分别统计了插入与检索阶段的 GPU 缓存缺失率。尽管 GPU 显存仅能容纳整体数据集的一小部分，但在整个动态负载运行过程中，SVFusion 始终维持**低且稳定**的缓存缺失率，证明其缓存放置策略能精准捕捉时间局部性访问特征，即便在频繁更新场景下也可避免不必要的缓存淘汰。

同时，**插入阶段缓存缺失率平均比检索阶段低 19.6%**。主要原因由负载构成决定：实验中检索查询仅占总操作量的 15%–25%，缓存管理器更容易拟合和命中插入操作的访问规律。

**删除性能分析**
如表 3 所示，所有方法的**惰性删除（墓碑标记）** 耗时相差不大。
在图修复效率上，凭借**局部拓扑感知修复策略**，SVFusion 大幅优于 FreshDiskANN。SVFusion 不依赖开销极高的周期性全图重构，而是以增量方式逐步修复连通性缺陷；这种持续维护机制既防止了图结构持续退化，又降低了全局合并的复杂度与执行频次，最终实现更快的删除处理效率。

这证明本文 CPU-GPU 协同检索方案，**无需数据压缩**即可高效支撑大规模向量检索场景。

## 七、相关工作
**向量索引**
近似最近邻检索（ANNS）是经典研究问题，已有大量相关工作。
**基于划分的方法**利用树结构、量化技术将向量按空间划分组织，但易受**维度灾难**影响；
**基于哈希的方法**将高维向量投影到低维哈希桶中，以加速相似度检索；
**基于图的方法**构建近邻图：节点代表向量、边连接相似向量，检索时可高效遍历跳转。
但现有多数算法仅面向**静态数据集**，在频繁更新场景下性能表现很差。

**流式向量检索**
近年研究开始聚焦向量检索系统的动态更新能力，以适配流式数据场景。
FreshDiskANN 将新向量先驻留内存，再定期合并到磁盘常驻图索引中；
SPFresh 采用聚类架构，并引入**轻量增量重均衡（LIRE）** 机制，在数据分布发生变化时动态调整划分结构；
还有其他研究侧重在流式更新中维持图连通性或做并行加速。
尽管已有诸多进展，现有系统仍存在明显局限：要么支持数据集规模有限、仅能处理单一插入操作，要么在高维场景下检索精度明显下降。

**GPU 加速的近似最近邻检索**
为满足大规模业务的算力需求，利用 GPU 加速向量检索已被广泛研究。
Faiss、RAFT 等早期工作率先将向量检索从 CPU 迁移至 GPU；
SONG 重新设计图检索流程以更好适配 GPU 并行特性，GANNS、CAGRA 则通过定制核函数进一步优化 GPU 检索算法。
在系统架构优化方面，已有研究采用**CPU-GPU 异构调度**、CPU/GPU 分层数据存储、定期同步等设计思路。
但这类方案大多面向**静态数据集**，检索阶段索引固定不变。
近期的 RTAMS-GANNS 虽支持 GPU 端实时插入，却无法扩展到大尺度数据集。
因此，**在 GPU 上实现高效的动态索引更新**，尤其是大规模图类索引的动态维护，仍是尚未解决的难题。
# 八、总结
本文提出了 **SVFusion**，一种面向大规模向量检索、支持实时更新的 **CPU-GPU 协同框架**。该框架设计了跨多层存储的**分层向量索引**，实现异构硬件间的数据高效分布。同时引入**自适应缓存机制**，结合访问模式与图结构关联关系优化数据放置策略。SVFusion 还采用**轻量级删除策略**，支持各类操作无阻塞并发执行，适配动态负载场景。

实验结果表明，在多种流式负载下，相较于现有主流方法，SVFusion 在保持高召回率的同时，吞吐性能最高可提升 **72 倍**。

CAGRA 的一项核心优化为**条件式距离计算**：仅对从未评估过的节点计算距离，避免迭代过程中的重复计算。当候选池状态趋于稳定（连续多轮迭代未发现更优候选节点）时算法收敛，最终返回 Top-k 近邻结果（第 10 行）。
## 9.2 执行细节
我们通过**扩展 cuVS 24.12 库 [2]** 实现了 SVFusion，为流式向量检索新增了 CPU-GPU 协同机制。具体而言：

（1）我们对 CAGRA [41] 代码库进行了**数万行代码的深度重构**，使其支持大规模图向量索引的动态更新与 CPU-GPU 协同处理，并通过基于位图（bitset）的过滤机制实现高效的向量插入与删除操作。
（2）我们开发了一套**缓存管理器**，维护 CPU 内存与 GPU 内存之间的双向映射关系，以实现高效的数据传输。
（3）我们设计了**多分区策略**，将缓存空间划分为多个独立区域，每个区域采用独立锁进行管理，以最小化资源竞争并最大化系统吞吐量。该分区方案支持多线程并发执行缓存操作，同时保证数据一致性。
