LAANN: I/O-Aware Look-Ahead Search for Disk-Based Approximate Nearest Neighbor Search
# 摘要
近似最近邻搜索（ANNS）是大规模信息检索、智能推荐以及人工智能系统中的基础核心技术。随着向量数据集规模增长至数十亿乃至数万亿级别，磁盘式近似最近邻搜索系统应运而生，该类方案将向量数据与索引结构存放于存储设备中以适配超大存储体量场景，但其查询性能依旧受磁盘 I/O 延迟严重制约。
现有磁盘式近似最近邻搜索方案大多仅从优化 I/O 执行效率、实现计算与 I/O 并行重叠两个方向入手优化，普遍将中央处理器运算任务与磁盘读写访问相互割裂开来。这种分离式设计错失了一项关键优化思路：在发起磁盘读写请求前，优先对已缓存至内存中的候选向量进行筛选处理，以此减少无效磁盘访问，同时提升检索效果。
但落地该优化思路存在诸多难点：过量的前置计算会挤占关键 I/O 操作的执行时机，造成读写任务滞后；若前置计算策略选择不当，不仅无法带来性能收益，反而会拉高整体查询延迟。

本文提出**LAANN**—— 一款磁盘式近似最近邻搜索系统，通过对 CPU 运算与磁盘 I/O 访问进行协同联合优化，让图检索流程充分感知硬件 I/O 行为特征。LAANN 融合三大核心优化技术：

其一为**前瞻搜索机制**，在检索不同阶段动态调整搜索策略，兼顾削减 I/O 次数与及时下发读写请求两大目标；
其二为**优先级 I/O-CPU 流水线**，利用 I/O 空闲等待时段，依据候选向量对后续 I/O 决策的影响权重，优先处理内存中已缓存的候选数据；
其三为**轻量化快速内存图索引**，能够输出高质量检索初始候选样本，加快检索收敛速度，进一步减少磁盘访问次数。

基于千万级与十亿级向量数据集开展的大量实验结果表明，LAANN 性能大幅超越当前业界主流磁盘式近似最近邻搜索方案。在召回率 Recall@10 取值为 0.9 的实验条件下，LAANN 检索吞吐量提升 1.41 至 4.66 倍，查询延迟降低 29% 至 79%，磁盘 I/O 操作次数减少 1.59 至 6.34 倍。
# 一、引言
近似最近邻搜索（ANNS）能够从数百万乃至数十亿规模的向量数据集中，筛选出与查询向量相似度最高的样本，是机器学习、检索增强生成、生物信息学等各类系统中的核心基础组件。在众多近似最近邻搜索技术中，基于图结构的检索方案在检索速度与召回精度之间实现了最优平衡，其中 HNSW、Vamana 等主流内存索引算法可在亚毫秒级延迟下实现高召回率检索。

但当向量数据集规模拓展至数十亿甚至数万亿级别时，此类算法暴露出严重的可扩展性瓶颈 —— 数据集体量与图索引规模会随向量数量同步增长，无法全部载入内存运行。磁盘式近似最近邻搜索方案应运而生，该类方案将向量数据与图边信息存储至固态硬盘，把内存占用压缩至数据集总规模的极小比例，代价则是引入了极高的磁盘 I/O 延迟。
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260708111825.png)
现有磁盘式近似最近邻搜索算法已通过多种优化手段降低 I/O 延迟，包括优化磁盘数据布局、调整 I/O 调度策略、优化内存检索入口节点、将高频访问节点缓存至内存等。但如图 1 (a) 所示，即便叠加所有优化策略，磁盘 I/O 延迟依旧占据整体查询延迟的九成以上。不仅如此，随着并发线程数不断增加，磁盘 I/O 资源争用问题愈发严重，PipeANN 算法表现尤为明显：该算法为最大化 I/O 并行重叠效果会主动发起更多读写请求，进一步加剧资源争抢，在高并发场景下检索延迟大幅飙升。

与此同时，由图 1 (b) 数据可知，即便是综合性能最优的基线算法，CPU 仍有超 74.3% 的查询处理时间处于空闲状态。这一现象表明，现有检索系统将 CPU 运算与磁盘 I/O 访问相互割裂，造成了大量性能资源闲置。传统优化思路仅围绕 I/O 本身展开，要么降低读放大系数，要么实现读写操作并行执行，却忽略了利用闲置轻量 CPU 运算来削减整体 I/O 请求数量这一重要优化方向。

本文核心研究发现：CPU 运算与磁盘 I/O 操作存在强关联性。在下发新一轮磁盘读写请求前，提前对已缓存至内存的向量数据进行预处理，能够有效减少磁盘访问总次数。具体而言，优先拓展当前相似度排名并不靠前的候选向量，依旧可以推进检索边界，让检索流程以更少的 I/O 次数定位到真实近邻向量。此外，磁盘图检索过程中仅依靠近似距离值做判断，部分排名靠后的候选向量经过精确距离计算后，往往是目标真实近邻。对这类候选向量进行前置处理，既能提升检索精度，也能缩短达成目标召回率所需的磁盘访问次数。

但单纯增加内存候选向量的预处理操作，会形成无法规避的性能取舍关系：一方面，额外的 CPU 运算能够优化检索决策，减少后续磁盘读写行为；另一方面，过量的计算任务会加重 CPU 负载，延误新 I/O 请求的下发时机，最终拉高整条查询链路的处理延迟。反之，若预处理候选向量数量过少，便无法充分挖掘缩减 I/O 次数的优化潜力。因此，磁盘图检索场景下的核心难点，在于合理平衡 CPU 运算开销、I/O 削减效果与 I/O 请求及时下发三者之间的关系。

利用 CPU 空闲等待 I/O 完成的空档期开展额外运算，是调和上述矛盾的可行思路，但实际落地仍存在两大难题。第一，内存中缓存的各类候选向量优化价值差异极大：部分候选向量直接决定下一轮 I/O 调度策略，必须在发起新磁盘请求前优先处理；部分候选向量延后处理也不会阻碍检索推进；还有一部分候选向量几乎无法对检索流程起到正向作用。这就要求系统不仅要把控额外运算的整体工作量，还要明确不同候选向量的处理优先级。第二，单次 I/O 等待时段内，具备优化价值的可处理缓存候选向量数量有限，完成这批数据运算后 CPU 会再次进入空闲状态，大量 I/O 空闲时段依旧无法得到充分利用。

针对以上现存难题，本文提出**LAANN**磁盘式近似最近邻搜索系统，让图检索流程深度贴合 I/O 硬件运行特性。该系统搭载前瞻搜索机制，依据检索所处阶段动态调整运行策略，化解缩减 I/O 次数与及时下发读写请求之间的矛盾；设计优先级 I/O 与 CPU 协同流水线，依托带溢出缓冲区的专属候选池，按照候选向量对后续 I/O 决策的影响程度排布运算顺序，充分填满 CPU 空闲等待时段；同时发现现有内存图索引与磁盘图检索存在精度不匹配问题，极易导致检索初期候选池数据匮乏，大幅增加检索前期的无效 I/O 开销，为此本文设计一款轻量化高速内存图索引，从检索起始阶段就为磁盘图检索提供高质量候选向量，减少检索趋近阶段的磁盘访问次数。以上三项核心技术通过打通 CPU 运算与磁盘 I/O 操作的内在关联，从根源上破解 I/O 性能瓶颈。

本文主要研究贡献如下：
1. 明确 CPU 运算与磁盘 I/O 操作之间的强耦合关系，划分检索趋近阶段与收敛阶段，剖析两个阶段截然不同的 I/O 运行特征；
2. 提出前瞻搜索策略、搭载溢出候选池的优先级 I/O-CPU 协同流水线，以及轻量化高速内存图索引三大核心技术；
3. 基于亿级、十亿级两类规模共五组标准数据集，与五种主流顶尖磁盘式检索算法完成全面对比测试，实验结果证明，LAANN 检索吞吐量提升 1.41 至 4.66 倍，查询延迟降低 29% 至 79%，磁盘 I/O 操作次数减少 1.59 至 6.34 倍。

本文其余章节内容安排如下：第二章介绍图结构检索与磁盘式近似最近邻搜索相关基础理论；第三章深入剖析 I/O 性能瓶颈，阐明 CPU 运算与 I/O 操作的耦合关系，梳理利用该耦合特性开展优化所面临的各类难题；第四章详细阐述 LAANN 整体设计方案；第五章说明系统具体实现细节；第六章将 LAANN 与五种主流顶尖磁盘式检索算法开展全面性能评测；第七章梳理国内外相关研究现状；第八章对全文研究内容进行总结。
# 二、研究背景
## 2.2 基于磁盘的图ANNS
磁盘式近似最近邻搜索（ANNS）方法将向量数据与图连接关系卸载至固态硬盘（SSD）存储，并按需从磁盘读取数据，同时仅在内存中保留向量压缩表示。与内存图检索相比，磁盘式检索在四个核心维度存在显著差异：

第一，磁盘式检索每轮并非仅拓展单个距离最近的未访问向量，而是最多选取**W 个（光束宽度）** 距离最近的未访问向量开展拓展；
第二，针对每个选中的向量，系统会并行下发 I/O 请求，从 SSD 读取其原始向量值与邻居节点 ID；
第三，检索过程不计算精确距离，而是利用内存中已有的向量压缩表示，估算邻居节点与查询向量之间的近似距离，避免为获取邻居节点原始值产生额外 I/O 开销；
第四，检索收敛后，检索路径上所有已访问向量的全精度数值均已通过前期 I/O 读取缓存至内存，因此系统会基于全精度距离完成最终重排序，并返回前 k 个最优结果。
但该架构也带来核心挑战 ——**I/O 延迟过高**。每一轮图遍历都需从 SSD 读取 W 个数据页，而图检索的随机访问特性，导致难以对读取操作进行批量处理或预取优化。现有主流方案从四个互补方向入手，攻克这一难题：

**磁盘布局优化**
Starling 与 MARGO 通过最大化单个数据页内节点间的互邻接关系，优化存储布局；PageANN 则构建页级节点图，将图节点粒度与 SSD 数据页大小对齐，彻底消除 I/O 读放大问题。
**I/O 调度优化**
PipeANN 重新设计检索算法，为多组异步 I/O 请求构建流水线，通过重叠请求执行时长，降低单步检索的有效 I/O 开销。但相较于标准光束检索，流水线调度会产生非最优 I/O 读取，导致整体 I/O 操作次数增加。
**高频访问节点缓存**
磁盘式 ANNS 内存占用低，可预留空闲内存空间用于缓存优化。DiskANN、Starling、MARGO 及 PageANN 均基于数据集样本数据识别高频访问节点，并将其缓存至内存，直接减少 I/O 操作次数。
**内存检索入口优化**
Starling、MARGO 与 PipeANN 会基于部分向量子集构建内存图索引，减少定位近邻节点所在数据区域所需的磁盘读取次数。
# 三、研究动机
尽管上述各类优化方案已取得一定进展，但现有方法依旧存在严重的 I/O 延迟问题。如图 1 (a) 所示，在内存配额设定为数据集规模 0.4 倍的实验条件下，DiskANN 对高频访问节点进行缓存优化，PipeANN 依托内存图优化检索入口，Starling 则同时采用两种优化手段；即便如此，在不同并发线程数下，所有方法的 I/O 延迟占总查询延迟的比重依旧超过九成。

与此同时，由图 1 (b) 可见，现有各类算法中，CPU 执行有效运算的耗时仅占总查询耗时的 4.7% 至 25.7%。以上实验数据充分说明，即便叠加多项优化策略，系统依旧存在严重的 I/O 与 CPU 资源调度失衡问题：磁盘 I/O 耗时占据查询耗时的绝对主体，而绝大部分查询处理时段内 CPU 都处于空闲待命状态。

事实上，CPU 运算工作量与磁盘 I/O 操作次数之间存在紧密的关联关系。如第二章所述，现有磁盘式检索方法都会利用富余内存空间缓存高频访问节点。倘若在每一轮下达磁盘读写指令之前，提前对已存入内存的向量数据进行运算处理，便有望从以下两方面减少后续整体磁盘 I/O 操作次数。
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260708112656.png)
第一，图检索流程中，从检索入口到近邻向量聚集区域往往存在多条遍历路径。提前拓展已驻留内存、但并非当前距离最近的未访问向量，依旧能够向前推进检索边界，减少定位前 k 近邻区域所需的磁盘读取次数。图 3 直观展示了该优化逻辑：在第二轮检索中，系统不再为距离最近的未访问节点 5 发起磁盘读取请求，转而优先拓展已缓存至内存的节点 7。

对节点 7 完成拓展后，节点 8 被纳入候选池，且节点 8 与查询向量的相似度高于节点 5。最终节点 5 被检索流程舍弃，系统在尚未为其发起 I/O 请求时就将其淘汰。后续检索流程依次遍历节点 8 与节点 11，无需额外读取磁盘数据，即可持续推动检索边界向目标区域靠近。

在此过程中，新发现的高相似度候选向量会逐步替换候选池内关联性较低的向量，使得节点 5、节点 3 这类低价值节点在触发磁盘读取操作前就被剔除，从根源上规避无法对最终检索结果产生增益的无效磁盘访问行为。

第二，磁盘式图检索依靠向量压缩数据计算得出的**近似距离**对候选向量排序，而非精准真实距离，这就导致候选队列的排序结果存在偏差。部分在候选池中排名靠后的向量，经过精确距离计算后，实则与查询向量相似度更高。

传统贪心光束检索往往会在拓展这类向量之前就将其剔除，进而错失通往真实近邻节点的有效检索路径。而提前处理这类已加载至内存、但排序靠后的候选向量，能够重新挖掘被遗漏的有效检索路径，既可以提升最终检索精度，也能减少达成既定召回率指标所需的磁盘访问次数。

如图 4 所示，在发起下一轮磁盘读取请求前，筛选出检索候选池中所有已加载至内存且尚未遍历的候选向量，按照与查询向量的近似距离排序，选取排名前 X% 的向量优先拓展。实验结果表明，随着选取比例 X 不断增大，两组数据集上的整体磁盘 I/O 操作次数均呈现稳步下降趋势。该实验结果印证了本文思路：对已有缓存候选向量开展额外 CPU 运算处理，能够优化检索行进方向，有效减少后续磁盘访问行为。
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260708112922.png)
**现存核心难题：平衡 CPU 运算开销与检索 I/O 推进效率**
虽然处理更多内存候选向量能够减少磁盘访问次数，但该优化方式存在无法规避的性能取舍矛盾。在每一轮检索中增加候选向量处理量，会直接加大 CPU 运算负担，还有可能延误候选池中高优先级向量的 I/O 请求下发时机。

由图 4 变化趋势可知，适度增加内存候选向量处理量会降低查询延迟，但当处理比例超过临界值后，查询延迟不再下降甚至开始回升。造成这一现象的原因主要有两点：其一，持续处理关联性越来越弱的候选向量，所能带来的 I/O 缩减边际效益持续递减，但 CPU 运算开销却在不断攀升；其二，延误高优先级候选向量的磁盘读取请求，会直接拉长整体查询耗时，因为检索流程必须获取这类向量的全精度数据后才能完成最终检索任务。
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260708113001.png)
表 1 实验数据也进一步佐证该结论：缩小光束宽度 W 虽然能够减少总 I/O 次数，但会拖慢 I/O 请求下发节奏，最终使得整体查询延迟远大于采用大光束宽度的检索方案。

因此当前亟待解决的核心问题是：依托拓展候选向量处理来削减磁盘 I/O 次数的同时，严格控制 CPU 运算开销，并且保障高优先级候选向量的 I/O 请求能够及时下发执行。

最为直观可行的优化思路，是将这类额外的 CPU 运算任务安排在 CPU 空闲等待磁盘 I/O 完成的时段执行，充分利用图 1 (b) 中大量闲置的 CPU 算力资源。但想要高效利用这段空闲时段，还需要攻克两大现实难点：

第一，并非所有内存候选向量都适合延后处理。部分候选向量完成拓展后，其衍生出的邻居节点会成为候选池中距离最近的未访问节点，直接决定下一轮磁盘 I/O 调度策略。倘若等到 I/O 请求下发完毕后再处理这类关键候选向量，系统就只能依据滞后的候选池状态制定读写决策，极易产生不必要的磁盘读取行为，大幅降低 I/O 整体利用效率。（解释：例如根据旧候选池判断应该先扩展page5，这时候已经把page5的IO发出去了，同时在扩展内存中的v7的时候，会发现page8更值得读，但page5已经发出去了，即使发现page8更好，也已经浪费了一次IO）

第二，单次 I/O 等待时段内，可执行的有效 CPU 运算任务总量十分有限。为了控制查询延迟，近似最近邻检索系统会维持规模较小的候选池，大量已加载至内存的候选向量还未被处理就会被提前淘汰。这就导致在大部分 I/O 空闲时段内，CPU 依旧处于无事可做的闲置状态，既无法充分挖掘削减 I/O 次数的优化空间，也难以通过额外运算进一步提升检索精度。
# 四、LAANN整体设计
本章详细介绍本文所提 LAANN 方案，该方案通过协同优化 CPU 计算与磁盘 I/O 访问，让图检索流程具备完备的 I/O 感知能力。
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260708115037.png)
## 4.1 系统整体架构
LAANN 依托三项互补核心技术破解 I/O 性能瓶颈：
第一，前瞻搜索机制能够依据检索所处阶段自适应调整执行策略，在削减 I/O 次数与及时下发 I/O 请求之间实现平衡；
第二，优先级 I/O-CPU 协同流水线充分利用 I/O 等待过程中的 CPU 空闲时段，既降低 CPU 运算开销、减少磁盘 I/O 次数，同时提升检索精度；
第三，轻量化高速内存图索引可在检索初期输入优质候选向量，为磁盘图检索打好基础，进一步减少磁盘访问次数。
为支撑上述机制运行，LAANN 按照既定内存配额划分出三类内存存储区域，具体结构如图 5 所示：用于近似距离计算的压缩向量存储空间、存储各页节点质心向量之间边连接关系的轻量化内存图索引，以及存放高频访问磁盘图节点的磁盘页缓存区，每一个缓存页均对应一块固态硬盘物理页。

整体检索流程分为三个步骤：
1. 轻量化内存图索引全程在内存内完成遍历，为磁盘图检索的候选池注入高质量初始候选向量（详见 4.4 节）；
2. 启动前瞻搜索机制并根据检索阶段切换运行策略：在检索趋近阶段，通过持久性检测机制交替启用内存优先模式与常规检索模式；在检索收敛阶段，采用动态光束宽度策略，尽可能避免最终候选池内向量的 I/O 请求出现下发延迟（详见 4.2 节）；
3. 依托带有溢出缓冲区的定制化候选池搭建优先级 I/O-CPU 协同流水线，按照任务对后续 I/O 决策的影响优先级排布运算顺序，填满 CPU 空闲等待时段（详见 4.3 节）。
## 4.2 前瞻搜索机制
结合第三章分析可知，单纯无差别批量处理内存驻留向量会产生两类相互制约的负面开销，直接抵消缩减 I/O 次数带来的性能收益：一是每轮检索额外增加 CPU 运算负载，二是延误最终候选池内前列向量的 I/O 请求下发时机，进而拉长整体查询延迟。想要解决该问题，首先需要厘清磁盘图检索中各类 I/O 请求的本质构成。

磁盘图检索产生的 I/O 请求可划分为两大类：第一类是最终能够留存至检索结束候选池内的向量对应的 I/O 请求，这类请求的数量基本由目标召回精度决定；第二类是检索收敛后被淘汰、仅用于引导检索路径靠近目标区域的中间向量对应的 I/O 请求，该类请求的数量会随检索路径选择产生较大浮动。

已有研究表明，磁盘图检索可清晰划分为两个不同运行阶段：**趋近阶段**与**收敛阶段**。趋近阶段中，检索流程从入口节点逐步向近邻向量聚集区域靠拢；收敛阶段则在近邻区域内开展局部精细检索，直至候选池状态趋于稳定。
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260708115242.png)
如图 6 所示，两个阶段的 I/O 请求构成存在本质差异。趋近阶段内候选池向量更新速度快，绝大多数被访问向量都属于中途会被淘汰的中间节点，该阶段下发的 I/O 请求中，仅有 43.6% 至 54.8% 服务于最终留存向量，意味着绝大多数 I/O 请求都存在优化缩减空间；进入收敛阶段后，检索已锁定核心近邻范围，候选池状态基本稳定，此阶段 93.3% 至 98.9% 的 I/O 请求均对应最终有效结果，绝大部分读写操作都是完成检索任务必不可少的核心开销。

基于上述特征可以确定，两个检索阶段需要采用完全不同的最优检索策略。在趋近阶段，由于多数 I/O 请求不服务于最终结果，检索流程应当适当克制下发磁盘请求，优先处理内存中已就绪的邻居向量；在收敛阶段，绝大部分 I/O 请求均为必需请求，此时应当主动积极下发 I/O 请求，尽可能缩短获取向量全精度数据的等待时长。本文基于该思路设计出可自适应检索阶段的全新检索算法 —— 前瞻搜索。

**趋近阶段的前瞻搜索策略**
本文沿用已有方案的阶段判定方式：设定远小于候选池总规模 L 的数值 n，当候选池中排名前 n 的向量均已完成遍历拓展时，判定检索从趋近阶段转入收敛阶段。
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260708115825.png)
在趋近阶段内，前瞻搜索会在**内存优先模式**与**常规检索模式**之间动态切换，运行逻辑如图 7 所示，具体规则见算法 1。
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260708115839.png)
1. **内存优先模式**
    系统按照距离由近至远遍历候选池，优先选取至多 W 个已驻留内存的向量进行拓展遍历，暂时跳过存放在磁盘中的向量，同时记录本轮第一个被跳过的磁盘向量，留作下一轮判定依据。该模式优先处理内存数据，能够有效减少非最终有效向量对应的磁盘 I/O 请求。
2. **常规检索模式**
    检索流程切换为标准贪心光束检索逻辑，不再区分向量存储位置，从候选池中选取至多 W 个向量进行拓展，按需为磁盘向量下发 I/O 读取请求，同时记录候选池中剩余距离最近且未被访问的磁盘向量，作为下一轮切换判定依据。
两种模式的切换由每轮检索伊始执行的**持久性检测**机制控制：系统筛查候选池中排名前 W 的未访问向量，判断上一轮被跳过的首个磁盘向量是否依旧处于该区间内。

若该向量仍然留在前列区间内，说明期间没有出现距离更近的内存向量将其挤出排序区间，证明该向量对检索推进至关重要，必须从磁盘读取数据，本轮检索随即切换为常规模式；

若该向量已退出前列区间，则继续维持内存优先模式运行。

运行示例如图 7：第一轮检索跳过磁盘节点 0，优先处理内存节点 2 与节点 4；第二轮中节点 0 已退出前列未访问向量区间，继续沿用内存优先模式，跳过节点 5 并处理内存节点 6 与节点 8；第三轮中节点 5 持续停留在前列未访问区间内，系统切换为常规检索模式，不再区分存储位置，统一拓展前列所有未访问向量，并记录下一个待跳过磁盘向量。

**收敛阶段的前瞻搜索策略**
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260708125330.png)
判定进入收敛阶段后，检索核心目标转变为尽快为最终候选池内的向量下发 I/O 请求。如图 8 所示，收敛阶段内多数 I/O 请求均为必需请求，但候选池靠后位置的向量仍有可能因新邻居向量加入而被淘汰，且排序越靠后，被淘汰的概率越高。

因此检索策略需要遵循差异化原则：对候选池前列向量积极快速下发 I/O 请求，越靠近候选池末尾，下发请求的节奏越趋于保守。

传统 PipeANN 方案在进入收敛阶段后，采用固定线性增幅的方式逐轮增大光束宽度，该方式未考虑候选池后端向量淘汰概率持续上升的特点，造成收敛初期 I/O 请求下发过于保守、后期下发过于激进，既产生无效 I/O 开销，又容易引发请求下发延迟。

对此我们提出一种**动态光束宽度策略**：在检索进入收敛阶段初期快速拉高光束宽度，之后再逐步递减。具体而言，收敛阶段首轮光束宽度Wconv​初始值设为α×L；其中α取值介于 0 到 1 之间，是相对于候选池总规模L的初始激增比例，以此在收敛阶段伊始集中批量发起 I/O 请求。后续每一轮检索中，均按照既定规则更新Wconv​数值。
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260708125551.png)
其中衰减系数β的取值范围介于 0 到 1 之间，用于逐轮缩减收敛阶段光束宽度Wconv​，使得检索流程在遍历推进过程中，对候选池靠后位置的向量采取更为保守的 I/O 请求下发策略。

该调整机制持续生效，直至候选池内排名前L的所有向量全部完成遍历拓展。初始激增系数α取值越大，收敛阶段初期下发 I/O 请求的节奏越激进，但容易对候选池后端向量产生无效 I/O 开销；而衰减系数β通过对Wconv​进行逐级递减，让检索流程在遍历逐步靠近候选池末尾时主动趋于保守。

二者相互配合，既能最大程度缩减最终候选池内有效向量的 I/O 请求下发延迟，又能有效避免产生冗余磁盘读写操作。结合大量实验验证，本文建议将初始激增系数α设置为 0.25，衰减系数β设置为 0.95。
## 4.3 带定制化候选池的优先级 I/O-CPU 流水线
如第三章所述，现有检索方案中 CPU 在绝大部分 I/O 等待时段都处于空闲状态；同时前文也已说明，处理候选池中距离并非最近的内存驻留向量，能够提升检索精度，进而减少达成既定召回率目标所需的 I/O 操作次数。为此本文提出**优先级 I/O-CPU 协同流水线**，在 I/O 等待时段内按照任务对下一轮 I/O 决策的影响优先级，有序填充有效 CPU 运算任务。图 9 展示了两轮连续检索流程中该流水线的运行机制。
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260708152558.png)
**任务优先级划分**
所有可延后执行的 CPU 运算任务，均根据其对下一轮 I/O 决策的影响程度，划分为三个优先级等级：
- **优先级 1（最高）**：计算当前轮次选中拓展的内存驻留向量的邻居节点的近似距离。由于这些向量已被确定为拓展对象，其邻居节点的距离计算与插入操作会直接决定候选池的下一轮状态，进而影响后续 I/O 决策。因此如图 9 所示，P1 级任务始终优先执行且不可被中断 —— 若延迟该任务，系统将基于过时的候选池状态制定 I/O 决策。
- **优先级 2**：拓展候选池C中存在、但未被选入当前轮次拓展列表的内存驻留向量。与 P1 级任务不同，这类向量的邻居节点对下一轮 I/O 决策的直接影响较弱。但处理这类向量仍有可能更新候选池状态，减少后续 I/O 操作，并提升最终检索精度。LAANN 在 I/O 等待时段内逐个处理这些向量，每处理完成一个向量后，都会检查待处理的 I/O 请求是否已完成；一旦 I/O 请求执行完毕，立即停止处理 P2 级任务，避免延误检索推进节奏。
- **优先级 3（最低）**：为检索过程中访问过的所有向量计算全精度距离，并据此完成排序。这类计算与 I/O 决策无任何关联，完全独立于 I/O 请求下发流程。传统方法通常在检索结束后再执行重排序操作，而 LAANN 则在 P1 与 P2 级任务全部执行完毕后，利用 I/O 等待时段逐步计算并维护所有已访问向量的全精度距离排序，将检索结束后的收尾工作降至最低。

**带溢出缓冲区的定制化候选池**
如第三章所述，在 I/O 等待时段内，候选池中未访问的内存驻留向量会被快速处理完毕，导致 CPU 再次进入空闲状态，错失减少 I/O 次数、提升检索精度的优化机会。为解决这一问题，LAANN 将候选池总规模扩大至μ×L，但保持收敛判定条件不变：检索仍以候选池内前L个向量全部完成遍历为终止条件。
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260708153125.png)
如图 10 所示，额外新增的(μ−1)×L个位置构成溢出缓冲区，作为 I/O 等待时段内 P2 级任务的天然内存驻留向量储备池。向量通过常规候选池插入流程进入溢出缓冲区，因此已按照与查询向量的近似距离完成排序，无需额外计算即可评估其关联性。图中展示了内存驻留向量在标准候选池与溢出缓冲区中的分布情况，为 P2 级任务提供了持续稳定的运算来源。

增大系数μ能为 P2 级任务提供更多运算机会，但超过一定阈值后收益会逐渐递减，同时会增加候选池管理开销。实际应用中，μ取值在（2,3）区间内可实现性能与开销的良好平衡。溢出缓冲区无需额外数据结构支持，几乎不引入额外开销，也不改变优先级判定逻辑，是一种轻量级的检索算法与流水线协同优化方案。

**优化效果**
通过在 I/O 等待时段按照任务对下一轮 I/O 决策的关联性排序 CPU 运算任务，该流水线实现了三大优化目标：减少磁盘 I/O 操作次数、提升检索精度，并将大部分可延后执行的计算任务与 I/O 等待时段并行执行。

可执行的运算工作量会根据每轮 I/O 请求的实际延迟动态调整：I/O 等待时间较短时，仅能完成 P1 级任务；等待时间较长时，可额外执行 P2 与 P3 级任务。溢出缓冲区则确保检索全程都有充足的 P2 级运算任务可执行，在几乎不增加额外开销的前提下，让 CPU 在 I/O 等待时段始终保持高效运行。
## 4.4 轻量化内存图索引
**现有内存图索引的精度不匹配问题**
现有相关方法（如 PipeANN、Starling、MARGO）受 HNSW 启发，通过从数据集中随机采样向量，并基于采样向量构建 Vamana 图作为内存索引。在检索过程中，这类索引基于**全精度距离**完成遍历，因此需要将采样向量的原始数据加载至内存。

但磁盘图检索的核心排序依据，是压缩向量计算得到的**近似距离**。内存全精度检索定位到的入口节点，未必是磁盘近似距离排序逻辑下距离最近的向量。更关键的是，由于这种精度不匹配问题，这类入口节点仅能作为检索起始点，无法直接加入磁盘图检索的候选池，导致磁盘图检索只能从近乎空的候选池状态从零开始推进。
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260708154110.png)
**核心设计思路**
想要快速定位近邻向量所在区域，仅使用近似距离进行检索不仅完全足够，而且相比全精度距离计算，速度更快、内存利用效率也更高。此外，磁盘图检索真正需要的，是一个基于近似距离、装满与查询向量邻近向量的候选池，而非仅依赖少数入口节点、从空候选池开始检索。

**索引构建方法**
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260708153645.png)
LAANN 的轻量化内存图索引，基于 PageANN 的页对齐磁盘图设计构建 —— 该设计将空间上邻近的向量聚合至同一个磁盘页中，并将每个磁盘页视作单个图节点。

轻量化内存图索引，正是基于每个磁盘页的**质心向量**构建的 Vamana 图。如图 11 所示，每个磁盘页内均存储着空间邻近的向量，其质心向量可自然代表该页覆盖的局部向量区域。

索引通过为这些质心向量添加长程边连接，使检索流程无需访问磁盘，仅靠少数几步跳转即可在图中完成大范围遍历。由于索引检索过程直接使用内存中已加载的向量压缩表示计算距离，无需额外为向量存储分配内存开销。

当内存配额充足时，系统为每个磁盘页都维护一个质心向量，实现全图覆盖；当内存资源受限时，可通过随机采样部分质心向量的方式缩减内存占用。LAANN 的前瞻搜索机制与优先级 I/O-CPU 流水线均基于该磁盘图设计实现，同时也可便捷迁移至其他磁盘式近似最近邻检索系统。

**磁盘图检索候选池初始化流程**
如图 11 所示，轻量化内存图索引的检索流程从入口质心节点出发，全程在内存内完成遍历，无需下发任何磁盘 I/O 请求，即可定位近邻向量所在的目标区域。

当索引检索流程收敛后，LAANN 会对结果集中的所有质心向量进行遍历：首先确定每个质心向量对应的磁盘页 ID，再根据页 ID 定位该页内的所有向量 ID，计算这些向量与查询向量的近似距离，最后将其插入磁盘图检索的候选池。

该流程将质心候选集合，转化为一批集中分布在真实近邻向量附近的高质量向量候选。由于内存索引检索与磁盘图检索采用的是同一套近似距离度量标准，不存在精度不匹配问题，因此这些初始化的候选向量可直接用于后续磁盘图遍历。

磁盘图检索流程直接从这个预填充完成的候选池启动，大幅减少了趋近阶段定位近邻向量所需的磁盘 I/O 操作次数。
# 五、实现细节
LAANN 基于 PageANN 代码库，采用 C++ 语言实现，代码总量约 4000 行。系统沿用了 PageANN 的磁盘图设计：将空间上最接近质心的向量聚合至同一磁盘页，并将每个磁盘页视作单个图节点。这种页对齐设计，使每个图节点与一块固态硬盘（SSD）物理页一一对应，既消除了 I/O 读放大问题，也让初始化流程能够高效地将质心候选扩展为该页内的所有向量。

**异步 I/O**
LAANN 采用 Linux `io_uring` 接口实现异步磁盘读取。当前瞻搜索进入常规模式时，系统通过 `io_uring_submit` 接口，将所有磁盘页的 I/O 请求一次性批量提交至 `io_uring` 的提交队列。在优先级流水线利用 I/O 等待时段执行 CPU 任务的同时，系统会在优先级任务执行间隙，通过非阻塞的 `io_uring_peek_batch_cqe` 接口轮询并收集已完成的 I/O 请求。若所有可延后执行的 CPU 任务均已处理完毕，系统则会调用 `io_uring_wait_cqe` 接口阻塞等待，直至所有 I/O 请求完成。该设计确保 CPU 能够在 I/O 等待时段持续处理优先级任务，仅当无有效任务可执行时才进入阻塞状态。

**缓存页加载策略**
LAANN 通过在数据集 1% 的采样数据上运行轻量化内存图索引检索，统计各磁盘页的访问频率，并按访问频率对页节点进行排序，将排序结果写入文件。在系统加载阶段，页节点将按照该排序依次载入内存，确保访问频率最高的页面被优先缓存。查询时，系统通过哈希表检查页面是否已驻留内存。此外，也可根据该排序文件对磁盘上的页面进行物理重排，从而消除哈希表带来的内存开销：此时仅需通过简单的索引比较即可判断页面是否已缓存 —— 若页面索引小于已加载页面的总数，则该页面已在缓存中。
# 六、实验评估
本节在多种工作负载与内存配额条件下，对比评测 LAANN 与当前主流磁盘式近似最近邻检索方案的性能表现。
## 6.1 实验环境配置
**硬件配置**
所有实验均在一台工作站上完成，设备搭载 Intel Core i9-13900 CPU（基础频率 2.0 GHz，24 核心）、128 GB DDR5 内存、1 TB KIOXIA NVMe 固态硬盘，运行 Ubuntu 24.04 操作系统。

**对比基线算法**
选取五种当前主流磁盘式近似最近邻检索方法作为对比基线：DiskANN、Starling、MARGO、PipeANN 和 PageANN。

**实验数据集**
所有方案均在五组具备代表性的标准数据集上完成评测，覆盖不同数据规模与向量维度，分别为 SIFT100M、SPACEV100M、DEEP100M、SPACEV1B 和 SIFT1B，具体参数见表 2。
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260708155155.png)
**评测指标**
本次实验重点关注三项核心性能指标：
- **Recall@10**：检索结果前十位中，真实前十近邻向量的命中占比；
- **查询延迟**：单条查询请求的平均处理耗时，单位为毫秒（ms）；
- **系统吞吐量**：并发工作负载下每秒可处理的查询请求数（QPS）。

**实验参数设置**
所有系统均在相同硬件、数据集和索引构建参数下配置，以确保磁盘占用和内存占用完全一致。具体而言，所有方法在平均向量度数、向量压缩率、建图阶段候选池大小以及单页向量数量上均采用统一参数。检索阶段，所有数据集的搜索光束宽度W（即 I/O 读请求的批处理大小）均设为 5，查询线程数固定为 16。对于光束宽度非固定的 PipeANN，W设置为作者推荐的最大值 32。对于 LAANN，常规搜索模式的光束宽度设为 5，激增比例α=25%，衰减系数β=0.95，候选池扩展系数μ=2.4。对于其他方法的特定超参数，我们采用各自论文或官方代码仓库推荐的最优值。
## 6.2 亿级规模数据集性能
实验在内存配额为数据集大小 0.5 倍的条件下，对所有方法进行评估。在此配额下，所有方法分配 0.2 倍用于存储压缩向量，0.2 倍用于存储缓存磁盘页；带内存图索引的方法（Starling、MARGO、PageANN 和 LAANN）额外分配 0.1 倍用于索引。无内存图索引的 DiskANN，将剩余的 0.3 倍全部分配给缓存磁盘页；不支持缓存磁盘页的 PipeANN，则将剩余的 0.3 倍全部分配给其内存图索引。
![](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260708155343.png)
图 12 展示了三组亿级规模数据集上的召回率 - 延迟、召回率 - QPS 权衡曲线。LAANN 在所有数据集上均实现了最优的性能权衡，在相同召回率水平下，延迟显著更低、吞吐量显著更高。这种优势在 SPACEV100M 和 DEEP100M 数据集上尤为突出，LAANN 在广泛的召回率目标范围内，吞吐量均为次优方法的 2 倍以上。

值得注意的是，尽管 PipeANN 维护了内存图索引，但其性能始终低于其他所有方法。如图 1 (a) 所示，PipeANN 在 2 线程时延迟表现尚可，但随着线程数增加，其延迟恶化最为明显。根本原因是 PipeANN 为最大化 I/O 并行性，发起了更多的 I/O 操作，在高线程数下的 I/O 竞争中，这会变得越来越昂贵。相比之下，LAANN 每个查询的 I/O 操作比 PipeANN 少 77%-89%（表 3），因此每个线程对 SSD 的负载按比例降低，随着线程数增加，LAANN 天生对 I/O 竞争具有更强的鲁棒性。
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260708155443.png)
表 3 提供了 Recall@10=0.9 条件下的详细对比数据。在三组数据集上，LAANN 的吞吐量均为最优基线 PageANN 的 1.41 倍至 2.12 倍，同时平均 I/O 操作减少了 37%-58%。值得注意的是，吞吐量的提升幅度始终大于仅靠减少 I/O 操作所能带来的提升，这表明 LAANN 不仅受益于更少的 I/O，还通过优先级 I/O-CPU 流水线实现了更好的 CPU 利用率。与应用最广泛的基线 DiskANN 相比，LAANN 的吞吐量提高了 3.47 倍至 4.66 倍，证明了 I/O 感知搜索相较于标准贪心光束搜索的根本性优势。
## 6.3 十亿级规模数据集性能
实验在内存配额为数据集大小 0.3 倍的条件下，将 LAANN 与 DiskANN、PipeANN 和 PageANN 在 SIFT1B 和 SpaceV1B 数据集上进行评估。由于 Starling 和 MARGO 的构建或检索阶段超出了实验设备的内存容量，因此未纳入对比。在此配额下，所有方法均预留 0.2 倍用于存储压缩向量。对于剩余的 0.1 倍，PageANN 和 PipeANN 将其全部分配给内存图索引，用于存储采样节点的图边和原始向量值；LAANN 仅用 0.05 倍构建轻量化内存图索引，因其使用压缩向量进行搜索，无需存储原始向量值，因此剩余的 0.05 倍可用于额外的缓存磁盘页。三种内存图索引均基于相同数量的采样节点构建，以确保公平比较。无内存图索引的 DiskANN，将剩余的 0.1 倍全部分配给缓存磁盘页。
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260708155508.png)
如图 13 所示，LAANN 在两个数据集的整个召回率范围内，均实现了最低的延迟和最高的吞吐量。在 SIFT1B 数据集 Recall@10=0.9 时，LAANN 的吞吐量为 1147.7 QPS，延迟为 13.87ms，吞吐量分别是 PageANN 的 1.42 倍、DiskANN 的 2.38 倍、PipeANN 的 4.03 倍。在 SpaceV1B 数据集上，优势更为显著。与亿级规模的结果相比，在十亿级规模上，LAANN 与所有基线方法的性能差距显著扩大。这是因为十亿级数据集的检索路径更长，每个查询的 I/O 操作更多，加剧了 I/O 竞争并提高了单请求 I/O 延迟。在这些条件下，LAANN 减少总 I/O 操作数的能力，转化为了成比例的、更大的性能优势。
## 6.4 缓存图节点的影响
本实验评估了不同方法利用缓存图节点提升检索性能的效果。具体而言，通过调整用于存储缓存图节点的内存配额，将总内存配额从数据集大小的 0.3 倍变化到 0.9 倍。在每个配额下，预留 0.2 倍用于存储压缩向量；对于支持内存图索引的方法（Starling、MARGO、PageANN 和 LAANN），额外分配 0.1 倍用于索引。剩余配额决定了可缓存到内存中的图节点数量。无内存图索引的 DiskANN，将剩余配额全部分配给缓存图节点；不支持通过缓存图节点加速检索的 PipeANN 未纳入本实验。
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260708155523.png)
如图 14 所示，LAANN 在所有三个数据集和内存配额下，QPS 和延迟性能均始终优于所有基线方法。更重要的是，随着内存配额的增加，LAANN 的性能显著提升，而基线方法，特别是 DiskANN、Starling 和 MARGO，仅表现出边际改善。例如，在 Spacev100m 数据集上，LAANN 的 QPS 从 0.3 倍配额时的 6122.6 提升到 0.9 倍配额时的 13346.6，提升了 2.18 倍；而 Starling 在同一范围内仅提升了 1.16 倍。这种鲜明对比表明，LAANN 比现有方法能更有效地利用额外的缓存数据。其根本原因在于：现有方法无论节点是否被缓存，都严格按照距离顺序拓展节点，因此额外的缓存数据不会改变检索路径。相比之下，LAANN 通过前瞻搜索和优先级 I/O-CPU 流水线主动利用缓存邻居，将额外的内存直接转化为更少的 I/O 操作和更低的延迟。
## 6.5 组件拆解分析
![image.png](https://raw.githubusercontent.com/waibibab-cs/blog_img/main/cdnimg/20260708155539.png)
以 PageANN 为基线，逐步添加 LAANN 的组件，分析各组件的独立贡献。基线 PageANN 缓存与 LAANN 相同数量的磁盘页，但不使用内存图索引，仅采用标准贪心光束搜索算法，无 LAANN 的任何优化。需要注意的是，表 3 中的 PageANN 受益于其自身的内存图索引，这是其 QPS（SIFT100M 上为 2718.4；SPACEV100M 上为 3761.6）高于此处基线（SIFT100M 上为 1825.8；SPACEV100M 上为 2686.6）的主要原因。这种受控设置确保了表 4 中的每一行都能精确隔离一个算法组件的贡献，单元格中显示的百分比反映了该组件相对于基线的独立贡献。

**前瞻搜索**
前瞻搜索在所有指标上都带来了最大的单项提升。在 SIFT100M 数据集上，相较于基线，其使 QPS 提升了 70%，延迟降低了 41%，平均 I/O 操作减少了 31%；在 SPACEV100M 数据集上，增益更为显著，QPS 提升了 96%，延迟降低了 49%，平均 I/O 操作减少了 43%。这证实了前瞻搜索是影响最大的组件，因为它通过在趋近阶段优先处理内存邻居，并在收敛阶段最大化 I/O 并行性，直接减少了 I/O 操作的数量。

**优先级 I/O-CPU 流水线**
带有定制化候选池的优先级 I/O-CPU 流水线，在 SIFT100M 数据集上进一步带来了 14% 的 QPS 提升和 9% 的 I/O 减少；在 SPACEV100M 数据集上则带来了 30% 的 QPS 提升和 12% 的 I/O 减少。I/O 延迟的降低尤为显著，因为该流水线将更多计算隐藏在 I/O 等待时间之后，并通过更好地对 CPU 工作进行优先级排序来提高 I/O 效率。

**轻量化内存图索引**
轻量化内存图索引在 SIFT100M 数据集上进一步带来了 30% 的 QPS 提升和 13% 的 I/O 减少；在 SPACEV100M 数据集上则带来了 80% 的 QPS 提升和 10% 的 I/O 减少。在 SPACEV100M 上显著更高的 QPS 增益表明，在趋近阶段占主导地位的数据集上，在检索开始时用高质量候选初始化磁盘图候选池特别有效，因为它显著减少了趋近阶段的 I/O 操作并加快了收敛速度。

总体而言，这三个组件对 LAANN 的性能均有显著且互补的贡献。前瞻搜索通过直接减少 I/O 提供了最大的单项增益，流水线通过利用 CPU 空闲时间进一步提高了效率，而轻量化内存图索引则从检索一开始就加速了收敛过程。
# 七、相关工作
**分布式近似最近邻检索方案**
分布式近似最近邻检索系统通过将索引分散到多个节点，实现了 Web 级规模数据集的扩展。独立分片方案将不相交的数据分区分配给每个节点，合并各分片的查询结果，以牺牲查询质量为代价换取实现上的简洁性；全局索引方案在节点间构建统一索引，避免了冗余计算，但代价是通信开销更高。这两类方案都旨在提升跨机器的可扩展性，与专注于最大化单机 I/O 效率的 LAANN 形成互补关系。LAANN 的 I/O 感知搜索机制可作为这类分布式系统中单个节点的检索引擎使用。

**磁盘式近似最近邻检索方案**
如第 2 节所述，磁盘式方法将向量和图边存储在固态硬盘中，以降低内存需求。DiskANN 提出了基础性的设计：在内存中存储压缩向量，按需从磁盘读取全精度向量，并缓存高频访问节点以减少 I/O 操作。Starling 和 MARGO 优化了每个磁盘页内的空间局部性，并维护内存入口点图，以减少到达近邻区域所需的跳转次数。PageANN 则通过将图节点粒度与 SSD 页大小对齐，消除了 I/O 读放大问题。

与本研究最相关的工作是 PipeANN，该方案同样区分了检索阶段，并在收敛阶段调整光束宽度。然而，PipeANN 为了最大化 I/O 并行度，在发起 I/O 之前会尽量减少 CPU 计算，这导致了更多的 I/O 操作，在 I/O 竞争激烈的场景下会造成严重的性能下降。相比之下，LAANN 在做出 I/O 决策前，会优先处理与后续 I/O 相关的 CPU 工作，包括缓存中的向量和刚从磁盘加载的向量，以此减少总的 I/O 操作次数。此外，PipeANN 在收敛阶段线性增加光束宽度，没有考虑到候选池尾部向量被淘汰的概率会不断增加，导致了后期 I/O 浪费和前期 I/O 提交延迟。LAANN 的动态光束宽度策略通过在收敛初期快速拉高，随后逐渐衰减，解决了这一问题。从根本上说，包括 PipeANN 在内的所有现有方法，都仅从 I/O 视角优化 I/O 操作，且无论数据是否已缓存，都严格按照距离顺序拓展节点。LAANN 是首个利用 CPU 计算与 I/O 操作之间耦合关系的方案，实现了更好的 CPU 与 I/O 平衡，并降低了整体查询延迟。

**专用硬件近似最近邻检索方案**
部分系统通过将检索算法与专用硬件协同设计，来规避 CPU 与 I/O 瓶颈。FusionANNS 将距离计算卸载到 GPU，使其与 SSD I/O 并行执行；HM-ANN 将图的上层存储在持久化内存中，下层存储在 DRAM 中，以利用内存容量层级；SmartANN 则将计算直接推入存储设备，彻底消除了主机端 I/O 操作。

虽然这些方案在其目标硬件上取得了出色的效果，但它们天生依赖于特定的硬件配置。LAANN 则完全通过算法层面的优化来解决相同的 I/O 瓶颈，不依赖特定硬件，可部署在任何通用服务器上。
# 八、结论
本文提出了 LAANN—— 一款磁盘式近似最近邻检索系统。通过识别 CPU 计算与 I/O 操作之间的强耦合关系，以及不同检索阶段 I/O 构成的本质差异，LAANN 让检索算法具备了完备的 I/O 感知能力。LAANN 引入了三项互补的核心技术：
1. **前瞻搜索机制**：能够根据当前检索阶段动态调整执行策略；
2. **优先级 I/O-CPU 协同流水线**：在带溢出区的定制化候选池支持下，按照任务对后续 I/O 决策的影响优先级，有序填充 CPU 空闲时段；
3. **轻量化高速内存图索引**：从检索初始阶段就为磁盘图检索提供高质量候选向量。
在三组亿级规模和两组十亿级规模数据集上的实验结果表明，与当前主流基线方案相比，LAANN 的吞吐量提升了 1.41 倍至 4.66 倍，查询延迟降低了 29% 至 79%，I/O 操作次数减少了 1.59 倍至 6.34 倍。这证实了对于磁盘式近似最近邻检索而言，让检索算法具备 I/O 感知能力，是对传统按距离排序的贪心搜索算法的根本性改进。
